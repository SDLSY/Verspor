## 3. 云端 Supabase 表结构与 migration 来源

下面只写当前项目第 6 章最关键的云端表族，并标明来源 migration。

### 3.1 核心睡眠与推理表族

来源：[0001_core_schema.sql](/D:/newstart/cloud-next/supabase/migrations/0001_core_schema.sql)

#### 3.1.1 devices

字段：

- `id uuid primary key`
- `user_id uuid not null`
- `device_id text not null`
- `device_name text`
- `firmware_version text`
- `created_at timestamptz not null`
- 约束：`unique(user_id, device_id)`

#### 3.1.2 sleep_sessions

字段：

- `id uuid primary key`
- `user_id uuid not null`
- `sleep_record_id text not null`
- `session_date timestamptz not null`
- `bed_time timestamptz`
- `wake_time timestamptz`
- `total_sleep_minutes int`
- `deep_sleep_minutes int`
- `light_sleep_minutes int`
- `rem_sleep_minutes int`
- `source text not null default 'mobile_upload'`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`
- 约束：`unique(user_id, sleep_record_id)`

#### 3.1.3 sleep_windows

字段：

- `id uuid primary key`
- `user_id uuid not null`
- `sleep_record_id text not null`
- `window_start timestamptz not null`
- `window_end timestamptz not null`
- `hr_features jsonb`
- `spo2_features jsonb`
- `hrv_features jsonb`
- `temp_features jsonb`
- `motion_features jsonb`
- `ppg_features jsonb`
- `edge_anomaly_signal numeric(5,4)`
- `created_at timestamptz not null`

#### 3.1.4 inference_jobs

字段：

- `id uuid primary key`
- `user_id uuid not null`
- `sleep_record_id text not null`
- `status text not null`，枚举为 `queued/running/succeeded/failed`
- `idempotency_key text not null`
- `model_version text`
- `error_message text`
- `created_at timestamptz not null`
- `started_at timestamptz`
- `finished_at timestamptz`
- 约束：`unique(user_id, idempotency_key)`

#### 3.1.5 sleep_stage_results

字段：

- `id uuid primary key`
- `user_id uuid not null`
- `sleep_record_id text not null`
- `epoch_index int not null`
- `stage_5 text not null`，枚举 `WAKE/N1/N2/N3/REM`
- `stage_legacy text not null`，枚举 `AWAKE/LIGHT/DEEP/REM`
- `confidence numeric(5,4)`
- `model_version text not null`
- `created_at timestamptz not null`
- 约束：`unique(user_id, sleep_record_id, epoch_index)`

#### 3.1.6 anomaly_scores

## 4. 主外键与业务关联规则

这一节直接回答“表与表怎么连”。

### 4.1 Android Room 中有物理外键的关系

| 子表 | 子字段 | 父表 | 父字段 | 删除策略 |
|---|---|---|---|---|
| `recovery_scores` | `sleepRecordId` | `sleep_records` | `id` | `CASCADE` |
| `doctor_assessments` | `sessionId` | `doctor_sessions` | `id` | `CASCADE` |
| `assessment_answers` | `sessionId` | `assessment_sessions` | `id` | `CASCADE` |
| `intervention_executions` | `taskId` | `intervention_tasks` | `id` | `CASCADE` |
| `medical_metrics` | `reportId` | `medical_reports` | `id` | `CASCADE` |
| `prescription_bundles` | `profileSnapshotId` | `intervention_profile_snapshots` | `id` | `CASCADE` |
| `prescription_items` | `bundleId` | `prescription_bundles` | `id` | `CASCADE` |

### 4.2 Android Room 中只有逻辑关联、没有物理外键的关系

这些关系在业务上明确存在，但 Room 没有声明 FK：

- `health_metrics.sleepRecordId -> sleep_records.id`
- `ppg_samples.sleepRecordId -> sleep_records.id`
- `doctor_messages.sessionId -> doctor_sessions.id`

也就是说，Android 本地数据库里并不是所有“看起来像外键”的字段都被 Room 强制约束。

### 4.3 云端 Supabase 中有物理外键的关系

| 子表 | 子字段 | 父表 | 父字段 |
|---|---|---|---|
| `intervention_executions` | `(user_id, task_id)` | `intervention_tasks` | `(user_id, task_id)` |
| `medical_metrics` | `(user_id, report_id)` | `medical_reports` | `(user_id, report_id)` |
| `prescription_recommendations` | `snapshot_id` | `prescription_snapshots` | `id` |
| `prescription_generation_logs` | `snapshot_id` | `prescription_snapshots` | `id` |
| `prescription_generation_logs` | `recommendation_id` | `prescription_recommendations` | `id` |

### 4.4 云端 Supabase 中常见的逻辑关联规则

#### 用户与设备

- 云端设备表按 `devices.user_id + devices.device_id` 约束唯一。
- 本地 `DeviceEntity` 不带用户字段，用户隔离主要发生在云端。

#### 一份报告对应多条医疗指标

- 本地：`medical_reports.id -> medical_metrics.reportId`
- 云端：`medical_reports(user_id, report_id) -> medical_metrics(user_id, report_id)`
- 这是标准的 1 对多。

#### 一个问诊 session 对多条 message，对应一份 assessment

- 本地：
  - `doctor_sessions.id -> doctor_messages.sessionId` 是 1 对多，但只是逻辑关联
  - `doctor_sessions.id -> doctor_assessments.sessionId` 是 1 对 1 或 1 对多的实现空间，当前产品语义更接近“一次 session 产出一份最新 assessment”
- 云端：
  - 云端没有完整 `doctor_sessions` / `doctor_messages` 表
  - 云端只存 `doctor_inquiry_summaries(user_id, session_id)` 这种摘要层

#### 一个 intervention task 和 execution、effect、trend 的关系

- 本地：
  - `intervention_tasks.id -> intervention_executions.taskId` 是 1 对多
  - `effectScore` 存在 `intervention_executions` 和 `relax_sessions`
- 云端：
  - `intervention_tasks(user_id, task_id) -> intervention_executions(user_id, task_id)` 是 1 对多
  - 趋势和效果页不会读单独的 `effect` 表，而是聚合 `intervention_executions`
  - 代码参考：[recommendation-effects.ts](/D:/newstart/cloud-next/src/lib/recommendation-effects.ts)

#### recommendation trace 如何关联患者时间线

这是当前最容易写错的点。

真实情况是：

- `recommendation_traces` 有 `user_id`、`related_snapshot_id`、`related_recommendation_id`
- 但 SQL 中没有把它们声明成物理外键
- 当前后台患者时间线聚合函数 [getAdminPatientTimeline()](/D:/newstart/cloud-next/src/lib/admin-patients.ts) 并没有把 `recommendation_traces` 合入 `AdminTimelineEvent`
- 当前患者时间线实际来自这些源表：
  - `sleep_sessions`
  - `inference_jobs`
  - `nightly_reports`
  - `intervention_tasks`
  - `intervention_executions`
  - `medical_reports`
  - `medication_analysis_records`
  - `food_analysis_records`
  - `audit_events`

所以更准确的表述是：

- `recommendation_traces` 与患者是通过 `user_id` 关联
- 与处方快照/推荐结果是通过 `related_snapshot_id`、`related_recommendation_id` 做语义关联
- 它当前不是后台患者时间线的直接事件源，而是建议、效果和解释页面的独立数据源

### 4.5 没有单独“患者时间线表”

当前后端没有一张物理表叫“patient_timeline”。

后台时间线是服务层拼出来的：

- 代码位置：[admin-patients.ts](/D:/newstart/cloud-next/src/lib/admin-patients.ts)
- 输出类型：`AdminTimelineEvent`
- 做法：从多张源表按 `occurredAt` 转成统一事件，再按时间倒序排序

这意味着：

- 写文档时不要把“患者时间线”写成数据库中的一张表
- 应写成“后台工作台上的服务层聚合视图”

## 5. 第 6 章可直接引用的结论

下面这些句子可以直接改写进第 6 章。

- Android 本地数据库当前采用 Room，已注册 20 个实体对象，覆盖睡眠记录、健康指标、恢复分、设备、问诊、量表、医检报告、干预执行、画像快照、处方以及药物和饮食分析记录。
- 本地数据库并非所有关联都采用物理外键约束，部分关系如 `health_metrics.sleepRecordId`、`ppg_samples.sleepRecordId`、`doctor_messages.sessionId` 目前属于业务逻辑关联。
- 云端数据库以 Supabase PostgreSQL 为基础，围绕 `user_id` 实现多租户隔离，并通过 RLS 策略限制用户只能访问自身数据。
- 医检报告与医疗指标在云端采用标准 1 对多结构，其中 `medical_metrics` 通过复合外键 `(user_id, report_id)` 关联 `medical_reports`。
- 干预任务与干预执行在云端采用 `task_id` 为业务键、`user_id + task_id` 为复合关联的结构，执行记录通过复合外键挂到任务记录上。
- 建议轨迹 `recommendation_traces` 当前不属于后台患者时间线的直接事件源，而是建议解释、效果分析和模型追踪的独立数据源。
- 后台患者时间线不是一张物理表，而是服务层对睡眠、推理、报告、干预、药食分析和审计事件的统一聚合结果。

## 6. 写第 6 章时的禁写项

- 不要把 `doctor_inquiry_summaries` 写成云端完整问诊消息表，它只是摘要表。
- 不要把 `recommendation_traces` 写成已经和患者时间线做了物理外键绑定。
- 不要把 Android 本地所有 `xxxId` 字段都写成已声明外键。
- 不要把“患者时间线”写成数据库真实表。
- 不要把本地 Room 表和云端 Supabase 表写成完全同名、完全同结构的一套镜像；它们有相同业务域，但并不完全对等。
