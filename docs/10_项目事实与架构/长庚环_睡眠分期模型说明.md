# 标题
长庚环睡眠分期模型说明

- 适用任务：技术文档、比赛材料、官网技术介绍、AI 投喂
- 推荐阅读优先级：高
- 是否允许对外直接引用：允许，但必须保留“能力边界”表述，不能夸大为医疗诊断

## 1. 文档结论

长庚环仓库里的“睡眠分期模型”不是单一一份模型文件，而是一组分层能力：

1. 训练与研究层：`ml/` 下存在多版睡眠分期训练脚本，包括基线随机森林、聚合特征 Transformer，以及更重的原始波形多模态 Transformer 原型。
2. 云端推理层：`cloud-next` 支持通过 `model_registry` 配置，将睡眠分析指向 HTTP 推理服务、`local://baseline`、`local://ensemble`，或完全退回到 worker 内部 fallback 逻辑。
3. 当前 ONNX 服务层：`hf-inference/` 与 `cloud-next/models/` 中已经存在 `sleep-transformer-v2.onnx`，其目标是输出 5 类睡眠阶段和异常分数。
4. Android 端侧层：当前 Android 端真正部署的不是完整 5 类睡眠分期模型，而是一个轻量的异常检测 TFLite 模型 `sleep_model.tflite`，用于本地异常评分与离线兜底。

这意味着一个必须写清楚的事实：

- “仓库支持睡眠分期建模”是成立的。
- “Android 端本地直接运行完整 5 阶睡眠分期模型”当前并不成立。
- “云端默认一定在跑 ONNX 睡眠分期模型”也不成立；实际是否走学习模型，取决于 `model_registry` 的活动配置。

## 2. 分期标签体系

仓库统一采用 5 阶睡眠标签：

- `WAKE`
- `N1`
- `N2`
- `N3`
- `REM`

源码定义位置：

- `ml/train_baseline.py`
- `ml/train_multimodal_transformer.py`
- `ml/AggregatedTransformer.py`
- `hf-inference/app.py`

标签映射：

- `WAKE`：清醒
- `N1`：浅睡过渡期
- `N2`：稳定浅睡
- `N3`：深睡
- `REM`：快速眼动睡眠

## 3. 输入数据与特征工程

### 3.1 原始时间窗

预处理脚本 `ml/preprocessing.py` 默认将输入数据按以下方式组织：

- 窗口长度：`30` 秒
- 滑动步长：`15` 秒
- PPG 采样处理：带通滤波，`0.5Hz ~ 5.0Hz`
- 加速度低通滤波截止：`5.0Hz`

### 3.2 单窗聚合特征

当前仓库中最稳定、最贯通的特征模式是 17 维聚合特征。顺序在多个脚本与服务中保持一致：

1. 心率组 `hr`：4 维
2. 血氧组 `spo2`：3 维
3. HRV 组 `hrv`：4 维
4. 体温组 `temp`：3 维
5. 运动组 `motion`：1 维
6. PPG 组 `ppg`：1 维
7. 边缘异常信号组 `edge`：1 维

合计：`17` 维

字段来源如下：

- 心率：`heartRate`、`heartRateAvg`、`heartRateMin`、`heartRateMax`
- 血氧：`bloodOxygen`、`bloodOxygenAvg`、`bloodOxygenMin`
- HRV：`hrv`、`rmssd`、`sdnn`、`lfHfRatio`
- 体温：`temperature`、`temperatureAvg`、`temperatureTrend`
- 运动：`motionIntensity`
- PPG：`ppgValue`
- 边缘异常：`edgeAnomalySignal`

对齐代码位置：

- `ml/train_multimodal_transformer.py`
- `ml/train_aggregated_transformer.py`
- `cloud-next/src/lib/local-baseline-inference.ts`
- `hf-inference/app.py`

### 3.3 基线模型的上下文拼接

`ml/train_baseline.py` 不是只用单个 17 维窗口，而是使用相邻窗口上下文：

- 当前窗前后各取 `1` 个窗口
- 总上下文窗口数：`3`
- 总输入维度：`17 x 3 = 51`

这也是为什么训练脚本中的 baseline 输入维度是 `51`，而云端本地基线 fallback 只消费 `17` 维。两者不是同一份运行时实现。

## 4. 仓库中的几类睡眠分期模型

## 4.1 基线训练模型：RandomForestClassifier

训练脚本：

- `ml/train_baseline.py`

核心特点：

- 模型类型：`RandomForestClassifier`
- 树数：`500`
- 类别不平衡处理：`class_weight="balanced_subsample"`
- 训练样本再平衡：对少数类做随机过采样
- 输入：`51` 维上下文特征
- 输出：5 类睡眠阶段

以脚本为准时，该基线模型代表的是“传统机器学习 baseline”。

但是这里有一个重要事实需要写明：

- 仓库中已提交的 `ml/artifacts/week2/baseline/baseline_report.json` 写的是 `NearestCentroid`
- 同目录下 `baseline_model.json` 保存的是类别中心点
- 这说明仓库中的 baseline 脚本与已提交 artifact 存在版本漂移

因此对外描述时应当采用下面的表述：

- “当前 baseline 训练脚本使用随机森林”
- “仓库中保留的 baseline artifact 含有更早期的 centroid 版本痕迹，仅可作为历史参考”

### 当前仓库中 baseline artifact 的测试指标

`ml/artifacts/week2/baseline/baseline_report.json` 的已提交结果为：

- 测试集 Accuracy：`0.3917`
- 测试集 Macro F1：`0.3835`

分阶段 F1：

- `WAKE`：`0.6411`
- `N1`：`0.2684`
- `N2`：`0.3485`
- `N3`：`0.4129`
- `REM`：`0.2465`

这组结果说明：

- baseline 的作用更偏“对照组”和“伪标签生成起点”
- 单独作为最终商用睡眠分期模型，性能偏弱

## 4.2 聚合特征 Transformer：AggregatedSleepTransformer

模型定义：

- `ml/AggregatedTransformer.py`

训练脚本：

- `ml/train_aggregated_transformer.py`

导出脚本：

- `ml/export_aggregated_onnx.py`

核心结构：

- 输入仍为 7 组模态，共 `17` 维聚合特征
- 每个模态先通过独立线性投影映射到统一隐空间
- 使用 `CLS token + positional embedding + TransformerEncoder`
- 主头输出 `stage_logits`
- 副头输出 `anomaly_score`

默认结构参数：

- `d_model = 64`
- `num_heads = 4`
- `num_layers = 2`
- `dropout = 0.15`

多任务训练：

- 主任务：5 类睡眠分期
- 辅任务：异常检测
- anomaly loss 权重默认：`0.3`

### 当前已提交聚合 Transformer 的验证结果

产物位置：

- `ml/artifacts/week2/aggregated_transformer/aggregated_report.json`

测试集结果：

- Accuracy：`0.8052`
- Macro F1：`0.8027`

分阶段 F1：

- `WAKE`：`0.9299`
- `N1`：`0.8309`
- `N2`：`0.7834`
- `N3`：`0.8634`
- `REM`：`0.6058`

这组结果显著优于 baseline，说明：

- 在当前仓库的已提交训练结果里，`AggregatedSleepTransformer` 是最像“正式睡眠分期主模型”的候选实现
- `REM` 仍然是最难的类别，性能弱于其他阶段

### 模型体积

当前已提交产物大小：

- `best_model.pth`：约 `433 KB`
- `sleep-transformer-v2.onnx`：约 `490 KB`

这意味着它是一个偏轻量、可云端 CPU 推理、也具备进一步端侧压缩可能性的模型。

## 4.3 原始多模态 Transformer 原型：MultiModalTransformer

模型定义：

- `ml/MultiModalTransformer.py`

这是更“理想化”的研究型结构，特点是：

- PPG 使用 `1D-CNN` 编码，输入长度默认 `1500`
- 运动数据使用 `LSTM` 编码，输入默认 `750 x 6`
- 统计特征如 HR、HRV、SpO2、体温使用 `MLP`
- 全部特征融合后进入 Transformer
- 同时输出：
  - 睡眠分期
  - 恢复指数
  - 异常检测

它代表的是仓库中的“全模态、多任务研究方向”，但当前没有看到与之完全对应的完整已提交训练产物与实际部署链路。因此更适合写成：

- `研究/原型能力`

而不是：

- `当前实际落地主模型`

## 4.4 云端 HTTP 推理服务：Sleep Transformer V2

服务代码：

- `hf-inference/app.py`

服务说明：

- `hf-inference/README.md`

模型文件：

- `hf-inference/sleep-transformer-v2.onnx`
- `cloud-next/models/sleep-transformer-v2.onnx`

服务输入：

- 一组窗口 `windows`
- 每个窗口为 17 维聚合特征

服务输出：

- `stages`
- `anomalyScore`
- `factors`
- `insights`

服务内部逻辑：

1. 读取 17 维特征
2. 使用 ONNX Runtime 执行推理
3. 对 `stage_logits` 做 softmax
4. 输出每窗睡眠阶段和置信度
5. 额外汇总平均 `anomalyScore`

这条链路是当前仓库中最完整的“云端睡眠分期模型服务化”实现。

## 4.5 云端本地回退：local://baseline 与 local://ensemble

在 `cloud-next/src/lib/model-inference.ts` 中，如果活动模型 profile 的 `runtimeType = "http"`，但 endpoint 不是外部 URL，也可以指向：

- `local://baseline`
- `local://ensemble`

### `local://baseline`

实现：

- `cloud-next/src/lib/local-baseline-inference.ts`

特点：

- 不是 `train_baseline.py` 中的随机森林
- 它是一个硬编码 17 维 centroid 的最近中心式推断
- 直接基于与各类别中心的欧氏距离选 stage
- 再结合 `edge_anomaly_signal` 和清醒比例生成 `anomalyScore`

这条逻辑本质上是“轻量基线回退”，不是当前最优学习模型。

### `local://ensemble`

实现：

- `cloud-next/src/lib/local-ensemble-inference.ts`

特点：

- 将 `local://baseline` 的结果与一套启发式 Transformer 概率进行 soft voting
- 权重由环境变量 `LOCAL_ENSEMBLE_RF_WEIGHT` 控制，默认 `0.31`
- 最终融合成阶段与异常分数

这条逻辑属于：

- 云端内置的轻量融合回退

它的作用是：

- 在外部 ONNX 服务不可用时，依然给出相对合理的睡眠分期结果

## 4.6 worker 的最终兜底 fallback

如果没有可用 HTTP 模型，或 profile 本身是 fallback 运行模式，则 worker 会进入 `cloud-next/src/app/api/internal/worker/run/route.ts` 内的 fallback 逻辑。

这个 fallback 的特点是：

- 按异常分数构造一条模板化的阶段序列
- 基础模板大致为：`N2 -> N3 -> REM -> N2 -> N1`
- 高异常时插入更多 `WAKE / N1`

所以需要明确一个重要边界：

- 这条 fallback 能产出“可展示的阶段结果”
- 但它不是严格意义上的学习模型推理
- 如果部署环境没有激活 HTTP 模型 profile，系统展示出来的睡眠阶段，可能来自 fallback 而不是 ONNX 模型

## 4.7 Android 端实际部署模型：异常检测，不是完整睡眠分期

Android 侧模型实现：

- `core-ml/src/main/java/com/example/newstart/ml/SleepAnomalyDetector.kt`

资产路径：

- `ml/sleep_model.tflite`

来源脚本：

- `ml/export_android_sleep_model.py`

这条模型链的作用是：

- 输入 HR / SpO2 / HRV 聚合特征
- 输出单个 `anomaly_score`
- 用于端侧异常检测与离线兜底

它不输出 5 类睡眠阶段，因此必须在文档中写清楚：

- Android 当前本地模型是“睡眠异常检测模型”
- 不是“完整睡眠分期模型”

## 5. 端到端运行链路

## 5.1 云端标准链路

标准睡眠分析入口：

- `cloud-next/src/app/api/sleep/analyze/route.ts`

流程如下：

1. 客户端上传或触发 `sleepRecordId`
2. 云端将单窗特征写入 `sleep_windows`
3. 创建 `inference_jobs`
4. worker 读取活动模型 profile
5. 根据 profile 决定：
   - HTTP ONNX 推理
   - `local://baseline`
   - `local://ensemble`
   - 纯 fallback
6. 结果写回：
   - `sleep_stage_results`
   - `anomaly_scores`
   - `nightly_reports`

返回给上层页面的数据包括：

- `sleepStages`
- `sleepStages5`
- `recoveryScore`
- `sleepQuality`
- `insights`
- `anomalyScore`
- `factors`
- `modelVersion`

## 5.2 model_registry 对运行时的影响

代码位置：

- `cloud-next/src/lib/model-registry.ts`
- `cloud-next/src/lib/model-inference.ts`

当前仓库中的默认 profile 是：

- `version = "mmt-v1"`
- `featureSchemaVersion = "v1"`
- `runtimeType = "fallback"`
- `confidenceThreshold = 0.6`

这意味着：

- 如果数据库中没有激活一条更具体的 `model_registry` 记录
- 系统默认不是去调用 ONNX 服务
- 而是直接进入 fallback 运行模式

因此对外描述时必须区分：

- “仓库支持 ONNX 睡眠分期模型服务”
- “某个具体部署环境当前是否已经激活该模型”

## 6. 指标、性能与可写作表述

## 6.1 推荐引用的模型指标

如果需要描述“当前仓库里最成熟的睡眠分期模型表现”，优先引用 `AggregatedSleepTransformer` 的已提交指标：

- 测试集 Accuracy：`80.52%`
- 测试集 Macro F1：`80.27%`

原因：

- 这是仓库里性能最完整、与服务化链路最接近的一版
- 已有 ONNX 导出产物
- 已有服务化推理实现

## 6.2 不推荐直接写死的说法

不建议直接写：

- “当前 App 端本地就运行完整 5 类睡眠分期模型”
- “线上所有睡眠分期结果都来自 Sleep Transformer V2”
- “baseline 模型就是最终生产模型”

更准确的写法是：

- “项目已建立 5 类睡眠分期建模链路，并实现了从训练、导出到云端推理服务的完整技术路径。”
- “当前仓库中性能最成熟的主模型候选为聚合特征 Transformer，并已具备 ONNX 服务化部署形态。”
- “Android 端当前落地的是轻量异常检测模型；完整 5 类分期结果主要由云端分析链路提供。”

## 7. 当前存在的版本漂移与工程事实

截至当前源码，睡眠分期相关存在以下工程事实：

### 7.1 训练脚本与 artifact 漂移

- `train_baseline.py` 当前实现是随机森林
- `baseline_report.json` 却记录为 `NearestCentroid`
- `baseline_model.json` 为 centroid 形式

说明 baseline 产物不是与当前脚本完全同步生成的。

### 7.2 云端 local baseline 与训练 baseline 不同

- `local-baseline-inference.ts` 用的是 17 维 centroid 最近邻式推断
- `train_baseline.py` 用的是 51 维随机森林

两者不能被当成同一份模型。

### 7.3 Android 端与云端职责不同

- Android 端：异常检测
- 云端：完整睡眠分期与夜间报告

这意味着端云协同是当前睡眠分析的真实形态。

## 8. 推荐对外描述版本

如果需要给其他 AI、官网、技术报告或比赛材料使用，推荐采用下面这段总结：

“长庚环的睡眠分期能力采用 5 类分期体系（WAKE、N1、N2、N3、REM），以 30 秒窗口为基础，从心率、血氧、HRV、体温、运动、PPG 与边缘异常信号中构建多模态特征。仓库中已实现基线机器学习模型、聚合特征 Transformer、ONNX 云端推理服务以及端侧轻量异常检测模型。其中，聚合特征 Transformer 是当前性能最成熟、最接近实际部署的睡眠分期主模型候选，测试集 Accuracy 约 80.5%，Macro F1 约 80.3%。当前产品形态采用端云协同：Android 端负责轻量异常检测与离线兜底，完整的 5 类睡眠分期结果主要通过云端睡眠分析链路生成。” 

## 9. 文档使用提醒

后续若继续基于本仓库撰写任何睡眠分期相关文案，建议遵循以下原则：

1. 先确认部署环境 `model_registry` 是否已激活 HTTP 推理模型。
2. 如果要写“当前线上能力”，必须区分“支持能力”与“当前默认配置”。
3. 如果要写“端侧模型”，必须明确它是异常检测，不是完整睡眠分期。
4. 如果要写“模型效果”，优先引用 `AggregatedSleepTransformer` 的测试指标。
5. 如果要写“baseline”，必须说明 baseline 在仓库内存在版本漂移，只可作历史或对照参考。
