## 3. 系统架构说明

### 3.1 架构分层

- 端侧采集层：`BleConnectionManager`, `DataCollectionService`
- 端侧数据层：Room（`AppDatabase` + DAO + Entity）
- 端侧智能层：`SleepAnomalyDetector`（TFLite 优先 + 规则回退）
- Android AI 服务层：`app/service/ai/*`，用于隔离 UI 与具体模型客户端
- 云侧 API 层：Next.js Route Handlers
- 云侧数据层：Supabase Postgres + RLS
- 云侧推理编排：`/api/v1/inference/nightly` + `/api/internal/worker/run`
- 云侧生成 provider 层：Vector Engine（primary） + OpenRouter（secondary） + DeepSeek（tertiary）

## 4. 模块结构与依赖关系

### 4.1 Android 关键模块

- `bluetooth` -> 负责 BLE 连接、协议编解码、数据回调
- `service` -> 后台前台服务持续采集与落库
- `repository` -> 聚合 DAO 与网络请求，向 ViewModel 提供统一接口
- `ui/*` -> 按页面分包，`MorningReportViewModel` 负责晨报主流程

- `service/ai/*` -> 结构化文本、本地检索、本地 OCR、本地异常检测、本地干预规划服务封装

依赖主链：`UI/ViewModel -> Repository -> (Room DAO + ApiService)`。

### 4.2 云端关键模块

- 认证：`requireAuth` 解析 Bearer token 并校验 Supabase 用户
- 业务 API：`/api/sleep/*`, `/api/report/*`, `/api/intervention/*`, `/api/sync`
- 推理任务：`/api/v1/inference/nightly`（入队）+ `/api/internal/worker/run`（执行）
- 模型运行时：`model-registry` + `model-inference`（支持 HTTP 与本地 fallback）
- AI provider env：`PRESCRIPTION_PROVIDER_PRIMARY/SECONDARY/TERTIARY` + `VECTOR_ENGINE_*` + `OPENROUTER_*`

## 5. 数据流

### 5.1 端侧数据流

1. 戒指数据经 BLE 回调进入 `DataCollectionService`。  
2. 采样节流后分别写入 `health_metrics` 与 `ppg_samples`（Room）。  
3. `MorningReportViewModel` 读取最新睡眠与指标，调用 `SleepAnomalyDetector` 计算异常分。  
4. 本地生成恢复分与建议；在用户登录云端后触发最小闭环上传与分析。

### 5.2 云侧数据流

1. `POST /api/sleep/upload` 写入 `sleep_sessions`。  
2. `POST /api/sleep/analyze` 写入/更新窗口数据并触发推理编排。  
3. `POST /api/v1/inference/nightly` 创建 `inference_jobs`。  
4. `POST /api/internal/worker/run` 消费队列，写入 `sleep_stage_results`、`anomaly_scores`、`nightly_reports`。  
5. 客户端通过 `GET /api/v1/reports/nightly/:sleepRecordId` 或兼容接口读取结果。

## 6. 核心算法与逻辑

### 6.1 端侧异常检测（`SleepAnomalyDetector`）

- 优先加载 `app/src/main/assets/ml/sleep_model.tflite`
- 输入特征：HR、SpO2、HRV、温度派生统计
- 若模型缺失/推理失败：使用规则评分回退
- 输出：`anomalyScore`、`AnomalyLevel`、`confidence`、主导风险因子

### 6.2 PPG 峰值分析（`PpgPeakDetector`）

- 峰值阈值 + prominence + 最小峰距检测
- 输出 RR 间期、估算心率、RMSSD、信号质量
- 单元测试：`PpgPeakDetectorTest`

### 6.3 云侧推理与门控（`/api/internal/worker/run`）

- 队列状态机：`queued -> running -> succeeded/failed`
- 信心门控：低于 `confidenceThreshold` 的分段降级到 `N1`
- fallback：模型失败时走规则推理并标记 `model_fallback`

### 6.4 AI 路由（当前事实）

- 端侧 UI 已不再直接调用 `OpenRouterDoctorApiClient`、`SleepAnomalyDetector`、`EdgeLlmOnDeviceModel` 或 `MedicalReportParser`
- 这些调用已通过 `app/service/ai/*` 服务层收口
- 云端处方生成已接入 `VectorEnginePrescriptionProvider`
- Gemini 活跃路径已清退，不再作为当前架构的一部分

## 8. 部署方式

### 8.1 Android

- Debug 构建：`gradlew.bat :app-shell:assembleDebug`
- Release 构建：`gradlew.bat :app-shell:assembleRelease`（需 `keystore.properties`）
- API 地址通过 `DEBUG_API_BASE_URL` / `RELEASE_API_BASE_URL` 注入 `BuildConfig`

### 8.2 cloud-next

- 本地开发：`cd cloud-next && npm run dev`
- 构建验证：`cd cloud-next && npm run build`
- 代码静态检查：`cd cloud-next && npm run lint`
- 环境变量：`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`
- AI 相关变量：
  - `PRESCRIPTION_PROVIDER_PRIMARY`
  - `PRESCRIPTION_PROVIDER_SECONDARY`
  - `PRESCRIPTION_PROVIDER_TERTIARY`
  - `VECTOR_ENGINE_API_KEY`
  - `VECTOR_ENGINE_TEXT_FAST_MODEL`
  - `VECTOR_ENGINE_TEXT_STRUCTURED_MODEL`
  - `OPENROUTER_API_KEY`
