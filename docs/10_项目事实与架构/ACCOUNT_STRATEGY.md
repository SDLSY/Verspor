# 账户策略

## 目标

- 不在代码、资源文件或文档中预置可登录的真实账号。
- 区分正式用户、内部演示账号和自动化测试账号，避免数据混用。
- 所有账号都通过 Supabase Auth 管理，不使用共享密码。

## 账号分层

### 正式用户账号

- 来源：Android 端注册页或网页注册页。
- 用途：真实用户登录、设备同步、建议追踪、效果回看。
- 要求：
  - 使用真实邮箱。
  - 如开启邮箱确认，必须先完成确认再登录。
  - 不与演示或测试数据混用。

### 内部演示账号

- 来源：由管理员在 Supabase Dashboard 手动创建。
- 用途：路演、演示、教学、非自动化测试。
- 命名建议：
  - `demo+姓名缩写@你的域名`
  - 例如 `demo+team01@vespero.example`
- 要求：
  - 独立数据集。
  - 定期重置密码。
  - 不作为开发调试默认账号写入代码或文档。

### 自动化测试账号

- 来源：测试脚本或管理员临时创建。
- 用途：接口联调、回归测试、数据链路验证。
- 命名建议：
  - `test+时间戳@你的域名`
  - 例如 `test+20260310_1015@vespero.example`
- 要求：
  - 测试完成后删除或停用。
  - 不复用正式用户数据。
  - 不用于人工长期登录。

## 管理规则

- 真实账号列表统一在 Supabase Dashboard 的 `Authentication -> Users` 查看。
- 不在仓库内保存邮箱、密码、access token。
- 不在截图、路演材料、日志中暴露真实账号信息。
- 若需要演示固定流程，使用内部演示账号，不使用真实用户账号。

## 当前项目约束

- Android 未登录状态只显示访客文案，不再显示 `user@example.com` 这类占位邮箱。
- 注册与登录统一走云端接口：
  - `POST /api/auth/register`
  - `POST /api/auth/login`
- 建议解释、效果回看、云端同步都依赖真实登录态。

## 推荐流程

1. 正式用户：在 App 或网页注册，完成邮箱确认后登录。
2. 路演演示：使用管理员创建的演示账号。
3. 接口测试：创建一次性测试账号，测试结束后清理。
