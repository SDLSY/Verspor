# 戒指通讯协议使用说明与阶段性总结

本文基于当前仓库实现与本轮迭代对话整理，作为联调、验收与后续迭代的统一参考。

## 1. 适用范围
- 设备通信协议：Hi90B / Clevering 自定义 BLE 协议（`FFFA/FFFC/FFFB`）。
- Android 端实现入口：`app/src/main/java/com/example/newstart/bluetooth/BleManager.kt`。
- 命令与编解码：`Hi90BCommandBuilder.kt`、`Hi90BFrameParser.kt`、`Hi90BWorkParams.kt`、`CleveringCommandSender.kt`。

## 2. BLE 通信总览
### 2.1 GATT 约定
- 服务 UUID：`0000FFFA-0000-1000-8000-00805F9B34FB`
- 通知特征：`0000FFFC-0000-1000-8000-00805F9B34FB`
- 写入特征：`0000FFFB-0000-1000-8000-00805F9B34FB`
- CCCD：`00002902-0000-1000-8000-00805F9B34FB`

### 2.2 帧格式
- 通用帧：`AA 55 | LenH LenL | CMD | DATA... | CHK`
- `Len = CMD(1) + DATA(n) + CHK(1)`
- `CHK = LenH ^ LenL ^ CMD ^ DATA...`
- 字节序：大端（Big Endian）

### 2.3 上传数据帧 DATA 头
- 结构：`type(1) + taskSeq(4) + packetNo(4) + ts(6) + payload(m)`
- `packetNo == 0xFFFFFFFF` 代表结束包（不解析 payload）。

## 3. 指令字典（当前实现）
| CMD | 含义 | 方向 | 说明 |
|---|---|---|---|
| `0x20` | 查询固件版本 | App -> Ring | 读取版本号 |
| `0x21` | 设置时钟 | App -> Ring | 下发年月日时分秒 |
| `0x22` | 查询电量 | App -> Ring | 返回充电状态与电量 |
| `0x30` | 设置工作参数 | App -> Ring | 周期采集配置（最多 8 组） |
| `0x31` | 读取工作参数 | App -> Ring | 读取当前生效参数 |
| `0x32` | 即时任务配置 | App -> Ring | 立即采集任务，含持续时长 |
| `0x33` | 查询历史数据量 | App -> Ring | 返回历史数据条目数 |
| `0x34` | 启停数据传输 | App -> Ring | `01` 启动，`00` 停止 |
| `0x50` | 周期上传 | Ring -> App | 上报周期采样数据 |
| `0x52` | 即时上传 | Ring -> App | 上报即时采样数据 |

## 4. 数据类型 type 映射（`0x50/0x52`）
| type | 含义 | payload 解析 |
|---|---|---|
| `0x01` | PPG | 支持 `channel+rate+N*(IR3+RED3)` 与 `N*3byte` 两种格式 |
| `0x02` | 心率 | `uint16` |
| `0x03` | 心率+血氧 | `HR(uint16)+SpO2(uint16)` |
| `0x04` | HRV | `uint16` |
| `0x05` | 体温 | `int16 * 0.1` ℃ |
| `0x06` | 加速度 | `XYZ`，3 轴各 `int16` |
| `0x07` | 陀螺仪 | `XYZ`，3 轴各 `int16` |
| `0x08/0x81` | 步数 | 当前 App 策略为忽略，仅日志记录 |
| `0x80` | 运动组合 | 优先按 `ACC(6)+GYRO(6)` 组包解析 |

## 5. 推荐初始化流程（当前线上代码）
当前逻辑位于 `BleManager.initAfterNotifyEnabled()`，顺序如下：
1. `0x34 stop` 停止传输，清理现场。
2. 对 `[1,3,4,5,6,0x80]` 下发 `0x32 duration=0` 清理旧即时任务。
3. 下发 `0x21` 同步时钟。
4. 下发 `0x22` 查询电量。
5. 下发 `0x30` 写入“生命体征优先”工作参数。
6. 下发 `0x31` 回读参数核验。
7. 下发 `0x33` 查询历史数据量。
8. 下发 `0x34 start` 开始传输。
9. 延迟 5 秒下发一次 `0x32 item=0x80` 运动 Burst（500Hz，10s）。

## 6. 当前默认采集策略（生命体征优先）
### 6.1 周期任务（`0x30`）
- `item=3`（心率+血氧）：`channel=0x0003`，`rate=5`，`period=30s`，`duration=12s`
- `item=5`（体温）：`period=120s`
- `item=4`（HRV）：`period=60s`
- `item=6`（运动）：`channel=0x0007`，`rate=25`，`period=60s`，`duration=15s`

### 6.2 即时任务（`0x32`）
- 默认不再以 PPG 即时任务作为主入口。
- 额外保留运动 Burst：`item=0x80`，`channel=0x0030`，`rate=0x01F4(500Hz)`，`duration=10s`。

## 7. 联调与验收要点
### 7.1 成功判据
- 连接后可见 `0x31` 参数回读日志。
- `0x33` 返回历史数据量（可为 0 或正数）。
- `0x34` 返回传输状态响应。
- 持续收到 `0x50/0x52` 且 `type` 覆盖生命体征项（`0x03/0x04/0x05/0x80`）。

### 7.2 常见问题与处理
- 只收到 `type=3`，无 PPG：
  - 先确认当前策略是否本就非 PPG 优先；这是预期行为。
  - 若需 PPG，请显式下发 `item=1` 即时任务或切换到 PPG profile。
- 出现 `writeCharacteristic failed`：
  - 检查是否并发写太密，保持命令间隔与重试机制。
  - 当前实现已使用队列、写间隔、重试与 `NO_RESPONSE` 兜底。
- 历史数据量长期为 0：
  - 检查是否已经 `0x34 start`，以及设备端是否开启对应任务。
- 步数无数据显示：
  - 当前产品策略为“步数不采集”，运动强度由加速度/陀螺仪估算。

## 8. 阶段性总结（基于本轮迭代）
### 8.1 P0 阶段
- P0-2：补齐 BLE `0x01/0x80` 解析路径，运动数据解析增强。
- P0-3：统计逻辑去硬编码，切换数据库计算。
- P0-4：“最佳睡眠日”改为“综合恢复分最高”，采用跨表计算。
- 演示模式：改为可控，不默认强开。
- P0-5：完成云端最小闭环（登录 -> 上传 -> 分析 -> 建议 -> 同步），当前主线为 `cloud-next` 联调。

### 8.2 P1 阶段
- 端侧 AI 真接入（TFLite），替代纯规则占位。
- 晨报接入 AI 可信度、主导因子与解释链路。
- 风险提示 UI 从“仅文字”升级为“横幅+风险等级+风险分+卡片高亮”。

### 8.3 近期优化（本轮）
- 采集策略从 PPG 优先切换为生命体征优先。
- 趋势页增加异常值过滤与曲线平滑优化，减少错误点干扰。
- 运动强度改由运动传感器估算，不依赖步数采集。

## 9. 即时任务冲突状态机（已回退，当前未启用）
当前版本已回退到“无仲裁直发”模式，`0x32` 即时任务不再统一入队。

### 9.1 当前行为
1. 清任务（`duration=0`）直接下发。
2. PPG fallback 即时任务直接下发。
3. 运动 Burst（`0x80`）直接下发。
4. 不再维护 `IDLE/SWITCHING/RUNNING` 状态机与优先级队列。

### 9.2 回退原因
- 现场日志显示在部分机型/固件组合下，初始化阶段存在“清任务延后执行，反向清掉刚设置任务”的时序风险。
- 回退后优先保证数据链路稳定与可观测，再逐步重引入更稳健的仲裁方案。

## 10. 下一步建议（协议侧）
1. 增加“协议 profile 开关”：
`vital_sign_first` / `ppg_first` / `motion_lab`，便于现场快速切换。
2. 完善“任务配额与时间片”：
在状态机上增加每类任务最大占用时长，避免某一类任务长期饿死其他任务。
3. 完善协议回归测试：
基于录包回放验证 `0x30~0x34` 与 `0x50/0x52` 的稳定性。
