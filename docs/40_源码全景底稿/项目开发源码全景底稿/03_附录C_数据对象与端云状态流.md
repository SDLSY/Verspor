# 标题
附录 C：数据对象与端云状态流

- 适用任务：数据库说明、数据流梳理、同步链路写作
- 阅读优先级：高
- 是否允许对外直接引用：允许，但必须保留“本地 / 云端 / 演示灌库”区分

## 1. 数据层的总体结构

项目的数据层不是单一数据库，也不是只靠接口返回临时 UI 数据。当前真实结构至少有三层：

1. Android 本地 Room
2. 云端 Supabase 结构化数据
3. Android 与云端之间的 DTO / contract / bootstrap payload

这些层共同服务于：

- 页面展示
- 历史记录
- 建议生成
- 干预执行回写
- 演示账号灌库

## 2. Android 本地数据对象

按当前实体与 DAO 分布，本地 Room 至少覆盖这些对象域：

- 睡眠记录
- 健康指标
- PPG 样本
- 设备记录
- 医生消息 / 会话 / 评估
- 医检报告 / 医检指标
- 干预任务 / 执行
- 画像快照
- 处方 bundle / item
- 放松会话
- 恢复分
- 评估量表
- 药物识别记录
- 饮食分析记录

这些对象并不是每个页面都直接访问，但它们构成了 Android 端“无网时仍有内容”的基础。

## 3. 云端结构化数据

按 `cloud-next` API 与迁移可知，云端当前至少维护这些结构化业务域：

- sleep sessions / nightly reports
- recovery / trend / period summary
- medical reports / medical metrics
- doctor inquiry summaries
- intervention tasks / executions
- recommendation traces / effect tracking
- model registry / inference jobs
- medication analysis records
- food analysis records
- assessment baseline snapshots
- audit events

这些对象既服务 Android，也服务后台患者工作台和运维页面。

## 4. 端云状态流的几条主链

### 4.1 采集数据主链

戒指 -> BLE 解析 -> 本地指标落库 -> 今日页 / 趋势页 -> 条件允许时上传云端

这条链的意义是：

- Android 页面不必等云端才能有基本数据
- 云端分析又可以消费结构化上传结果

### 4.2 医检报告主链

导入 PDF / 图片 -> OCR / 理解 -> 本地或云端结构化结果 -> 可读报告 -> 医生问诊 / 干预

关键点：

- 报告理解不是最终页，而是一个中间结构化环节
- 它后面会影响问诊、时间线、后台报告队列

### 4.3 干预与执行主链

建议或处方 -> 干预任务 -> 执行会话 -> 执行记录 -> 效果回写 -> 趋势 / 后台

这里当前已经支持 metadata 扩展，因此干预类型、是否启用触觉、是否有实时反馈等都能被记录。

### 4.4 演示 bootstrap 主链

演示账号登录 -> `/api/demo/bootstrap` -> payload 下发 -> 本地 Room 清库 / 灌库 -> 页面立即进入可演示状态

这条链说明 Android 页面的很多内容不是完全靠线上实时拉取，而是支持“先有本地事实层，再有页面展示层”。

## 5. DTO 与 contract 的作用

`contracts/` 目前规模不大，但角色清晰：它是 Android / 云端 / 训练和导出链之间的共享契约层，承担“统一字段语义”的职责。

在整体项目里，它的价值不是数量，而是：

- 减少端云写法漂移
- 为 AI 能力、推理任务和结构化结果提供稳定类型基础

## 6. 本地优先与云端增强

当前项目很多能力都不是“要么纯本地，要么纯云端”，而是：

- 本地先提供基本可用
- 云端补更强结构化和解释
- 失败时保留回退路径

在写开发文档时，必须把这些能力按这个模式描述：

- 睡眠与健康指标：本地优先，云端增强
- 医检报告：本地初步整理，云端理解增强
- 药物 / 饮食：云端解析主导，记录回写本地与云端
- 问诊：端侧组织交互，云端增强推理
- TTS：云端合成主导，本地缓存播放

## 7. 演示账号为什么必须单独写

演示体系对当前项目不是附属物，而是数据层的重要特例。因为很多页面优先消费本地 Room，所以如果没有演示 bootstrap，云端 seed 过的数据不一定能直接出现在 Android 端。

因此，开发文档必须单独写清：

- 云端 seed 负责创建账号和云端结构化数据
- Android bootstrap 负责把演示快照灌入本地库
- 同一套 demo 账号同时支撑 App 和后台

## 8. 当前必须强调的数据层边界

1. Room 不只是缓存，而是当前 Android 主页面的重要事实源。
2. 云端 Supabase 不只是存用户账号，而是结构化业务中枢。
3. DTO / contract / bootstrap payload 是把“同一事实”传到不同层的关键桥梁。
4. 演示数据链路是项目正式演示能力的一部分，必须作为开发事实写入。
