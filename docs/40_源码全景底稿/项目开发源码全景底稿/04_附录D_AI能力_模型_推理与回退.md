# 标题
附录 D：AI 能力、模型、推理与回退

- 适用任务：AI 架构说明、技术研究稿支撑、能力边界说明
- 阅读优先级：高
- 是否允许对外直接引用：允许，但必须严格保留端云边界与模型状态说明

## 1. AI 在项目中的位置

当前项目的 AI 不是“单一大模型入口”，而是一组分层能力：

- 本地轻量模型与规则
- 云端提示词与结构化编排
- 多 provider 路由
- 语音、图片、报告、问诊、推荐、机器人讲解等不同能力栈

因此任何开发文档都不能用一句“项目接入了某某大模型”概括全部实现。

## 2. 睡眠模型体系

### 2.1 当前真实分层

当前仓库中的睡眠模型至少有四层：

1. 训练层：`ml/`
2. ONNX 导出与服务层：`hf-inference/`
3. 云端推理与回退层：`cloud-next`
4. Android 端轻量异常检测层：`core-ml`

### 2.2 聚合特征 Transformer

当前仓库里最重要的睡眠分期主模型候选是：

- `AggregatedSleepTransformer`

对应事实链：

- 训练脚本：`ml/train_aggregated_transformer.py`
- 模型定义：`ml/AggregatedTransformer.py`
- ONNX 导出：`ml/export_aggregated_onnx.py`
- 已提交训练产物：`ml/artifacts/week2/aggregated_transformer/aggregated_report.json`

当前可核对指标：

- 测试集 Accuracy：约 `80.52%`
- 测试集 Macro F1：约 `80.27%`

这些值来源于训练脚本自动评估与写入的 JSON 产物，不是手工填写。

### 2.3 baseline 与研究层

仓库中也保留了：

- baseline 传统机器学习模型
- 更重的多模态 Transformer 原型

因此文档应该写成：

- “仓库支持多种睡眠分期建模路径”

而不是：

- “当前线上默认只用某一个单模型”

### 2.4 Android 端当前主链

Android 当前真实主链不是完整五阶段主模型，而是：

- 轻量异常检测
- 本地健康分析和兜底

所以正确写法是：

- “五阶段睡眠分期能力主要通过云端分析链路生成”

而不能写：

- “Android 本地已默认运行完整五阶段主模型”

## 3. 报告理解与问诊

### 3.1 医检报告理解

云端当前承担：

- OCR 后结构化理解
- 可读报告生成
- 指标提取与异常解释

Android 承担：

- 文件导入
- 本地整理与展示
- 失败时回退

### 3.2 医生问诊

医生页不是简单聊天页，当前已包含：

- 问诊轮次推进
- 结构化评估
- 问诊单生成
- 本地与云端双路兜底

当前必须强调：

- 问诊生成链已落地
- 但它属于健康辅助与信息整理，不应写成医疗诊断替代

## 4. 图片理解：药物与饮食

当前药物与饮食图片分析已经接入云端多模态理解，不再停留在本地 OCR 方案。

药物链：

- 图片 -> 药名 / 用途 / 风险提示 / 建议

饮食链：

- 图片 -> 餐次 / 食物项 / 热量 / 营养结构 / 建议

它们后续会影响：

- 画像
- 恢复分解释
- 建议链
- 后台时间线

## 5. 桌面机器人与讲解层

当前桌面机器人链路包括：

- 页面上下文构建
- 云端 narration 文案生成
- 本地 / 云端回退
- TTS 合成
- 本地缓存播放

因此必须写清：

- 它不是独立 3D 模型展示
- 而是和页面、问诊、讲解、播报深度绑定

## 6. TTS 当前状态

当前 TTS 主路径已经切到豆包语音接口，经由 `cloud-next` 代理，Android 侧主要负责请求和播放。

历史讯飞 TTS 相关代码仍然保留痕迹，但已经不是当前主播报路径。写文档时应强调：

- 主路径：云端豆包 TTS
- Android：云端合成结果的播放与缓存
- 历史讯飞：兼容遗留，不是当前主讲解链事实

## 7. AI 能力写作模板

后续任何 AI 能力都建议按这个模板写：

1. 这项能力解决什么问题
2. 输入是什么
3. 输出是什么
4. 在 Android 端、云端还是两者协同
5. 是否有回退
6. 当前是否为主链

## 8. 当前必须禁止的误写

- 把所有 AI 能力归结为一个模型
- 把 Android 端写成承担全部推理
- 把机器人写成单纯 UI 装饰
- 把药物 / 饮食识别写成独立于画像和恢复分的孤立工具
- 把问诊和报告理解写成医疗确诊系统
