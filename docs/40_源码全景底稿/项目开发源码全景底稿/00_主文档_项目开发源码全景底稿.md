长庚环《项目开发文档》源码全景底稿

- 适用任务：AI 投喂、项目开发文档写作、技术答辩准备、功能链路核对
- 阅读优先级：最高

## 1. 本文档是什么

这不是一份比赛排版稿，也不是宣传文案，而是一份以当前仓库源码为唯一事实基线的开发全景底稿。它服务两个目标：

1. 让另一个 AI 或新成员在不依赖口头说明的情况下，快速建立对项目当前状态的正确理解。
2. 为后续撰写正式《项目开发文档》、技术研究报告、测试文档、答辩讲稿提供可追溯事实层。

本文档默认只把仓库自有源码视为事实来源，主要覆盖：

- Android 现行运行链：`app-shell`、`core-*`、`feature-*`
- 云端与后台：`cloud-next/src`、`cloud-next/scripts`、`cloud-next/supabase`
- 共享与训练层：`contracts`、`ml`、`hf-inference`
- 历史归档：`app/`

明确不把以下目录写入“项目真实实现”层：

- `cloud-next/node_modules`
- `cloud-next/.next`
- `.gradle`
- 各类 `build/`
- 日志、截图、数据库导出、缓存目录、临时目录
- 第三方大资源包和 SDK 附带目录

## 2. 当前项目的准确定义

长庚环不是一个“只展示睡眠图表”的 App，也不是单一的睡眠监测壳。按当前代码，它更准确的定义是：

> 一个以智能戒指为生理感知入口、以 Android 应用为用户交互中心、以云端 AI 与后台工作台为增强能力、围绕评估、建议、执行、复盘构建端云协同闭环的健康辅助系统。

当前主线不是“展示更多监测数字”，而是把多源健康数据收敛成一条可执行、可回写、可在后台追踪的干预闭环。代码里真正贯穿 Android 与云端的主叙事是：

`统一画像 -> 处方/建议 -> 执行 -> 反馈 -> 再进入画像`

项目当前已经覆盖的能力不只包括睡眠分析，还包括：

- 戒指连接、BLE 数据采集、本地落库
- 今日恢复分与周期趋势
- 医检报告 OCR / 可读报告
- 医生问诊与问诊单生成
- 症状引导、基线评估、干预中心、执行、复盘
- 药物识别与饮食分析
- 桌面机器人讲解与 TTS 播报
- 云端后台患者工作台、建议闭环、报告分流、运维页
- 演示账号、seed 数据、Android 登录后 bootstrap 灌库

## 3. 仓库当前结构与事实层级

### 3.1 Android 现行结构

当前 Gradle 实际编译的 Android 模块由 `settings.gradle.kts` 决定，只有：

- `:app-shell`
- `:core-common`
- `:core-model`
- `:core-data`
- `:core-ble`
- `:core-network`
- `:core-db`
- `:core-ml`
- `:feature-home`
- `:feature-device`
- `:feature-doctor`
- `:feature-relax`
- `:feature-trend`
- `:feature-profile`

其中：

- `:app-shell` 是唯一 Android 运行入口。
- `core-*` 是共享基础层。
- `feature-*` 是业务特性层。
- `app/` 已经不在当前 Gradle 运行链中，它是历史单体归档，用于映射、追溯、核对迁移关系。

### 3.2 云端与后台结构

`cloud-next/` 是当前唯一的云端主线，采用：

- Next.js App Router
- Route Handlers 形式的 API
- Supabase 作为认证、数据库与管理端数据源
- `src/app` 承载页面和 API 路由
- `src/lib` 承载后台聚合、AI 编排、推理、权限、推荐、报告等逻辑
- `scripts/demo-accounts/seed.mjs` 承载演示账号种子工具

### 3.3 训练与模型结构

项目并不是单一云模型调用，而是保留了训练、导出、服务、端侧轻量兜底四层：

- `ml/`：训练、评估、导出脚本
- `hf-inference/`：ONNX 睡眠推理服务
- `cloud-next/src/lib/ai` 与 `cloud-next/src/lib/model-*`：云侧 AI 编排和模型运行时
- `core-ml/`：Android 端轻量模型、本地解析与回退

## 4. Android 当前真实运行链

### 4.1 入口与应用壳

现行 Android 真正的入口类是：

- `app-shell/src/main/java/com/example/newstart/SleepHealthApp.kt`
- `app-shell/src/main/java/com/example/newstart/MainActivity.kt`

应用启动后，`SleepHealthApp` 负责初始化应用级能力，包括数据库、仓储、云端会话、演示 bootstrap 等。`MainActivity` 负责：

- 承载主导航
- 管理底部标签页
- 管理桌面机器人和讲解层
- 处理页面标题、副标题、页面 key 与机器人讲解文案上下文

也就是说，当前 App 的顶层壳不是传统“单纯导航 + Fragment Container”，而是“导航 + 全局数字人 + 页面讲解 + 音频播报”的复合壳。

### 4.2 主导航与页面分布

当前主导航资源位于：

- `core-common/src/main/res/navigation/mobile_navigation.xml`

主路径页面分布为：

- 今日：`feature-home/ui/home`
- 设备：`feature-device/ui/device`
- 医生：`feature-doctor/ui/doctor`
- 趋势：`feature-trend/ui/trend`
- 我的：`feature-profile/ui/profile`

此外，干预与健康辅助链路大量位于：

- `feature-relax/ui/relax`
- `feature-relax/ui/intervention`

这部分承载了：

- 症状引导
- 医检报告分析
- 药物识别
- 饮食分析
- 呼吸训练
- Zen 轻交互
- 干预中心
- 执行会话
- 复盘

### 4.3 当前页面主线

按源码组织，Android 用户可见主线至少有 5 条：

1. 今日页主线  
   睡眠摘要、健康指标、恢复分、建议与解释。
2. 医生主线  
   医生问诊、问诊单生成、评估快照、桌面机器人讲解。
3. 趋势主线  
   近 7 天 / 近 30 天趋势、周报 / 月报、干预效果总结。
4. 设备主线  
   戒指扫描、连接、采集、生理数据刷新。
5. 我的主线  
   云端账户、资料页、账户设置四页、通知 / 隐私 / 关于。

## 5. 端云协同的关键闭环

### 5.1 戒指 -> 采集 -> 今日页 -> 趋势

采集侧代码主要位于：

- `app-shell/service/DataCollectionService.kt`
- `feature-device/ui/device/DeviceViewModel.kt`
- `core-ble`
- `core-db`

它的目标不是单纯显示“已连接”，而是把实时或准实时生理数据写进本地数据库，再被今日页与趋势页消费。当前系统已经修正了“离开设备页就断采集”和“今日页无睡眠记录时整页 mock 覆盖真实指标”的问题，因此这条链已经是当前真实主链之一。

### 5.2 医检报告 -> 可读报告 -> 问诊 -> 干预

报告链路跨 Android 和云端：

- Android：`MedicalReportAnalyzeFragment/ViewModel`
- 云端：`/api/report/understand`
- 医生：`DoctorChatFragment/ViewModel`
- 干预：`InterventionCenterFragment`

这里的关键不是 OCR 本身，而是“结构化理解结果是否进入问诊与干预”。当前代码已经把“本地回退”和“云端理解失败”的边界处理清楚，并将可读报告与医生问诊、问诊单生成、后续干预连接起来。

### 5.3 药物/饮食 -> 画像 -> 恢复分 -> 建议

生活方式链路位于：

- Android：`MedicationAnalyze*`、`FoodAnalyze*`
- 云端：`/api/medication/analyze`、`/api/food/analyze`
- 后续记录：`.../records/upsert`

图片解析当前走云端多模态分析链，不再做本地 OCR 两段式。药物与饮食并不是孤立工具页，而是被设计成画像、恢复分解释和后续建议的输入因素。

### 5.4 干预执行 -> 复盘 -> 趋势/后台时间线

干预主线位于：

- `InterventionCenterFragment`
- `InterventionSessionFragment/ViewModel`
- `RelaxReviewFragment/ViewModel`
- `SleepTrendViewModel`

这条链路现在已经不止有音频或倒计时，还包括：

- 实时生物反馈视觉化呼吸训练
- 触觉节拍
- Zen 轻交互
- 动态音景干预

干预执行会写入本地与云端的执行记录，并进一步影响复盘、趋势、画像和建议。

### 5.5 演示账号 -> bootstrap -> 本地灌库 -> App / 后台观演

这是当前项目中很重要但容易被忽略的一条链：

- 云端 seed 工具创建演示账号与场景数据
- 登录 demo 账号后，Android 调 `/api/demo/bootstrap`
- `DemoBootstrapCoordinator` 把云端快照灌入本地 Room
- App 与后台都能立即进入“可讲解状态”

这条链路说明：项目并不只面向真实运行时，也已经实现了稳定演示与评审展示所需的数据准备机制。

## 6. AI 能力的当前真实形态

### 6.1 不是单一路径，不是单一 provider

项目的 AI 能力并非统一由某一个云模型承担。当前代码里至少同时存在：

- 云端 AI 编排
- OpenAI 风格兼容 provider 路由
- OpenRouter / Vector Engine / DeepSeek 等上游
- 豆包 TTS
- 本地轻量模型或规则回退

因此后续任何文档都不能写成“系统所有 AI 都由同一个模型完成”。

### 6.2 睡眠模型是分层体系

睡眠分析层不是单一模型文件，而是分层体系：

- 训练层：`ml/`
- ONNX 推理层：`hf-inference/sleep-transformer-v2.onnx`
- 云端模型运行时与回退：`cloud-next`
- Android 端当前真实主链：轻量异常检测与离线兜底，而不是完整五阶段主模型

这意味着：

- 可以说“项目支持五阶段睡眠分期建模”
- 但不能写“Android 端当前本地直接跑完整五阶段主模型”

### 6.3 桌面机器人不是装饰层

桌面机器人当前承担三类工作：

- 页面讲解
- 点击交互生成文案
- TTS 语音播报

它背后连接了：

- 页面上下文拼装
- 云端 `avatar/narration`
- 本地或云端回退
- 语音合成与缓存播放

因此在项目定位里，它更接近“交互层解释器”，不是 UI 装饰。

## 7. current 与 legacy 的边界

`app/` 中的大量类与现行模块仍然保留同名、同包或近似实现，但不能再把它当作运行入口。对后续 AI 来说，这个边界必须写死：

- `app-shell + core-* + feature-*`：现行实现
- `app/`：历史单体归档

`app/` 仍然有价值，但价值主要是：

- 查看旧单体如何组织业务
- 做迁移映射
- 对照历史实现

它不再代表当前 APK 的编译和运行事实。
