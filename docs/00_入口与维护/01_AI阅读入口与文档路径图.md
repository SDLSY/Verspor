# AI 阅读入口与文档路径图

## 1. 文档目的

本文档是整个仓库面向 AI 的总入口文档。它的目标不是介绍单个模块，而是为后续任何大模型、自动写作助手、代码理解系统或文档生成工具提供一条稳定的阅读路径，使其能够：

1. 快速建立对长庚环项目的整体认知。
2. 明确哪些文档属于第一事实源，哪些只适合作为专项补充。
3. 把“系统概览、技术架构、专题事实、模块边界、云端接口、关键数据流、配置机制、写作素材层”按正确顺序拼起来。
4. 避免把模板、历史总结、规划材料误写成当前已上线事实。

## 2. 阅读总原则

### 2.1 先读 AI 基线文档，再读工程、写作和历史材料

当前仓库已经整理出一套面向 AI 的基线文档。这些文档更适合模型读取，也更适合后续填充到计划书、开发文档、技术研究报告和测试文档中。

推荐顺序应当是：

1. 先读基线事实层
2. 再读写作补充层
3. 再读工程核对层
4. 最后按需读模块入口和专项材料

### 2.2 代码事实优先于叙述性材料

当文档之间出现冲突时，优先级如下：

1. 当前运行代码与当前运行入口
2. AI 基线文档
3. 写作补充文档中的统一口径
4. 当前维护中的工程事实文档
5. 模块 README
6. 阶段总结、模板和展示材料

### 2.3 不要把研究规划和历史材料写成当前上线事实

仓库中存在多类高价值但不应直接写成“当前上线真相”的材料，例如：

- 研究性推荐模型文档
- 历史 BLE 文档
- 模板文档
- 阶段总结与完成版材料

这些材料可以作为补充和写作参考，但不应抢在事实层之前输入。

## 7. AI 使用本文档时必须遵守的约束

1. 先按顺序读，不要跳过第一层基线文档。
2. 写正式技术材料时，必须回到工程事实层核对。
3. 不要把模板和完成版材料当成系统现状的第一事实源。
4. 不要夸大辅助诊断、数字人和推荐模型的成熟度。
5. 不要忽略 Android 已经单入口这一事实。
6. 不要写成“所有模块均已彻底迁移完成”，更准确的表述是“单入口已完成，模块化持续收敛中”。
7. 不要把历史 BLE 文档与当前协议总结文档当成同等优先级。
8. 不要把研究规划文档默认写成已上线。

## 8. 一句话结论

如果后续 AI 只能先读一份文档，那就先读本文档，再按本文给出的 A 层到 H 层顺序逐步展开；不要从模板、历史材料或局部专题文档直接开始。
