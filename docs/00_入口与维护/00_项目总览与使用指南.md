# 项目总览与使用指南

本文档面向当前仓库维护者，而不是比赛评审。目标是帮助开发者快速回答四个问题：

1. 项目现在的主线是什么
2. 当前真实运行入口在哪里
3. 代码结构和迁移状态如何
4. 应该先看哪些文档

## 1. 项目定位

VesperO 是一个面向睡眠、恢复与日内干预场景的端云协同项目。当前业务主线是：

`统一画像 -> 处方 -> 执行 -> 反馈`

项目重点不再是“展示更多监测图表”，而是把监测、问诊、医检、干预历史整合成可执行的处方，并通过执行效果继续反哺画像。

## 2. 当前代码入口

### Android
- 当前运行入口：`app-shell/`
- 当前唯一构建入口：`:app-shell`
- 当前运行时代码已收敛到 `app-shell/` 及参与依赖的 `core-*` 模块，不再通过 `sourceSets` 编译 `app/`。
- `app/` 仅保留为 legacy archive，供历史代码参考与后续清理。

### Cloud
- 当前云端主线：`cloud-next/`
- 技术栈：Next.js Route Handlers + Supabase

### Contracts / ML
- 契约层：`contracts/`
- 模型流水线：`ml/`

### AI Runtime
- 端侧实时能力仍以本地检测、规则回退、本地 OCR 和本地干预草案为主。
- 云侧生成能力目前以 `cloud-next` 为统一入口，已开始采用 `Vector Engine -> OpenRouter -> DeepSeek` 的 provider 优先级。
- Android UI 不再直接调用具体模型客户端，正在向服务层访问 AI 能力收口。

## 6. 当前主线在代码中的落点

### 统一画像
主要由旧业务层先承载，核心入口可从以下方向理解：
- 睡眠与健康指标
- 问诊结论
- 医检结果
- 干预执行历史

### 处方
当前已在旧业务层出现独立处方仓库与 provider 链，说明“处方”已经不是页面内硬编码逻辑，而是在向独立决策层收口。

### AI
当前 AI 已形成三层主线：
- 端侧实时层：异常检测、规则回退、本地 OCR、本地干预兜底
- 云侧生成层：处方生成已切入 provider 优先级；医生问诊、医检增强、语音与多模态任务将复用同一入口
- 策略编排层：UI -> 服务层 -> provider / local fallback，而不是页面直接访问模型客户端

### 执行
当前已抽出 `InterventionTask`、`InterventionExecution`、`InterventionEffect` 等对象，并在 contracts / core-model / cloud API 中形成雏形。

## 7. 当前维护建议

- 新维护者先不要把 `feature-*` 当成已完成业务模块。
- Android 真正排查逻辑时，优先从 `app/` 和 `app-shell/` 入手。
- 如果是跨端问题，先看 `contracts/` 与 `cloud-next/README.md`。
- 如果是主线问题，先看画像、处方、执行三层是否在同一条链上，而不是盯某个页面。
