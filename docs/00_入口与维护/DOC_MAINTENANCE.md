# 文档维护规则

本文档定义整个仓库的文档维护方式，目标是降低两类问题：

- 文档入口太多，没人知道先看哪份
- 文档内容太旧，和代码实际状态脱节

## 1. 文档分层

### 入口层
- `README.md`
- `docs/README.md`

职责：告诉维护者项目是什么、怎么运行、该看哪些文档。

### 结构层
- `docs/00_入口与维护/02_当前项目现状与上下文基线.md`
- `docs/00_入口与维护/03_当前材料总索引与归档建议.md`
- `docs/10_项目事实与架构/MODULE_MAP.md`
- `docs/10_项目事实与架构/TECH_REFACTOR_STATUS.md`

职责：说明模块职责、迁移进度、运行入口。

### 技术事实层
- `docs/10_项目事实与架构/06_技术文档基线_2026-03-05.md`
- `docs/20_专题系统说明/戒指通讯协议使用说明与阶段性总结.md`
- `cloud-next/README.md`
- `contracts/README.md`

职责：记录可由代码验证的事实、协议、数据流、API、部署方式。

### 历史/展示层
- `docs/90_历史交付与模板/`
- 各类比赛模板与总结文档

职责：用于汇报、比赛、总结，不直接作为工程真相来源。

## 2. 变更触发器

发生以下变更时，必须同步更新文档：

| 变更类型 | 必须更新的文档 |
| --- | --- |
| 新增/删除模块 | `02_当前项目现状与上下文基线.md`、`MODULE_MAP.md`、`README.md` |
| 运行入口改变 | `02_当前项目现状与上下文基线.md`、`README.md`、`00_项目总览与使用指南.md`、`docs/00_入口与维护/DOC_MAINTENANCE.md` |
| Android / Cloud / ML 数据链路变化 | `02_当前项目现状与上下文基线.md`、`06_技术文档基线_2026-03-05.md` |
| BLE 协议、采样任务、解析规则变化 | `戒指通讯协议使用说明与阶段性总结.md` |
| v2 schema / DTO 变化 | `contracts/README.md` 和相关 schema 说明 |
| 云端部署与环境变量变化 | `cloud-next/README.md` |
| 重构阶段变化 | `02_当前项目现状与上下文基线.md`、`TECH_REFACTOR_STATUS.md` |
| 新增/删除正式材料、提交包、证据包或归档策略变化 | `03_当前材料总索引与归档建议.md`、`docs/README.md` |

## 3. 编写规则

- Markdown 统一使用 UTF-8。
- 一份文档只负责一个层级，不重复承担 README、设计文档、比赛材料三种角色。
- 文档里引用“当前状态”时，优先写可验证事实，不写愿景式表述。
- 若某信息仍处于计划阶段，必须明确标注“规划中”或“迁移中”。
- 不在根 README 中塞入长篇比赛叙事、图表和历史路线图。

## 4. 维护流程

每次提交前，至少做一次 3 分钟文档检查：

1. 这次改动是否改变了运行入口或模块边界。
2. 这次改动是否让已有文档失真。
3. 新增能力是否已经有可查入口。
4. 历史文档是否需要被标记为“仅供参考”。

## 5. 推荐维护顺序

如果时间有限，优先级如下：

1. `docs/00_入口与维护/02_当前项目现状与上下文基线.md`
2. `docs/00_入口与维护/03_当前材料总索引与归档建议.md`
3. `README.md`
4. `docs/README.md`
5. `docs/10_项目事实与架构/MODULE_MAP.md`
6. `docs/10_项目事实与架构/06_技术文档基线_2026-03-05.md`
7. 具体子系统文档

## 6. 当前已知问题

- 根 README 曾存在编码和内容双重失真。
- 历史上部分入口文档曾长期停留在单模块 `:app` 时代。
- `docs/00_入口与维护/00_项目总览与使用指南.md` 偏比赛材料，不适合作为仓库入口。
- `docs/` 中历史材料较多，容易误导新维护者。

## 7. 建议执行节奏

- 小改动：当次提交顺手更新。
- 阶段性重构：在功能合并当日补齐结构层文档。
- 版本切换：做一次入口层与技术事实层通读校对。
