## 4. 一句话定义项目当前形态

当前项目不是单一 Android App，而是一个已经形成“移动端 + 云端 + 合同层 + 本地/云端 AI 能力 + 训练与导出脚本”的端云协同睡眠健康系统。

它的主运行入口是多模块 Android 工程 `:app-shell`，配套一个 `cloud-next` 的 Next.js + Supabase 云端，`contracts/` 负责跨端 DTO / schema 对齐，`ml/` 承担睡眠分期与相关模型训练、导出、推理脚本，`tools/` 提供一批辅助脚本、资料处理和证据采集工具。

## 11. Android 关键业务链路

### 11.1 启动链路

应用启动后，`SleepHealthApp` 当前会执行：

- `DataManager` 初始化
- `ApiClient` 初始化
- 本地边缘模型初始化
- 讯飞凭证引导初始化
- mock 数据种子填充
- `DemoBootstrapCoordinator.bootstrapForCurrentSession()`

这意味着“启动即准备 demo 数据和部分 AI 基础设施”是当前默认行为。

### 11.2 设备采集链路

当前链路为：

1. `feature-device` 触发扫描与连接
2. `core-ble` 完成协议交互与解析
3. `DataCollectionService` 后台持续采集
4. 采样结果写入 `core-db`
5. 首页、趋势页等从 Room 流式读取

`DataCollectionService` 当前会：

- 维护前台通知
- 以 `realtime` 记录实时快照
- 按节流策略写入健康指标和 PPG
- 在原始 HRV 缺失时使用本地估计

### 11.3 晨报链路

当前晨报并非静态拼接，而是：

1. 从 Room 读取最近睡眠/恢复/指标
2. 本地异常检测与恢复评分计算
3. 聚合用药、饮食、干预执行等证据
4. 必要时上传睡眠到云端并请求分析
5. 同步结果后刷新推荐与解释

### 11.4 医生问诊链路

当前链路包括：

1. 本地风险与结构化问诊推理
2. 云端医生回合能力补充
3. 问诊摘要本地持久化
4. 登录态时回写云端 inquiry summary
5. 从建议中生成干预任务

### 11.5 周期报告链路

趋势页当前不是单纯本地统计：

1. 读取本地睡眠、恢复、指标、PPG
2. 生成本地兜底周期总结
3. 如云端可用则请求周期报告
4. 用云端 explanation / report 覆盖或增强本地展示

### 11.6 干预与放松链路

当前干预域已经形成闭环：

- 基线评估
- 干预画像生成
- 干预任务与执行
- 放松训练与复盘
- 医疗报告/用药/饮食分析作为画像证据
- 处方生成与效果追踪

云端已经形成成体系的 API 分层，而不是单一上传接口。

### 13.3 主要 API 领域

### 13.4 当前云端推荐/解释模型状态

## 19. 当前已暴露的工程风险与技术债

这部分不是规划，而是从现状中能直接观察到的问题。

- 宿主层 `MainActivity` 已承载大量全局 UI 与播报编排，后续重构时很可能需要拆分。
- `app/` 旧工程仍保留大量重复代码和测试，容易误导维护者。
- 文档体系数量很多，历史材料与事实文档混杂，极易造成误判。
- 移动端与云端功能都已扩展到多域，但自动化测试覆盖没有同步扩展。
- 一些第三方 AI / 语音能力仍通过应用配置直接注入，配置治理与安全治理需要专门收口。
- `feature-relax` 与 `feature-home` 已承担跨域职责，后续重构时需明确边界，避免继续堆叠。
