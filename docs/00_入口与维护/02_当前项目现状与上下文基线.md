## 5. 一句话定义当前项目

当前项目是一个端云协同睡眠健康辅助系统，而不是单一 Android App。

移动端以智能戒指采集和 Android 交互为主线，云端以 Next.js Route Handlers + Supabase 提供认证、数据持久化、AI 编排、后台管理和模型任务调度。`contracts/` 维护跨端 schema 和 DTO，`ml/` 与 `hf-inference/` 承担睡眠模型训练、导出和 ONNX 推理服务，`tools/` 提供证据采集、图表生成、资料整理和模型资产处理。

当前业务主线可以概括为：

`设备采集 / demo bootstrap -> 睡眠与健康指标 -> 医生问诊 / 医检 / 用药 / 饮食 -> 个性化画像 -> 处方与干预任务 -> 执行与回写 -> 趋势和周期报告 -> 后台观测`

## 12. app-shell 现状

- Application、Manifest、MainActivity、打包配置。
- 主导航、底部导航与页面切换。
- 全局 3D 头像/桌面机器人叠层。
- 页面进入讲解、点击讲解、TTS 播放、播报预热。
- 后台数据采集 Service。
- mock 数据初始化与 demo bootstrap。
- API、端侧模型、讯飞凭证启动初始化。

`MainActivity` 当前还承担：

- top-level destination 同步。
- 非顶层页面时底部 Tab 高亮保持。
- avatar 根据页面调整尺寸、位置、可见性。
- avatar 拖拽边界约束。
- avatar speaking / idle / tap 动画。
- 页面可见文本采集，用于构造讲解上下文。
- 本地讲解缓存和云端讲解请求。

重构风险：

- `MainActivity` 已经是全局交互编排器，后续应拆出 Avatar overlay controller、navigation observer、page narration context collector。
- 当前 avatar 使用 `feature-relax` 的 `HumanBody3DView`，造成宿主层到业务 feature 的反向耦合感较强，虽能构建，但架构上需要后续收口。

## 16. core-ble 现状

`core-ble` 是当前戒指通信层，核心对象包括：

- `BleManager`
- `BleConnectionManager`
- `Hi90BCommandBuilder`
- `Hi90BFrameParser`
- `Hi90BWorkParamsCodec`
- `CleveringCommandSender`
- `DataParser`
- `MockBleManager`
- `DemoDataGenerator`
- `DemoConfig`

当前 BLE 事实：

- 支持扫描名包含 `Clevering` 或 `Hi90B` 的设备。
- 使用 Hi90B / Clevering 自定义 UUID：`0xFFFA` service、`0xFFFC` notify、`0xFFFB` write。
- 帧格式为 `AA 55 + Len + CMD + DATA + XOR`。
- 写入策略倾向 `WRITE_TYPE_NO_RESPONSE`，内部有命令队列、间隔、重试和 no-response tick。
- 连接后会设置通知、同步时钟、读取电量、配置工作参数、读取参数、查询历史数据并开启传输。
- 当前采样项覆盖心率/血氧、体温、HRV、PPG、运动。
- PPG 有 watchdog 和 fallback profile。
- motion burst 使用 `0x80`，支持 ACC/GYRO 组合解析。
- 数据上报支持 `0x50`、`0x52`，并兼容多个 payload 格式。

测试覆盖：

- `Hi90BProtocolTest` 覆盖命令构帧、校验和、分片解析、无效校验帧丢弃。

风险：

- `BleManager.kt` 超过千行且承担扫描、连接、协议、采样策略、解析、回调，后续应拆成 scanner、connection、command queue、protocol decoder、sampling policy。

## 26. 本地关键业务链路

### 26.1 设备采集链路

当前链路：

1. `DeviceFragment` / `DeviceViewModel` 发起扫描、连接和采集控制。
2. `core-ble` 完成 GATT 连接、通知订阅、命令下发和帧解析。
3. `DataCollectionService` 作为前台服务维持后台采集。
4. 采样数据经 `SleepRepository` 写入 Room。
5. 今日页、趋势页、医生页等从 Room 读取最新数据。

`DataCollectionService` 当前策略：

- 2 秒节流写入健康指标。
- 500ms 节流写入 PPG sample。
- 维护心率、血氧、体温、HRV、运动样本新鲜度。
- 原始 HRV 缺失时使用 `HrvFallbackEstimator`。
- 外周体温通过 `TemperatureMapper` 映射。
- motion magnitude 由 accelerometer 和 gyroscope 综合计算。
- 使用固定 sleepRecordId `realtime` 保存实时快照类数据。

### 26.2 今日晨报链路

当前链路：

1. 读取本地最新睡眠、健康指标、恢复分。
2. 计算本地异常和恢复解释。
3. 聚合最近处方、干预执行、用药、饮食、个性化状态。
4. 如有云端 session，则进行睡眠上传/分析/同步/推荐解释获取。
5. UI 显示恢复分、关键指标、建议、可信度和个性化提示。

### 26.3 医生问诊链路

当前链路：

1. 用户输入文本或语音结果。
2. `DoctorDecisionEngine` 做本地风险、红旗、follow-up、fallback assessment。
3. `DoctorInferenceEngine` 处理结构化医生回合输出解析。
4. `NetworkRepository` / `DoctorAiService` 调用云端 doctor turn。
5. 会话、消息、评估写入 Room。
6. 登录态下回写云端 doctor inquiry summary。
7. 可从建议动作生成干预任务。

### 26.4 医检报告链路

当前链路：

1. 导入图片或 OCR 文本。
2. 先用本地解析和草稿格式化兜底。
3. 可调用云端 `/api/report/understand` 增强结构化理解。
4. 结果写入 `medical_reports`、`medical_metrics`。
5. 异常指标进入画像、周期报告和干预建议证据链。

### 26.5 用药/饮食链路

当前链路：

1. 导入图片。
2. Android 通过 multipart 调用 `/api/medication/analyze` 或 `/api/food/analyze`。
3. 云端多模态/结构化视觉生成结果。
4. Android 本地保存分析记录。
5. 登录态下回写 `/api/medication/records/upsert` 或 `/api/food/records/upsert`。
6. 用药风险和营养风险进入画像与处方证据链。

### 26.6 干预闭环

当前链路：

1. 基线评估、医生问诊、医检、睡眠、用药、饮食、执行历史形成画像证据。
2. `InterventionProfileRepository` 构造画像快照。
3. `PrescriptionRepository` 生成处方 bundle 和 item。
4. `InterventionSessionViewModel` / `BreathingCoachViewModel` / `ZenInteractionViewModel` 执行具体干预。
5. 执行结果写入 `intervention_executions`、`relax_sessions`。
6. 本地/云端效果趋势进入今日页和趋势页。

## 30. 云端核心业务链路

### 30.1 睡眠上传与分析

1. Android 调用 `/api/sleep/upload` 写入睡眠会话/窗口数据。

4. `/api/internal/worker/run` 抢占任务。

6. 调用真实模型 endpoint 或本地 baseline/ensemble fallback。
