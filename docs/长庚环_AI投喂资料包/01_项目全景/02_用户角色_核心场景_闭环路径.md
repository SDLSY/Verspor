# 用户角色、核心场景与闭环路径

- 适用任务：场景描述、用户故事、答辩讲稿、需求背景说明。
- 推荐阅读优先级：高。
- 是否允许对外直接引用：可以转写后引用。
- 最后核对日期：2026-03-15。

## 用户角色

## 1. 普通使用者

最核心的角色。典型诉求包括：

- 想知道昨晚和今天的恢复状态
- 想理解当前疲劳、压力、睡眠波动或不适背后的原因
- 想获得比“多休息”更具体的执行建议
- 想把症状、报告、药物和饮食信息一起纳入参考

## 2. 有症状或恢复困扰的用户

这类用户更关注：

- 哪些症状值得警惕
- 是否需要继续追问或线下就医
- 医检报告里哪些指标值得关注
- 药物和饮食是否正在影响恢复状态

## 3. 后台运营或管理视角

通过 `cloud-next` 后台工作台查看：

- 患者概览
- 报告积压
- 建议轨迹
- 模型资产和作业状态
- 审计与系统记录

这不是终端用户主入口，但对整个端云系统的运转很关键。

## 4. AI 协作角色

项目当前已经明确需要“别的 AI”继续参与写文案、整理资料、派生技术材料。因此资料包里必须给这个角色单独设计标准入口和禁写规则。

## 核心使用场景

## 场景一：每日状态查看

入口：今日页  
核心动作：查看恢复分、生理数据、风险摘要、建议入口  
结果：知道当前状态和建议优先级  
后续影响：进入趋势、干预中心或医生页

## 场景二：设备连接与持续采集

入口：设备页  
核心动作：扫描戒指、连接设备、前台采集或后台采集  
结果：心率、血氧、体温、PPG、运动等持续进入本地数据链  
后续影响：今日页、趋势页、干预反馈、恢复分都能消费这些数据

## 场景三：症状自查与辅助判断

入口：症状自查页  
核心动作：定位部位、补充症状、查看风险层级与疑似方向  
结果：形成辅助判断与下一步建议  
后续影响：可跳转医生问诊、报告分析或干预建议

## 场景四：AI 医生问诊

入口：医生页  
核心动作：围绕主诉进行追问、整理事实、生成结构化问诊单  
结果：得到更适合后续判断和沟通的结构化内容  
后续影响：进入干预、报告理解或就医建议

## 场景五：医检报告理解

入口：医检报告分析页  
核心动作：上传 PDF 或图片、OCR 识别、结构化指标提取、生成可读报告  
结果：把原始报告转成用户能读懂的重点摘要  
后续影响：风险提醒、医生页对话、推荐解释和画像更新

## 场景六：执行干预

入口：干预中心  
核心动作：选择呼吸训练、Zen 交互、音景会话、放松训练等  
结果：发生一次实际执行，而不是停留在建议层  
后续影响：执行记录写回本地/云端，次日恢复分和趋势可感知效果

## 场景七：药物与饮食分析

入口：干预中心扩展入口  
核心动作：拍照识别药物或食物、生成结构化结果与建议  
结果：把用药和饮食从“额外信息”变成画像的一部分  
后续影响：影响恢复分、建议解释、患者时间线和后续推荐

## 主闭环路径

## 闭环一：状态理解闭环

感知 -> 今日页 -> 异常/风险解释 -> 趋势对照 -> 再次感知

这是最基础闭环，重点是“看懂变化”。

## 闭环二：建议执行闭环

状态输入 -> 干预生成 -> 执行会话 -> 执行记录 -> 次日恢复分 / 趋势反馈

这是产品的核心闭环，重点是“从建议到效果”。

## 闭环三：辅助判断闭环

症状输入 / 报告输入 -> AI 医生或可读报告 -> 风险提示 / 下一步建议 -> 干预或线下处理

这是从静态信息走向行动判断的闭环。

## 闭环四：生活方式增强闭环

药物 / 饮食记录 -> 画像增强 -> 恢复分变化 -> 推荐解释 -> 后续记录

这是项目近期扩展的重要闭环，说明系统已开始把更多日内因素纳入参考。

## 对写作最有用的结论

1. 长庚环要写“连续使用场景”，不要写成单页工具。
2. 页面之间不是并列关系，而是围绕同一健康问题不断接力。
3. 真正的价值不在单点 AI，而在多个入口最终回到同一闭环。
