## 数据层的一句话事实

项目当前不是“页面状态拼装型应用”，而是有明确本地数据库、共享合同层、云端结构化结果和同步字段的多层数据体系。

## 本地领域对象与上层模型

除了 Room 实体，项目还在 `core-model` 中维护更贴近业务的领域模型，包括：

- 干预协议定义。
- 干预体验 metadata。
- 日恢复分与贡献项。
- 药物分析记录。
- 饮食分析记录。
- Zen 交互模式。
- 自适应音景状态。

这意味着：

- 页面不直接依赖数据库字段命名。
- repository 会做实体到领域对象的映射。
- 一些结构化信息以 JSON 字符串保存在本地表里，再映射回强类型对象。

## 干预执行与 metadata

最近的结构升级里，以下两类记录新增了 `metadataJson`：

- `relax_sessions`
- `intervention_executions`

这部分 metadata 用来表达：

- 干预模态。
- 是否启用触觉。
- 是否有实时反馈。
- 放松信号强弱。
- 音景混音状态。
- 手动调节次数。
- 完成质量。

因此，新的干预能力不是靠新增一堆零散日志表接入，而是通过既有执行记录扩展 metadata 接入全局链路。

## 医检报告数据

本地与云端都围绕“原始图/原始文档 -> OCR/结构化 -> 指标 -> 可读摘要”组织。

本地主要存：

- 报告基本信息。
- OCR 摘要。
- 解析状态。
- 风险等级。
- 指标条目。

云端进一步负责：

- 结构化理解。
- 可读报告生成。
- 云端 provider 调度与失败归一化。

## 恢复分与画像数据

### 恢复分

项目当前同时存在两层：

- 旧恢复分实体与历史恢复指标。
- 新的日恢复分贡献模型，已经把药物、饮食和干预贡献纳入更广的解释框架。

### 画像

干预画像不是单一表，而是由：

- 量表
- 问诊
- 生理指标
- 报告异常
- 生活方式记录
- 干预执行结果

共同生成，并沉淀为 `InterventionProfileSnapshotEntity`。

## 云端合同层 `contracts`

`contracts` 是跨栈共享合同层，职责是让 Android、Cloud、ML 至少在核心 DTO 和 envelope 结构上保持一致。

当前明确事实：

- `schemas/` 是 source of truth。
- `typescript/` 面向 cloud。
- `kotlin/` 面向 Android/core。
- 统一 envelope 约定为 `{ code, message, data, traceId }`。

这很重要，因为项目并不是“Android DTO 自己写一份，云端 JSON 自己再写一份”的完全松耦合状态。

## 数据流的真实层次

### 第一层：设备/用户输入

- 戒指实时数据。
- 用户手动输入。
- 拍照与上传。
- 问诊文本。

### 第二层：本地沉淀

- Room 实体。
- 偏好设置。
- 临时 UI 状态。

### 第三层：业务映射

- Repository 映射为领域对象。
- 画像与恢复分聚合。
- 干预/处方上下文整合。

### 第四层：云端结构化增强

- 多 provider 结构化分析。
- 统一结果 schema。
- 患者时间线与后台查询。

## 对外最值得强调的数据特点

- 数据链路贯穿“感知 -> 结构化 -> 建议 -> 执行 -> 回写”。
- 本地数据库不是缓存壳，而是完整的离线基础层。
- 云端与 Android 通过共享合同和同步字段保持一致性。

## 不应误写的点

- 不要写成“所有数据只存在云端”。
- 不要写成“Room 仅存睡眠数据”。
- 不要忽略药物、饮食、干预 metadata 已进入统一数据体系这一事实。
