# 长庚环（VesperO）技术研究报告

版本号：V1.0  
项目名称：长庚环  
英文名称：VesperO  
团队名称：夜巡者  
日期：2026-03-12

## 问题聚焦

长庚环的技术研究并不是围绕某一个孤立的算法点展开，而是围绕一个更接近真实生活的系统性问题展开：当用户已经能够通过可穿戴设备、互联网医疗和医检报告工具获得大量健康信息时，为什么仍然很难在日常生活中形成稳定、可信、低门槛的健康行动闭环。换言之，项目研究的并不是“如何再增加一个指标”或“如何把报告 OCR 再提一点准确率”，而是研究如何把多源输入组织成用户真正能理解、能执行、能反馈的完整系统。

从项目实践出发，当前数字健康类产品往往存在三类断层。第一类断层发生在“监测”和“行动”之间。用户可以知道自己昨晚睡得不好、今天恢复分偏低、某些指标波动明显，但这并不自动等价于知道今天应该如何安排运动、作息、呼吸训练、康复动作，或者是否需要就医。第二类断层发生在“症状描述”和“专业判断”之间。大量用户在面对问诊场景时，不知道从哪里开始描述问题，也不清楚哪些伴随症状更关键。第三类断层发生在“报告识别”和“用户理解”之间。即使 OCR 技术已经能够较好地提取文字，用户仍然不一定能够理解其中的指标含义，也不一定能够把报告结论转化为下一步行动。

长庚环把这些断层视为同一类问题的不同表现：即健康管理流程在真实使用场景中被拆散为多个工具和多个阶段，用户被迫自己完成大量信息整合和决策工作。因此，本项目的技术研究核心，是如何用端云协同、多模块组织、本地兜底、AI 增强和低门槛交互设计，把原本零散的数据、页面和能力重新收束为一个单入口系统。这个系统需要同时满足四个现实条件：一是硬件侧要能够持续提供有价值的数据；二是移动端要能够在弱网和复杂权限环境中稳定运行；三是云端要能够承担解释、推理和同步任务；四是整体输出必须保守、可信，不越过健康辅助系统应有的边界。

### 问题描述

#### 睡眠与恢复场景中的信息过载问题

在睡眠与恢复管理场景中，用户最常见的困境并不是数据太少，而是数据太多却没有行为指引。现代可穿戴设备普遍可以输出睡眠时长、深睡比例、心率区间、血氧波动、压力指数、恢复评分等一系列指标。从工程视角看，这说明感知和可视化能力不断提升；但从用户视角看，这些指标越丰富，用户越容易陷入“知道自己状态不佳，却不知该怎么办”的状态。特别是在晨间查看数据时，用户更需要的是“今天应该采取什么行动”，而不是一组只表征状态的图表。

长庚环在技术研究中把这一问题抽象为“数据可视化不等于决策支持”。如果系统只展示监测结果而不提供后续动作路径，那么监测本身的价值就会被压缩成“告诉你发生了什么”，而不能进一步延伸为“帮助你决定接下来做什么”。因此，项目研究的第一重点不是如何生成更多统计结果，而是如何把监测结果组织为风险提示、建议级别、干预入口和执行反馈。

#### 可穿戴设备“能监测、难行动”的现实断层

智能戒指、手环、手表等硬件已经证明连续感知具备现实价值，但许多产品仍然停留在“感知层”。用户佩戴设备的体验往往较好，却难以持续从设备中获得行为改变层面的价值。原因并不在于硬件能力本身，而在于系统没有将设备采集结果与用户的主动输入、问诊结果、报告内容和后续执行链路整合起来。

长庚环将这一现实断层转化为技术问题：感知层数据如何和解释层、决策层、执行层建立稳定接口。若没有这样的接口，设备再强，也只能成为“不断生产数据的终端”；而一旦有了这些接口，设备采集结果才能变成真正的输入条件，参与后续 AI 解释、风险分级、处方建议和趋势反馈。

#### 互联网医疗问诊与医检报告理解门槛高

传统互联网医疗问诊系统通常要求用户在较短时间内以较清晰的方式描述症状、病程、诱因和伴随表现。但普通用户并不具备良好的医学表达能力，尤其在头痛、胸闷、腹痛、出汗、膝盖痛等症状出现时，用户更倾向于从身体感知出发，而不是从专业术语出发。另一方面，医检报告工具虽然可以帮助用户识别文本，却无法天然完成“解释为什么重要”和“下一步如何做”的任务。

这意味着项目不仅要解决技术上的 OCR、语音或接口调用问题，更要解决输入路径与结果表达的问题。用户并不需要一个只能返回 JSON 的系统，也不需要一个只能机械识别文字的工具。用户需要的是一个能够接住模糊输入、组织结构化问题、再把结果转换成日常语言的系统。

### 问题抽象

#### 输入：连续生理信号、症状输入、医检报告、执行反馈

从系统建模角度看，长庚环面对的输入可以分为四类。第一类是连续生理信号输入，主要由智能戒指采集，包括睡眠记录、PPG 样本、心率趋势、血氧、温度映射和设备状态信息。这类数据的特点是持续、碎片化、高频，且通常缺少直接语义。第二类是用户主动输入，包括症状自查时的部位选择、症状标签、严重度、持续时间、文本描述与语音输入。这类数据语义强但结构松散，常带有不确定性。第三类是医检报告输入，包括图片、OCR 结果、结构化指标和用户上传的报告文本。第四类是执行反馈输入，包括呼吸训练完成情况、干预动作执行记录、体感反馈和后续趋势变化。

项目在研究中把这四类输入视为同一画像体系的不同切面。连续生理信号提供背景，症状输入提供主诉，医检报告提供证据，执行反馈提供结果。只有把这些输入整合起来，系统才能真正形成用户状态的动态理解，而不是针对某一个时刻或某一类数据做局部判断。

#### 输出：辅助判断、风险提示、干预建议、执行路径

系统输出也不能只是一段文本结论。长庚环将输出设计为多层次结构。第一层是辅助判断与风险提示，用于告诉用户当前状态应归入何种风险级别，哪些信息值得重点关注。第二层是干预建议，用于将抽象状态转化为轻量级动作建议，如休息、呼吸训练、放松练习、继续观察或进入医生问诊。第三层是执行路径，即为用户提供可立即进入的页面与任务，例如进入呼吸训练页、开始问诊、分析报告或跳转线下建议。第四层是反馈入口，使得系统能在执行后回收体感和状态结果，用于下一轮建议。

这个输出设计体现了项目的技术判断：健康系统不能把“输出”理解为一个文本结束点，而应当把输出理解为新的流程节点。只有这样，系统的每一次分析才不是终点，而是后续执行的起点。

#### 约束：低时延、弱网、隐私、安全表达、非确诊定位

如果只从理想条件出发设计系统，许多技术问题都会被简化。但长庚环明确把真实约束纳入研究对象。首先是低时延约束。设备页连接反馈、首页恢复状态、机器人页面讲解、症状自查结果都要求较快返回，否则用户会失去信任。其次是弱网和云端波动约束。健康场景不可能完全依赖稳定网络，因此需要本地兜底和回退方案。再次是隐私和账号约束。用户资料、报告和问诊内容都涉及敏感信息，账号、邮件、回跳和会话必须保持一致。最后是表达边界约束。系统定位于健康辅助和康复辅助，不能使用越界的诊断式语言，因此必须在输出设计上体现保守和安全。

这些约束共同决定了项目的技术路线：不能采用纯云端、不能采用单一模型、不能采用只面向技术展示的架构，而必须采用端云协同、多层回退和结构清晰的系统设计。

### 问题定位

#### 项目技术定位：居家健康管理与康复辅助系统

长庚环的技术定位首先不是临床确诊系统，而是居家场景中的健康辅助系统。这意味着系统要优先解决的是“让普通人更容易理解自己的状态并采取安全行动”，而不是“取代医生完成诊断”。在这样的定位下，系统的技术重点不在单次高精度判断，而在持续感知、可解释输出和流程组织。

这一定位也解释了为什么项目会同时重视智能戒指、症状自查、AI 医生、报告分析、呼吸训练和复盘页。因为它们在居家场景中分别承担不同环节的职责，只有联动起来，才可能形成真正有用的日常工具。

#### 非医疗确诊工具的边界说明

技术研究必须承认边界。长庚环可以提供风险分级、辅助判断和下一步建议，但不直接输出确诊结论。这一边界并不是削弱系统能力，而是让系统更适合真实场景。越接近用户的日常生活，越需要在表述上克制，在逻辑上保守，在结果上提供就医和进一步咨询的出口。

因此，项目在技术实现中不仅研究“如何生成内容”，也研究“如何控制内容”。例如在报告分析中强调可读性与下一步建议而非绝对诊断，在症状自查中强调“建议尽快就医”而不是“确认疾病”，在机器人讲解中强调页面说明而不是夸张判断。

#### 端侧即时反馈与云端增强理解的分工

项目将端侧和云端视为协作关系而非替代关系。端侧负责即时交互、状态展示、本地数据库、BLE 管理、部分本地算法和本地文案兜底。云端负责账号鉴权、同步、深度问诊、报告理解、讲解稿生成和后台管理。这样的分工使系统既能提供即时反馈，又能获得更复杂的解释能力。

如果所有能力都放在云端，系统会面临网络依赖和时延问题；如果所有能力都放在端侧，系统会失去模型灵活性和统一管理能力。因此，长庚环的研究重点之一就是找到这两者之间的合理分界，并通过明确的接口和配置策略把分工落到工程中。

### 问题评估

#### 技术评估指标：稳定性、时延、容错性、可扩展性

技术研究的价值必须体现在可衡量标准上。长庚环在技术层关注四类指标。第一类是稳定性，主要考察 Android 构建是否稳定、核心页面是否可进入、数据库与 BLE 链路是否可持续运行。第二类是时延，主要考察设备状态反馈、页面首屏展示、症状结果生成和机器人播报是否足够及时。第三类是容错性，主要考察云端不可用、讯飞能力未配置、网络波动、邮件链路异常等情况下是否仍有可用回退。第四类是可扩展性，主要考察模块边界、契约层、服务层和配置方式是否支持后续扩展更多设备、模型与页面。

#### 产品评估指标：触达率、执行率、可理解性、可回溯性

技术研究不应只停留在工程内部指标。长庚环同时关注用户层面的技术结果，包括晨报和建议是否能够触达用户、建议是否能顺利转化为实际执行、报告和问诊结果是否足够可理解、执行记录和状态变化是否能够回溯。这些指标虽然最终表现为产品结果，但其实现基础仍然是系统架构、页面交互和数据组织方式。

#### 工程评估指标：单入口、模块边界、自动化测试、部署一致性

工程质量是项目可交付性的直接体现。长庚环已经完成 Android 单入口收口，以 `:app-shell` 作为唯一运行入口，不再通过 legacy `sourceSets` 编译旧 `app/`。这意味着工程上已经具备清晰的运行边界。与此同时，`core-*`、`feature-*`、`cloud-next/`、`contracts/` 和 `ml/` 形成了较明确的模块分层。再结合单元测试、lint、云端 build 验证和域名统一部署，项目具备较好的工程完整性。

### 问题分解

#### 戒指采集与 BLE 链路问题

第一类要解决的问题是设备接入。包括设备扫描、连接、断开、命令发送、帧解析、工作参数读取和后台采集管理。这部分如果不稳定，整个系统就无法形成感知基础。

#### 端侧数据处理与本地兜底问题

第二类问题是 Android 端如何处理持续采集数据、本地数据库缓存、本地解析、本地算法、本地页面状态与本地回退。这部分决定系统是否能在弱网与云端波动环境下仍然可用。

#### 云端 AI 理解与路由问题

第三类问题是云端如何承担问诊、报告理解、讲解稿生成和后台管理接口。这里不仅涉及模型调用，也涉及 provider 路由、账号态、接口封装和错误处理。

#### 症状自查、问诊、报告理解的一体化组织问题

第四类问题是如何把本来属于不同产品领域的交互链路组织在一起，让用户不需要在多个工具之间切换。这部分同时涉及页面设计、导航结构和数据上下文传递。

#### 从建议到执行再到反馈的闭环问题

第五类问题是系统是否只会“说”，还是能真正让用户“做”，并在做完之后留下反馈。干预中心、呼吸训练、旧康复页保留入口和复盘页共同解决这一问题。

## 相关工作

### 智能戒指与睡眠监测研究现状

智能戒指和睡眠监测产品已经证明，体积较小、佩戴自然的设备非常适合承担夜间连续感知任务。与手表等设备相比，戒指在睡眠场景中对用户的干扰更低，用户更容易长期佩戴，这使其在连续数据采集方面具有天然优势。现有主流智能戒指产品通常关注睡眠、恢复、活动和压力等指标，形成了较为成熟的感知和可视化范式。

但从系统角度看，现有研究和产品更多解决的是“如何更稳定地采集数据”和“如何把结果展示出来”的问题，对“如何让用户形成行动”这一问题研究和产品化不足。长庚环在此基础上继续向前推进，把戒指不再视为终点，而是视为整个健康辅助系统的输入端。

### 居家健康管理与数字疗法相关工作

居家健康管理与数字疗法研究强调通过持续追踪、行为建议和轻量级干预帮助用户改善健康状态。这类研究在慢病管理、心理干预、睡眠行为干预和康复陪伴等领域已经积累了较多方法论。它们普遍强调三个关键点：第一，用户不能被要求承担过高认知成本；第二，建议必须尽量可执行；第三，系统应该具备一定的长期陪伴能力。

长庚环与这类工作存在明显关联。尤其在呼吸训练、旧康复辅助页、执行复盘与日内建议方面，项目实际上是在把数字疗法和居家健康管理的思想纳入智能戒指系统中。但长庚环并不直接等同于数字疗法产品，因为它还承担监测、问诊与报告理解的职责。项目因此更像一个“居家健康管理与康复辅助基础平台”，而不是某一个单病种的疗法工具。

### 智能问诊与症状自查相关研究

症状自查和智能问诊系统通常通过结构化问答、关键词识别和风险分流，帮助用户把主诉转化为可解释结果。在技术上，这类系统关注自然语言理解、意图识别、多轮问答和风险控制。在产品上，它们关注如何降低输入门槛、如何保持结果保守且有用。

长庚环在这里的切入点在于，把症状自查设计为问诊之前的轻交互层，而不是直接让用户进入文本对话。通过 2.5D 身体定位图、前后侧切换、热点点击和症状标签，用户可以更自然地开始输入。这种方式兼顾了互联网医疗问诊的结构性和消费级健康产品的易用性，具有较强的技术和交互综合价值。

### 医检报告 OCR 与自动理解相关研究

OCR 与文档解析技术已经较为成熟，但将医检报告转化为用户能读懂的解释仍然具有挑战。因为医检报告不仅包含文本，还包含复杂表格、指标单位、参考范围和异常标记；更重要的是，用户理解报告的目标并不是“看到所有内容”，而是“知道哪些内容重要、为什么重要、下一步做什么”。

长庚环在这方面的研究思路是把 OCR、结构化事实和自然语言解释做分层处理。OCR 只是将图像转为文本，结构化解析负责提取指标和值，最后的可读化层负责把结果转成用户可理解的说明。这一分层方式比直接把 OCR 原文交给用户更有效，也比单纯依赖大模型生成更可控。

### 相关工作的不足与本项目切入点

综合来看，现有相关工作普遍各有优势，但难以单独解决长庚环面向的系统性问题。智能戒指擅长连续感知却不一定擅长行动引导；问诊平台擅长结构化追问却缺乏设备背景；报告工具擅长识别却未必能组织后续行为；数字疗法工具擅长执行却通常缺少感知和问诊输入。长庚环的切入点正是把这些能力重新组织成一个闭环，并用统一入口和统一数据上下文降低用户与系统之间的转换成本。

## 技术方案

### 技术方向

#### 端云协同而非纯端侧或纯云端

长庚环选择端云协同，是因为健康辅助场景同时要求即时反馈和深度理解。纯端侧方案虽然可以保证低时延，但很难承载复杂的问诊总结、报告理解和后台管理能力；纯云端方案虽然便于集中处理，却会在设备接入、首屏体验和弱网环境下出现明显短板。因此，项目将感知、即时反馈、数据库与部分本地算法放在端侧，把账号、同步、深度 AI 路由和后台接口放在云端。

#### 连续感知与轻交互结合

项目不希望把用户输入完全替换为设备数据，也不希望设备数据在系统中沦为背景装饰。连续感知负责提供长期、生理层的背景，轻交互负责让用户以最自然的方式补充当前不适与上下文。二者结合后，系统才能同时知道“用户近期整体状态如何”与“用户此刻正在经历什么”。

#### 规则兜底与 AI 增强共存

为了避免把体验完全押注在某一个云端模型或第三方接口上，项目采用了规则兜底与 AI 增强共存的方案。无论是报告可读化、问诊讲解还是机器人播报，都可以优先使用云端增强结果，但必须保留本地文案、本地解析和本地最低可用结果。这种设计增强了系统的鲁棒性，也更符合健康辅助系统的风险管理思路。

### 技术选择

#### Android 单入口架构与多模块收敛思路

Android 端当前以 `:app-shell` 作为唯一运行入口，负责 Application、MainActivity、顶层导航和宿主资源。相较于早期多入口或旧 `app/` 参与构建的状态，当前单入口方案显著提升了系统的一致性。与此同时，项目采用 `core-common`、`core-model`、`core-data`、`core-ble`、`core-db`、`core-network`、`core-ml` 与 `feature-*` 组合的多模块结构，为后续代码整理、边界清晰和可维护性提供基础。

虽然当前 feature 模块尚未完全接管全部业务，但模块边界已经明确：设备相关能力主要集中在 `feature-device` 与 `core-ble`，问诊与数字人能力集中在 `feature-doctor` 与 `core-network`，症状自查、报告分析、干预执行与复盘集中在 `feature-relax`，个人中心与认证集中在 `feature-profile`。这一结构既支持当前交付，又为后续继续收敛代码质量提供空间。

#### BLE 协议接入与本地持久化方案

在设备接入层，项目选用 Nordic BLE 相关库并结合自定义协议管理类完成戒指连接与通信。`BleManager`、`BleConnectionManager`、`CleveringCommandSender`、`Hi90BCommandBuilder`、`Hi90BFrameParser` 和 `DataParser` 共同构成硬件通信链路。该方案的优点在于充分利用成熟 BLE 基础设施，同时保留项目自身协议解析与命令组织的灵活性。

本地持久化方面，项目选用 Room 作为主数据库方案。`AppDatabase` 以及 `sleep_records`、`doctor_messages`、`medical_reports`、`intervention_tasks` 等实体表使系统能够离线保存主线数据。这不仅方便页面展示，也为趋势分析、执行回溯和本地回退提供支撑。

#### 讯飞语音、OCR、数字人相关能力接入

项目在语音与 OCR 层选择了讯飞相关能力，包括语音听写、语音合成、实时语音、录音转写、OCR 和 Spark 对话能力。`core-network` 中已经包含 `XfyunIatWsClient`、`XfyunTtsWsClient`、`XfyunOcrClient`、`XfyunSparkWsClient`、`XfyunRtasrWsClient`、`XfyunRaasrClient` 等客户端封装，以及 `XfyunConfig` 和 `XfyunCredentialBootstrap` 用于配置读取。数字人相关控制逻辑则在医生模块中通过 `XfyunVirtualHumanController` 承接。

该方案的技术意义不在于“集成了多少接口”，而在于项目用统一配置与 BuildConfig 方式把第三方能力纳入工程体系，并在未配置或配置错误时提供明确提示，避免静默失败。近期针对 `local.properties` 和 BuildConfig 的配置修复，也说明了这一方案具备较好的可维护性。

#### 云端 Next.js + Supabase + AI 路由方案

云端方面，项目采用 Next.js Route Handlers + Supabase 的组合。一方面，Next.js 提供灵活的 API 路由和页面承载能力；另一方面，Supabase 提供身份认证、会话管理和与邮件链路结合的基础能力。当前 `cloud-next/src/app/api` 下已经包含 `auth/login`、`auth/register`、`auth/resend-confirmation`、`auth/password-reset`、`doctor/turn`、`report/understand`、`intervention/daily-prescription`、`user/profile`、`avatar/narration` 等大量路由，说明云端已不仅是简单转发层，而是完整的业务服务层。

邮件链路采用 Supabase Auth + Resend SMTP + 自定义域名统一方案。重置密码、确认邮箱与 Android API 基地址都已统一到 `cloud.changgengring.cyou`，这减少了客户端和网页回跳的错配问题，也是整个系统端云统一的重要体现。

#### 本地异常检测、OCR 草案和云端问诊/报告理解的分层方案

本地与云端 AI 分层是项目的一项关键选择。`core-ml` 中已经包含 `SleepAnomalyDetector`、`MedicalReportParser`、`MedicalReportDraftFormatter`、`HealthAnalyzers`、`PpgPeakDetector`、`HrvFallbackEstimator`、`RelaxationScorer` 等本地能力，可在端侧完成异常检测、信号处理和报告草案整理。云端则承担更复杂的问诊总结、报告理解与自然语言增强任务。这样做不仅提升了体验，也降低了系统对单一云端能力的依赖。

### 结果期望

#### 实现连续感知与用户可理解结果之间的桥接

项目预期通过戒指持续采集 + Android 本地处理 + 云端增强解释的组合，使用户不再只面对原始监测数据，而是面对经过组织的结果与建议。最终目标不是指标展示精美，而是让结果能够更自然地进入用户的决策流程。

#### 提升从监测到干预的闭环能力

通过首页、症状自查、问诊、报告分析、干预中心、呼吸训练和复盘页的串联，系统希望将原来分散在多个工具中的行为组织成一个可验证闭环。监测结果能够触发建议，建议能够引导执行，执行又能够反哺后续分析。

#### 在弱网和第三方能力波动下维持最低可用体验

通过本地数据库、本地文案预热、本地算法、本地 OCR 草案和明确的降级逻辑，项目期望在云端不可用、讯飞接口异常或邮件链路波动时，仍然保持主要页面不空白、主要结果不缺失、主要功能可回退的状态。

## 技术实践

### 使用的开发框架及依赖的库

#### Android 技术栈

Android 端使用 Kotlin 作为主要开发语言，`jvmTarget` 为 11。构建工具采用 Android Gradle Plugin `8.7.3`，编译目标为 `compileSdk 35`，最低支持 `minSdk 24`。宿主模块为 `:app-shell`。UI 层以 Fragment、ViewModel、LiveData 和导航图为主；数据层采用 Repository + Room DAO 结构；图表使用 MPAndroidChart；动画与加载效果使用 Lottie 与 Shimmer；依赖注入采用 Hilt；BLE 层使用 Nordic BLE 库；网络层采用 Retrofit 与 OkHttp。

#### 云端技术栈

云端采用 Next.js 路由处理器承接 API，使用 Supabase 实现用户鉴权和账号链路。项目在云端同时维护业务路由、内部管理路由和模型运行时路由，支持用户资料、睡眠数据、问诊、干预任务、报告理解和后台患者概览等能力。部署使用 Vercel，域名通过 Cloudflare 托管，邮件链路通过 Resend SMTP 与 Supabase 集成。

#### AI/OCR/语音相关技术栈

语音与 OCR 层主要接入讯飞能力，包括 IAT、TTS、OCR、Spark、RTASR、RAASR 和数字人相关配置。端侧轻量 AI 层主要由 `core-ml` 中的本地解析与检测类承担。云端 AI 则通过统一路由策略访问深度模型能力。除此之外，项目也保留 `contracts/` 作为 AI provider、capability 和 logical model 的契约定义层，使多端在语义上保持一致。

### 技术实践过程

#### Android 单入口整合过程

项目早期 Android 端曾存在新旧结构并存的问题，但当前已经完成单入口收口。`app/` 不再参与运行时构建，`app-shell` 成为唯一宿主。这一过程中，团队完成了 Activity、导航、资源、BLE、数据库与页面代码的收编，并逐步将底层能力迁入 `core-*` 模块。虽然 `feature-*` 仍有继续收敛空间，但用户侧已经面对一个统一运行的 App，而不是多个拼接入口。

#### 戒指 BLE 协议与设备页落地过程

在设备侧，项目完成了扫描、连接、断开、状态提示、工作参数读取与后台采集相关链路。过程中不仅要解决协议解析问题，还要解决 Android 不同版本权限模型、蓝牙开关状态和 UI 提示一致性问题。近期设备页的乱码提示修复，进一步提升了这一链路的完成度。

#### 症状自查与医生问诊链路落地过程

项目将原有 3D 干预页重构为 2.5D 症状自查与辅助诊断向导，同时保留旧康复页入口，形成“新症状链路 + 旧康复链路”并存的结构。症状自查页通过热点点击、症状标签和结果页组织轻量输入，再把上下文带入 AI 医生问诊页。医生页则结合本地与云端能力完成追问与说明。整个过程体现了从“界面改造”到“链路重组”的技术实践思路。

#### 医检报告分析可读化过程

报告分析的实践重点不在 OCR 接口本身，而在如何让 OCR 结果真正进入用户可读层。项目通过 `MedicalReportParser` 和 `MedicalReportDraftFormatter` 等本地能力，以及云端 `report/understand` 能力，把报告结果整理为摘要、重点关注项、逐项解释和下一步建议。近期还针对“未登录或云端不可用时内容空白”的问题补上了本地可读兜底，使报告模块更贴近真实场景。

#### 机器人讲解与本地预热 + 云端生成策略落地过程

桌面机器人与数字人是项目较有辨识度的功能之一，但其真正的技术价值在于讲解策略。项目实现了页面进入先用本地预热文案和本地缓存音频快速响应，点击触发时再调用云端 AI 生成更强上下文的解释。这种设计一方面避免了每次切页都等待云端结果，另一方面保留了更自然的页面解释能力。配合语音播报开关、页面级隐藏规则和对不同页面的专属文案，机器人系统从“装饰元素”转化为真正的辅助讲解层。

## 结果验证

### 构建与部署验证

Android 端当前可以通过 `gradlew.bat :app-shell:assembleDebug` 构建调试包，通过 `:app-shell:testDebugUnitTest` 运行本地单元测试，通过 `:app-shell:lint` 运行 lint。云端 `cloud-next` 可通过 `npm run build` 和 `npm run lint` 完成基础验证。自定义域名、账号接口、密码重置接口、确认邮件接口和 Android API 基地址已经统一到新的生产域名，说明部署链路已基本闭环。

### 单元测试与仪器测试验证

当前项目已具备 JVM 单元测试和 Android 仪器测试能力。虽然个别历史 UI 断言仍可能与当前页面表现不完全一致，但整体测试链路已能够运行，这说明项目并不是只依赖人工演示。更重要的是，测试范围已覆盖 AI 决策、解析器、设备能力与页面行为等关键部分，为后续继续提高代码质量奠定基础。

### 账号、问诊、报告分析和干预闭环验证

账号链路方面，注册、登录、确认邮件和忘记密码已经完成云端统一与路由修复，重置密码链接能够正确落入重置页面。问诊链路方面，症状自查到医生问诊上下文传递已形成闭环。报告分析方面，本地草案和云端增强均已具备，避免出现纯 JSON 或纯 OCR 原文暴露给用户的情况。干预链路方面，报告、自查和首页建议都能够进一步通向呼吸训练、旧康复页和干预执行模块，说明系统真正具备“从结果到行动”的技术完成度。

### 当前局限与后续研究方向

尽管当前系统已经形成可运行闭环，但仍存在进一步优化空间。第一，feature 模块虽然已有真实业务承接，但整体多模块架构仍可继续收敛，以进一步提升代码质量与可维护性。第二，症状热点、问诊深度和报告解释质量仍可继续细化，尤其在复杂多症状场景下仍有提升空间。第三，设备链路仍需更多真机环境和长期佩戴数据验证，以进一步评估稳定性。第四，机器人讲解虽然已经具备本地预热与云端增强能力，但未来仍可以引入更精细的上下文策略。第五，项目在商业化和更广泛落地前，还需要更多来自真实试点场景的数据支持。

总体而言，长庚环的技术研究结果已经证明：在一个面向居家健康管理的系统中，智能戒指、Android 单入口 App、云端 AI、规则兜底、本地数据库、低门槛症状输入、医检报告理解和执行反馈并非必须分散存在，而是可以被整合到一个真实可运行的端云协同系统中。项目的主要贡献不在于某一项技术绝对领先，而在于把这些技术以合理的边界、合理的分工和合理的表达整合成一条完整链路。
