# 项目测试文档

版本号：V1.0  
项目名称：长庚环  
英文名称：VesperO  
团队名称：夜巡者

## 文档说明

本文档用于描述长庚环项目当前阶段的测试策略、测试范围、测试环境、测试资产、测试结果、问题分析和后续改进建议。文档的目标不是单纯给出“是否通过”的结论，而是将项目从硬件接入、Android 端运行、云端 API、AI 推理、OCR 报告理解到干预执行闭环中的关键质量风险进行系统梳理，并给出适合答辩、验收和后续迭代的统一测试口径。

长庚环是一个由智能戒指、Android 单入口 App、云端分析服务和 AI 助手共同组成的端云协同健康系统。它既不是单一的睡眠监测 App，也不是纯软件形态的互联网医疗问诊工具，而是将智能硬件连续感知能力与移动端交互能力、云端理解能力以及康复辅助能力整合为一个可验证的闭环。基于这一定位，测试工作也不能仅停留在“页面能打开、按钮能点击、接口能返回”这一层，而必须覆盖以下四个方面：

1. 基础可运行性。包括 Android 端构建、安装、页面导航、云端服务、关键 API、数据库访问、BLE 链路、OCR 和 AI 组件是否能在目标环境中启动并维持稳定运行。
2. 业务闭环完整性。包括从睡眠/恢复监测到晨间结果展示，从症状自查到问诊和报告理解，再到干预执行与复盘等关键链路是否形成完整闭环。
3. 复杂场景稳定性。包括弱网、断网、账号未确认、邮件回跳、OCR 识别不完整、讯飞配置缺失、设备重连、真机权限不足等场景下的降级策略是否合理。
4. 可维护与可扩展性。包括单元测试、仪器测试、构建命令、模块边界、日志与错误提示是否有助于后续协作与评分，尤其是在比赛文档和答辩环节，是否能够清晰说明“项目不仅能跑，而且工程质量可控”。

基于当前仓库代码事实，Android 运行入口已经统一为 `:app-shell`，旧 `app/` 已不再参与 APK 构建；云端统一部署在 `https://cloud.changgengring.cyou/`；主要功能包括今日页、医生页、趋势页、设备页、我的页、症状自查、医检报告分析、干预中心、呼吸训练与执行复盘。本文档将围绕这些真实能力展开，不虚构不存在的功能，也不忽略当前仍需继续补完的边界。

## 1 测试计划

### 1.1 测试策略与目标

长庚环项目的测试策略采用“分层验证 + 场景闭环 + 风险导向”的组合方式。分层验证强调从代码级、模块级、页面级、系统级逐层收敛问题，避免把所有问题都堆到最终联调阶段处理；场景闭环强调不能只验证单点功能，而要验证健康管理场景中完整的数据流和动作流，例如从设备连接到数据采集，再到结果展示和建议执行；风险导向则强调对最可能影响用户体验、比赛评分和后续维护的关键路径优先投入测试资源，包括账号链路、BLE 链路、AI 服务链路、OCR 链路和干预执行链路。

从测试重点看，本项目当前并非传统信息管理系统，而是软硬件协同、端云协同和 AI 协同三者同时存在的复合系统，因此测试重点集中在以下五条主线。

第一，单一 Android 入口的稳定运行。项目已完成从 `app-shell + legacy sourceSets` 到 `app-shell` 单入口的收敛，因而测试需要重点关注是否仍存在类缺失、资源缺失、导航失效、运行时引用错误等架构收口后常见问题。

第二，账号与云端同步能力的有效性。登录、注册、邮箱确认、重发确认邮件、忘记密码和密码重置回跳是当前用户最早接触到的链路，也是最容易出现“产品看起来没问题，但核心入口被卡死”的部分，因此必须作为一级测试对象。

第三，智能戒指与设备链路的可靠性。设备页承担了扫描、连接、断连、同步与状态展示职责，既涉及 BLE 协议，也涉及前台服务、权限、UI 提示和本地数据库，因此其测试不能只覆盖“扫得到设备”，还要覆盖“连接后是否能形成稳定数据链路”。

第四，AI 与 OCR 结果是否真正“用户可理解”。本项目不是简单显示 OCR JSON，也不是把 AI 回答原样堆给用户，而是强调可读化、辅助化和保守表达。因此，症状自查、AI 医生问诊、医检报告可读报告、桌面机器人讲解都需要验证文本是否符合医疗辅助产品的表达原则。

第五，监测到干预的闭环是否真正成立。长庚环的核心价值在于“监测 -> 分析 -> 建议 -> 执行 -> 反馈”的闭环。若设备能连、报告能出、页面能看，但用户无法进入执行、执行后无法记录、记录无法回写趋势，那么这个系统的价值链条实际上是断裂的。因此，测试必须覆盖闭环执行质量，而非只覆盖单页显示。

本阶段测试总目标可以概括为四点：

1. 保证当前版本可以作为一个完整的 Android 软件运行，且主要功能入口齐全。
2. 保证云端服务与 Android 端配置一致，关键认证和 AI 接口可正常访问。
3. 保证用户能够在主要健康管理场景中完成关键闭环操作，不会被入口、权限、链路错误阻断。
4. 保证工程质量在可答辩、可验收、可持续迭代的范围内，至少具备构建通过、单元测试可运行、关键仪器测试可挂载、关键云端接口可联调的基本条件。

为实现上述目标，本文档将测试分为三层：

- 单元测试层：验证核心算法、规则引擎、文本整理器、问诊推理器等纯逻辑组件的正确性。
- 功能测试层：验证账号、设备、症状自查、问诊、医检报告、干预和个人中心等用户可见功能。
- 系统测试层：验证模型性能、端云协同、BLE 稳定性、安全与隐私等跨模块场景。

同时，本文档会将每一层中的测试对象继续拆为“已完成的验证”“当前已发现的问题”“建议补充的验证项”三类，以确保既忠于当前项目事实，也保留后续完善空间。

### 1.2 测试范围

本项目测试范围从广度上覆盖 Android 客户端、云端服务、BLE 链路、本地 AI 与 OCR 兜底能力、云端 AI 理解与讲解链路；从深度上覆盖构建验证、核心业务功能验证、异常场景验证、性能体验验证与安全边界验证。整体上，测试范围与项目当前产品闭环高度一致。

#### 1.2.1 账号、鉴权与云端同步范围

账号与云端同步部分覆盖以下对象：

- 注册、登录、重发确认邮件、忘记密码、密码重置页面回跳。
- 云端 `auth/login`、`auth/register`、`auth/resend-confirmation`、`auth/password-reset`、`user/profile` 等接口。
- Android 本地会话存储、资料展示、退出登录与重新进入。
- 生产域名 `https://cloud.changgengring.cyou/` 与 Android `BuildConfig.API_BASE_URL` 的一致性。

这一部分的重点不只在“接口是否有返回”，而在“用户是否能从 App 中顺畅完成注册、确认、找回密码、重新登录”的完整体验。由于本项目此前曾出现“忘记密码邮件点击后落到登录页”“未确认邮箱被误判为注册失败”“客户端和旧部署不一致”等问题，因此该部分在本轮测试中被定义为高优先级范围。

#### 1.2.2 设备连接、蓝牙与数据同步范围

设备链路测试覆盖以下对象：

- 设备页扫描、搜索列表、设备名称展示、连接状态提示、断开与重连。
- Nordic BLE 基础能力、Hi90B 协议命令构造、历史数据量查询与实时传输启停。
- `DataCollectionService`、BLE 数据回调、本地数据库写入、同步相关触发逻辑。
- 设备页 toast、加载状态、错误提示和中文文案完整性。

该部分既是软硬件联动的关键入口，也是对项目真实性的直接证明。因为长庚环不是纯软件原型，而是包含智能戒指侧协议与采集流程的完整方案，所以设备链路测试会直接影响项目说服力。

#### 1.2.3 今日页、趋势页与恢复评估范围

此部分覆盖：

- 今日页晨报、恢复分、风险提示、行动入口。
- 趋势页图表、历史数据展示、趋势解释。
- 本地异常检测与恢复评估逻辑在 UI 的呈现。
- 数据缺失、弱网、未登录等情况下的页面降级行为。

测试的关注点并不是图表是否华丽，而是页面能否传递出“今天该如何行动”的信息，是否与项目主叙事一致。

#### 1.2.4 AI 医生问诊与症状自查范围

此部分覆盖：

- 医生页文本问诊、连续语音问诊、数字人讲解链路。
- 症状自查页的 2.5D 症状定位、正反面切换、症状 chips、辅助判断结果。
- 页面间联动，例如症状自查转入医生问诊、旧康复场景保留入口、干预页衔接等。
- 本地预热文案、桌面机器人说明文案和点击后的云端 AI 生成文案。

由于该部分是当前产品差异化的重要展示点，测试不仅要验证功能，还要验证表达质量、提示时机和交互节奏。

#### 1.2.5 医检报告分析范围

此部分覆盖：

- OCR 图片识别、原始文本提取、指标解析。
- AI 可读报告生成、异常项总结、下一步建议。
- 未登录场景下的本地可读兜底。
- 编辑 OCR 文本后重新解析的能力。

这一模块的测试关键在于：不能只显示 OCR 原始 JSON 或原始长文本，而要确认整理后的结果对普通用户可读，并且不会出现“页面空白”“有结果对象但没有文本呈现”等严重体验问题。

#### 1.2.6 干预中心、康复辅助与复盘范围

此部分覆盖：

- 干预中心的主入口与分流逻辑。
- 呼吸训练、康复辅助动作、执行记录。
- 旧 3D 干预中心保留入口与新页面并存状态。
- 复盘页的完成率、体感反馈与效果变化说明。

这一部分与项目主线“执行与反馈”直接相关，是从“能分析”走向“能帮助用户做事”的关键。

#### 1.2.7 个人中心、通知与隐私范围

此部分覆盖：

- 我的页资料展示、字母头像、云端邮箱、登录状态。
- 通知设置、机器人语音播报开关、系统通知权限状态。
- 数据与隐私、退出登录、清除本地会话与偏好数据。
- 关于项目页面、版本信息与项目简介展示。

个人中心虽然不是项目的技术亮点，但它关系到账号闭环、隐私可信度和系统完整性，因此也纳入正式测试范围。

### 1.3 测试环境

#### 1.3.1 Android 软硬件环境

Android 侧当前唯一运行模块为 `:app-shell`。根据构建配置，项目采用如下环境基线：

- Android Gradle Plugin：8.7.3
- Kotlin：1.9.24
- compileSdk：35
- targetSdk：35
- minSdk：24
- Java / Kotlin 目标版本：11

这意味着从 Android 7.0 起的设备具备安装与运行可能，而高版本设备将承担更完整的权限与通知场景验证。应用构建命令为：

```powershell
gradlew.bat :app-shell:assembleDebug
gradlew.bat :app-shell:testDebugUnitTest
gradlew.bat :app-shell:connectedDebugAndroidTest
gradlew.bat :app-shell:lint
```

在设备形态上，测试优先覆盖以下三类环境：

- 真实 Android 手机：用于账号、桌面机器人、讯飞链路、通知权限、弱网场景验证。
- Android 平板：用于大屏页面布局与多栏结构验证，尤其是症状自查和设备页。
- 模拟器：用于快速执行仪器测试和回归验证。

#### 1.3.2 云端与网络环境

云端部署当前统一为：

- 生产域名：`https://cloud.changgengring.cyou/`
- 技术栈：Next.js + Supabase + Route Handlers
- 构建与检查命令：

```powershell
cd cloud-next
npm run build
npm run lint
```

测试需要覆盖三类网络状态：

- 正常网络：用于常规登录、问诊、OCR、云端 AI 生成和同步操作。
- 弱网环境：用于验证重试、超时、用户提示和本地兜底逻辑。
- 断网环境：用于验证页面是否崩溃、是否有本地保守结果、是否能给出明确提示。

#### 1.3.3 BLE 与设备环境

长庚环项目的硬件侧为智能戒指样机，协议层基于 Hi90B 指令体系和本地自定义 BLE 流程。测试环境包括：

- 支持 BLE 的 Android 终端。
- 戒指样机与相应蓝牙广播环境。
- 允许频繁重连、断连、扫描和历史数据查询的测试时段。

BLE 测试与纯软件功能不同，容易受到环境中其他蓝牙设备、手机省电策略、权限开关、距离和遮挡影响。因此系统测试中会将其单独作为稳定性项处理。

#### 1.3.4 AI 与讯飞能力环境

当前 Android 端集成了讯飞相关能力，包括语音听写、语音合成、OCR、Spark、AIUI/数字人等能力。配置由 `local.properties` 注入至 `BuildConfig`。当前测试重点包括：

- 配置是否真正进入构建产物。
- 某项能力未配置时是否给出明确提示，而不是静默失败。
- 某项配置写错、注释错位或账号环境不完整时，是否能被准确识别。

此处需要特别说明，讯飞类能力对开发机本地配置、账号后台、场景 ID、数字人 avatar id、AIUI appId 和网络环境都比较敏感，因此在测试报告中必须保留“配置正确但服务端账号条件不足时仍可能失败”的说明，不宜把所有问题都归因为代码。

#### 1.3.5 测试工具与辅助方法

本轮测试采用的主要工具包括：

- Gradle：用于构建、单元测试、lint、仪器测试。
- adb：用于安装、日志查看、仪器测试和基础自动化验收。
- Playwright：用于云端页面、登录页、重置密码页和部分 Web 路由验证。
- curl / PowerShell：用于验证生产域名下 API 响应。
- python-docx / pandoc：用于文档生成与模板套用。

测试方法上，采用“自动化命令验证 + 人工场景检查 + 定点问题回归”三者结合的方式。自动化验证保证构建与基础路径稳定，人工场景检查负责用户体验与跨页面链路，定点回归则用于处理登录链路、邮件链路、乱码、AIUI 配置等近期高频问题。

## 2 单元测试

单元测试是长庚环当前工程质量的重要支撑点。由于项目涉及硬件、云端、AI 和复杂 UI，很多问题无法仅靠端到端测试定位。因此，我们优先使用单元测试覆盖纯逻辑层，包括医生问诊决策、报告文字整理、PPG 峰值检测和医生推理引擎等组件，以保证关键规则和推理逻辑的可复现性。

当前已存在并可确认的单元测试包括：

- `app-shell/src/test/java/com/example/newstart/ui/doctor/DoctorDecisionEngineTest.kt`
- `feature-doctor/src/test/java/com/example/newstart/ui/doctor/DoctorInferenceEngineTest.kt`
- `app-shell/src/test/java/com/example/newstart/util/MedicalReportDraftFormatterTest.kt`
- `app-shell/src/test/java/com/example/newstart/util/PpgPeakDetectorTest.kt`
- `app-shell/src/test/java/com/example/newstart/ExampleUnitTest.kt`

当前阶段已确认的命令结果包括：

- `gradlew.bat :app-shell:testDebugUnitTest` 通过
- `gradlew.bat :feature-doctor:testDebugUnitTest` 通过

这说明至少在 JVM 侧，项目的纯逻辑测试链路已经打通，且不依赖真机或模拟器即可进行回归。

### 2.1 AI 问诊与决策模块

#### 2.1.1 测试用例与结果分析

AI 问诊与决策模块主要包括医生页中的问诊状态组织、用户主诉理解、结构化追问和建议整理逻辑。由于最终的问诊内容既受本地规则影响，也受云端问诊接口影响，因此单元测试重点不在于“最终话术是否完全一致”，而在于规则骨架是否稳定、结果结构是否可解释、关键字段是否能在不同输入组合下稳定生成。

当前单元测试覆盖的核心对象包括 `DoctorDecisionEngine` 和 `DoctorInferenceEngine`。其中，`DoctorDecisionEngineTest` 更偏向本地规则与建议骨架的验证，关注输入的主诉、风险级别、伴随症状、持续时间等字段变化后，输出的决策方向是否保持一致；`DoctorInferenceEngineTest` 则更偏向医生侧推理过程中的摘要、意图识别、结构化信息聚合与输出组织。

在用例设计上，本模块重点覆盖以下场景：

1. 主诉单一、伴随症状少的低风险输入。例如用户仅描述轻微不适、持续时间短且无红旗症状时，系统是否能给出保守的辅助判断和后续建议。
2. 多症状输入的结构化归并。例如用户既输入头痛，又输入胸闷或出汗时，系统是否能避免简单拼接，而是对主要症状和高风险症状进行优先排序。
3. 风险分层变化带来的输出差异。例如加入胸痛、呼吸困难、明显加重等信号后，建议是否从“观察与轻度处理”升级为“建议尽快线下就医”。
4. 输出完整性验证。即使用户输入信息较少，也应当输出明确的下一步建议、追问方向或说明，不应出现空结果、字段缺失或不成句表达。

从当前测试资产和实际代码行为看，这一模块的单元测试能够较好地承担以下作用：

- 提前发现逻辑分支回退问题。
- 为 UI 层提供稳定的结构化输出约束。
- 在云端问诊策略调整时，提供本地规则层的基线参照。

综合分析表明，本模块单元测试的价值不在于替代临床判断，而在于确保系统在“辅助问诊”这一定位下输出稳定、保守且结构清晰的结果。当前该部分逻辑测试能够通过，说明其规则骨架已经具备可维护性。

#### 2.1.2 测试结果综合分析及建议

从综合结果看，AI 问诊与决策模块的单元测试处于“结构稳定、细节仍需继续扩展”的状态。稳定体现在：

- 关键对象已有专门单测，不再完全依赖 UI 点点点回归。
- 医生推理测试已迁入 `feature-doctor`，模块归属更加清晰。
- 问诊结果的基础结构与建议方向可通过代码验证，而不是只在运行时观察。

仍需继续加强的点包括：

1. 更多边界值输入。当前应补充“输入过短”“全为空格”“纯口语碎片”“多个冲突症状同时出现”等用例，验证推理器在脏数据下的稳健性。
2. 对话多轮状态的单元化验证。随着医生页逐步承担更真实的多轮问诊职责，需要验证上下轮对话状态是否会污染本轮建议。
3. 云端回包结构变化后的兼容性测试。当前医生页并非完全本地逻辑，后续应增加 DTO 与映射层测试，避免服务端字段调整导致客户端静默异常。
4. 语音转文本噪声输入的容错验证。连续语音问诊容易引入口语化、重复词或误识别，建议将其抽成更明确的用例集。

结论上，本模块已达到“可支撑比赛展示与日常回归”的水平，但若要进一步强化答辩中的工程质量表现，应在多轮对话和脏输入场景上继续扩充测试集。

#### 2.1.3 测试经验总结

本模块的测试经验可以总结为三点。

第一，问诊类模块不能只测“最终文案”，必须优先测结构和决策骨架。因为文案容易随着产品迭代和 AI provider 切换而变化，但结构化结论、建议级别和下一步动作是长期稳定的质量锚点。

第二，医生问诊模块必须区分“本地逻辑保证”和“云端增强能力”。前者适合单测，后者更适合集成测试。将这两者混在一起会导致测试不稳定，也不利于问题归因。

第三，问诊模块最需要的不是追求“像真人”，而是保证“在风险场景下不漏判，在普通场景下不夸大”。这一点与项目定位高度一致，也应作为后续测试扩充时的总原则。

### 2.2 医检报告解析与可读化模块

#### 2.2.1 测试用例与结果分析

医检报告分析是长庚环当前较有辨识度的能力之一。该模块不仅需要完成 OCR 文本提取，还需要将原始识别结果转化为普通用户可以理解的文字说明。因此，单元测试重点不只在于文本解析是否“能跑通”，更在于结果是否稳定、可读、不会输出开发态信息。

当前已存在的核心单元测试对象是 `MedicalReportDraftFormatterTest`。该测试围绕报告整理草案生成逻辑展开，重点验证：

- 报告摘要是否能从结构化指标中生成可读结论。
- 有异常指标时，是否能优先输出重点关注项。
- 无异常或结构不完整时，是否仍能输出可解释的降级说明。
- 可读化文案中是否保留值、范围、偏高偏低等关键信息，而不是只输出空泛判断。

本模块的测试用例设计围绕以下几类输入展开：

1. 指标完整且有少量异常。测试关注重点是是否能准确提炼 1 至 3 项重点关注，并给出通俗解释。
2. 指标完整且异常较多。测试关注是否会在摘要中提升风险提示，并建议用户进一步复查或咨询医生。
3. OCR 文本存在但结构化指标不足。测试关注是否仍能基于原文生成可读说明，而不是页面空白。
4. 结果来自云端 AI 和本地兜底两种路径。测试关注最终文本结构的一致性，而不是完全一致的字句。

从当前实际代码和近期问题修复情况来看，报告可读化模块已完成一个重要转变：从“把 JSON 或原始文本直接堆给用户”转为“优先展示可读摘要，必要时保留原文折叠区”。这意味着测试必须把“页面有没有内容”“内容是不是普通用户能看懂”作为核心标准。

测试分析表明，`MedicalReportDraftFormatter` 的存在，为这一模块建立了稳定的本地基线。即使云端 AI 理解结果未返回，或者用户处于未登录状态，也可以通过本地格式化结果确保页面不至于空白。这种设计显著降低了云端波动对用户可用性的影响，也让测试工作更容易建立清晰的预期。

#### 2.2.2 测试结果综合分析及建议

综合来看，医检报告解析与可读化模块目前处于“本地兜底逻辑较稳、真实 OCR 输入复杂性仍需持续覆盖”的状态。

已经比较扎实的部分包括：

- 有明确单元测试承接可读化逻辑。
- 已形成“云端结果优先，本地草案兜底”的结构。
- 当前页面不再只展示原始 JSON 或完全空白结果。

仍建议继续加强的部分包括：

1. 增加更多报告类型的样本覆盖。例如血常规、生化、激素、心电报告等，不同格式下字段命名差异较大，需验证本地格式化对未知指标是否稳健。
2. 增加 OCR 脏文本测试。真实拍照环境中可能出现行断裂、错字、单位缺失等问题，这部分比“理想 OCR 文本”更具有实际价值。
3. 增加“登录态/未登录态”“云端可用/云端不可用”双维度测试。当前模块最容易出问题的不是普通 happy path，而是鉴权或服务异常导致的空状态。
4. 增加表达规范测试。尤其要避免“确诊式文案”、避免技术字段裸露，保持“辅助理解、不替代医生结论”的表述。

总体判断是，该模块已经具备较好的展示与答辩基础，但如果希望在正式测试文档中更具说服力，应增加真实样本输入与 OCR 噪声条件下的专项用例。

#### 2.2.3 测试经验总结

本模块的测试经验说明：对于“用户最终看到的是自然语言”的能力，不应只用最终 UI 人工检查，而必须在底层建立格式化器与结构化结果的测试锚点。这样一来，云端 AI 结果变化、页面文案调整或回退路径触发时，仍然可以保证核心表达不会崩。

另一个经验是，可读化能力需要始终保留兜底。与纯云端依赖相比，本地草案虽然不如云端理解丰富，但它为测试和产品稳定性提供了“最低可用结果”，这一点对于比赛场景和移动端场景都很关键。

### 2.3 生理信号分析与异常检测模块

#### 2.3.1 测试用例与结果分析

长庚环项目以智能戒指连续感知为底座，因此生理信号分析与异常检测模块是整个系统可信度的基础。在当前代码中，`PpgPeakDetectorTest` 是已明确存在且可验证的单元测试资产，主要用于验证 PPG 波峰识别、RR 间期估计、心率估算和噪声情况下的基础稳定性。

该测试模块的重要意义在于，它不依赖 UI、不依赖云端，也不依赖真机设备即可对生理信号处理逻辑进行回归验证。对于一个软硬件结合项目而言，这是非常重要的工程质量支撑点，因为真实设备联调成本高、重复性差，而单元测试能提供可重复执行的基线。

测试用例主要包括：

1. 合成 PPG 信号下的心率估计是否落在合理范围。
2. 噪声干扰条件下是否仍能提取主要峰值。
3. 极端输入下是否会崩溃或输出无意义数据。
4. 信号质量下降时，系统是否给出更保守的估计。

结合当前测试类的设计思路可以判断，本模块的核心目标并不是在单测中复现完整的临床级分析，而是在工程层面确保基础信号处理逻辑稳定可靠，不会因为后续算法调整导致基础指标完全漂移。

#### 2.3.2 测试结果综合分析及建议

当前结果说明，生理信号分析模块至少已经具备以下优点：

- 存在可重复的基础单元测试，不完全依赖真机数据。
- 能够用合成信号快速验证算法回归。
- 为异常检测和恢复评估提供了底层质量锚点。

需要继续补充的方面包括：

1. 与真实采样节奏一致的测试样本。合成数据便于稳定回归，但仍不能替代真实设备噪声特征。
2. 更复杂的信号质量标签测试。项目目前有“信号质量”概念，但建议把低质量、高抖动、短片段输入做成更明确的断言集。
3. 与上层异常检测结果的一致性测试。即底层峰值检测变化后，是否会影响异常分和恢复分的稳定性。

因此，本模块的结论是：已具备基础回归能力，但要在比赛答辩中更有说服力，还应配合真实戒指样本和异常样本展示更完整的验证证据。

#### 2.3.3 测试经验总结

对于软硬件项目来说，底层信号处理单元测试是很容易被忽视的部分，但它恰恰最能体现工程成熟度。长庚环目前至少已经在 PPG 峰值检测上建立了这一基础。后续如果继续扩展血氧、温度、HRV 等派生特征的测试，将能进一步强化项目在“硬件不是摆设，而是真实进入闭环”的说服力。

### 2.4 干预策略与执行状态模块

#### 2.4.1 测试用例与结果分析

虽然当前项目尚未为所有干预策略类对象补齐独立单测，但从仓库结构和运行逻辑看，干预策略与执行状态是产品闭环不可缺少的一层。干预中心、呼吸训练、复盘页、InterventionRepository 与 PrescriptionRepository 等对象共同构成了从建议到执行的主线。

本模块当前的单元测试更偏“建议补充项”，但仍可以基于现有代码事实明确测试设计方向。主要测试对象包括：

- 干预任务状态流转：未开始、进行中、已完成、已放弃。
- 不同风险等级对应的建议强度分层。
- 执行记录聚合后在复盘页和趋势页中的表现。
- 云端日处方、执行上传和效果回写之间的数据一致性。

就当前项目而言，干预策略的正确性不只是“给没给建议”，而是：

1. 建议是否与当天状态匹配。
2. 建议是否能落到可执行动作，而不是抽象口号。
3. 执行后是否能留下记录，供下一次推荐参考。

因此，虽然这一部分的自动化单测还不如问诊与报告格式化那样成熟，但它已经在产品逻辑上形成明确的测试对象，适合在后续版本中补足。

#### 2.4.2 测试结果综合分析及建议

本模块当前更适合作为“测试覆盖正在完善中的模块”来描述。它的优势在于业务价值清晰：所有监测和分析结果最终都要落到“用户能不能做点什么”上。它的短板在于自动化资产仍不够充分。

建议后续优先补充以下单元测试：

1. 干预建议选择逻辑测试，验证不同恢复分或风险标签下的命中结果。
2. 执行状态聚合测试，验证列表页、复盘页和趋势页展示的一致性。
3. 云端 effect/trend 与本地缓存结果的合并测试，避免重复统计或空状态误判。

如果从评分角度看，补齐这一模块的逻辑测试会非常有利，因为它恰好体现项目最核心的差异点：不是只看数据，而是把数据变成行动。

#### 2.4.3 测试经验总结

干预模块的经验是：业务价值越核心的模块，越不能只靠 UI 演示支撑。即使当前还没有全部补完，测试文档中也应明确指出其逻辑分层、状态流和预期补测方向，这本身就是工程意识的一部分。

## 3 功能测试

功能测试关注用户真实能看到和能操作的内容，是比赛答辩和最终展示中最直观的质量证明。长庚环的功能测试以“单入口 App + 端云协同闭环”为前提，重点验证用户从打开 App 到进入核心健康场景时，是否能完成完整路径而不被错误、中断、空白页或配置问题阻断。

### 3.1 账号与云端同步功能

#### 3.1.1 测试用例与结果分析

账号与云端同步功能是长庚环所有云端增强能力的基础。一旦这一层出问题，问诊、报告理解、资料同步、重置密码乃至云端 AI 讲解都会受到影响。因此该模块在本轮功能测试中被列为最高优先级对象之一。

核心测试用例如下：

1. 新用户注册后进入“待邮箱确认”状态，而不是被直接判定为失败。
2. 对未确认邮箱再次注册时，系统应提示继续确认或重发邮件，而不是陷入死循环。
3. 登录时错误凭据返回明确错误，正确凭据可正常进入已登录状态。
4. 忘记密码请求发出后，邮件链接应进入 `/reset-password` 页面，而不是回到 `/login`。
5. 点击包含 recovery 参数的旧样式登录链接时，登录页应自动跳转到重置密码页。
6. 已登录后我的页应显示云端邮箱、用户名和资料状态，退出登录后会话应被清理。

根据当前项目修复和验证记录，账号链路已经完成以下关键收口：

- Android 端指向的生产域名已统一为 `https://cloud.changgengring.cyou/`。
- 生产云端的 `login`、`register`、`resend-confirmation`、`password-reset` 路由均存在。
- `password-reset` 和 `resend-confirmation` 已不再返回 404。
- 密码重置链接落到登录页时，会自动转发至 `/reset-password`。
- 云端注册接口对“待确认邮箱”改用更合理的 pending 语义，而不再直接作为失败。

从功能测试分析看，当前账号链路已经从“接口存在但用户体验断裂”提升为“主路径可用、邮件链路可闭环”的状态。其意义在于，App 与云端配置已统一，用户不会再因为环境错配而被卡在入口阶段。

#### 3.1.2 测试结果综合分析及建议

综合分析认为，账号与云端同步模块已从风险高发区进入可控区间，但仍建议在正式验收前继续补足以下检查：

1. 多邮箱服务商验证。不同邮箱对 Resend + Supabase 邮件的接收率不同，应补充 QQ 邮箱、163 邮箱、Gmail、Outlook 等实际收件测试。
2. 邮件时效与失效行为验证。需要验证旧链接失效时提示是否明确，避免用户误以为系统错误。
3. 弱网场景下登录提示验证。网络抖动时，必须区分“密码错误”“服务暂时不可达”“邮箱未确认”等不同情形。
4. 我的页资料刷新验证。应确保登录、编辑资料、退出登录后，个人中心顶部显示内容实时同步。

目前该模块可得出的结论是：链路已打通，配置已统一，关键回跳问题已修复；后续重点转向“提高收件率、改善错误提示和做更多邮箱服务商兼容验证”。

#### 3.1.3 测试经验总结

账号问题往往不是业务逻辑最复杂的部分，却最容易对用户形成“一上来就用不了”的负面印象。本项目的经验是，认证链路测试必须同时覆盖客户端、服务端、邮箱服务和回跳地址四层，不能只盯某一层。只有前后端版本、域名、回跳地址和 SMTP 同步一致，账号链路才算真正打通。

### 3.2 戒指连接与设备管理功能

#### 3.2.1 测试用例与结果分析

设备管理功能承担了“让真实硬件接入系统”的第一责任。测试重点包括：

1. 打开设备页后是否能正常进入扫描状态。
2. 扫描设备过程中的加载文案、提示文案是否正常中文显示。
3. 搜索结果列表是否能显示设备名称或合理兜底名称。
4. 点击设备后是否能触发连接状态变化。
5. 断开连接、连接失败、读取工作参数失败时是否有明确提示。
6. 同步入口是否与设备状态、云端状态和本地服务状态保持一致。

结合近期问题回归情况，设备页目前已完成一项重要修复：运行态中的用户可见乱码，尤其是扫描设备时的 toast 和加载状态文本，已回到正常中文。这一修复虽然不属于“大功能新增”，但属于质量体验的重要加分项，因为乱码会显著降低系统可信度。

从代码结构和页面行为来看，设备页测试不能只停留在视觉层，还必须关注 BLE 连接链路和状态提示是否一致。例如：

- 页面显示“连接成功”时，是否真正进入了可同步状态。
- 页面显示“扫描中”时，是否同时触发了蓝牙搜索，而不是只有动画没有业务。
- 设备断开后，是否回到明确的可重连状态，而不是停留在错误中间态。

当前阶段，设备页功能测试的结论可以表述为：页面和基础状态提示已具备可展示性，BLE 作为真实硬件链路已接入运行体系，后续需要在真机与戒指样机条件下继续补充更高强度的压力验证。

#### 3.2.2 测试结果综合分析及建议

设备模块的当前强项是：协议存在、页面存在、状态入口存在、硬件属性明显；这意味着项目不是“画了设备页”的软件原型，而是具备真实接入路径的系统。

当前主要补测方向包括：

1. 不同 Android 版本和不同品牌手机的蓝牙权限兼容。
2. 长时间后台连接或前台服务运行时的稳定性。
3. 设备扫描列表中同名设备、弱信号设备、断电设备的行为。
4. BLE 数据回调和本地数据库写入的一致性检查。

若从比赛展示角度看，设备页测试需要至少再补一轮“真实戒指连接 - 数据同步 - 结果展示”的视频或截图证据，以进一步提高可信度。

#### 3.2.3 测试经验总结

设备类功能的测试经验是：真正重要的不是页面上写了多少状态，而是状态和底层 BLE 行为是否一致。后续所有设备测试都应围绕“状态是否真实反映链路状态”来设计，而不是仅看页面是否刷新。

### 3.3 症状自查与辅助诊断功能

#### 3.3.1 测试用例与结果分析

症状自查页是当前项目中最具互联网医疗感的新增能力之一。测试重点包括：

1. 进入症状自查页后，页面结构是否完整，2.5D 症状定位图是否正常显示。
2. 前侧/后侧切换是否正确替换图片。
3. 常见症状 chips 是否与当前页面主题一致，例如头痛、太阳穴痛、出汗、胸闷、腹痛、手腕痛、膝盖痛等。
4. 用户可通过点击热点或直接点击症状 chips 完成输入。
5. 选择症状后是否能进入辅助判断结果，并显示风险等级、疑似方向和下一步建议。
6. 结果页是否能跳转到 AI 医生问诊、上传报告或康复辅助动作。
7. 症状自查页是否仍保留进入旧康复辅助页的显式入口。

从当前实现和回归结果看，该模块已完成若干重要收口：

- 旧 3D 干预中心未被删除，而是保留为独立入口。
- 新症状自查页与旧康复辅助页可以并存，避免因重构造成旧功能消失。
- 右侧按钮选中态已改为按钮高亮，残留的蓝色圆点问题已在界面逻辑中被处理。
- 前后侧展示图已替换为新的症状定位图，左侧症状项也已按图片内容更新。

功能测试分析表明，这一页面当前已经具备良好的“引导用户选择症状并进入下一步”的能力。虽然它本质上仍然是辅助判断页面，而非诊断系统，但就用户体验而言，它已经从“炫技 3D 场景”转向更贴合医疗场景的轻交互引导器。

#### 3.3.2 测试结果综合分析及建议

当前症状自查模块适合定义为“功能完成、视觉与点位精细度可继续优化”的状态。建议后续重点补充：

1. 热点位置与图片发光点的像素级对齐验收。
2. 横屏、平板和窄屏手机的布局一致性验证。
3. 多症状同时选择后的聚合结果验证。
4. 红旗症状命中时的高风险提示流程验证。

由于症状自查是用户首次主动输入健康问题的重要入口，因此在功能测试文档中建议重点展示“输入方式多样”“结果表达保守”“可继续跳转到问诊与康复”的三点优势。

#### 3.3.3 测试经验总结

这一模块的经验是，医疗辅助产品的交互重点不在于做得多炫，而在于用户能不能以最低认知负担完成输入。测试中最关键的判断标准是：用户是否知道自己下一步该点什么、结果是否看得懂、是否能继续进入后续链路。

### 3.4 AI 医生问诊功能

#### 3.4.1 测试用例与结果分析

AI 医生问诊是连接“症状输入”和“结构化解释”的桥梁。功能测试覆盖：

1. 医生页文本输入和发送是否稳定。
2. 连续语音模式下是否能避免把机器人播报再次识别成用户输入。
3. 数字人或桌面机器人提示文案是否与当前页面语义一致。
4. 本地预热文案是否能在进入页面时快速播报，避免明显延迟。
5. 点击机器人后是否优先走云端 AI 生成回复，并回退到本地兜底。

当前项目在这一模块上已经做过数轮体验优化，包括：

- 桌面机器人取消自由录音模式，改为“页面说明 + TTS 播报 + 动作映射”。
- 进入页面时优先使用本地预热文案和本地缓存音频。
- 点击机器人时优先走云端 `/api/avatar/narration`，不可用时回退本地。
- 医生页语音改成半双工连续状态机，避免 TTS 语音被重新识别。

从功能测试角度看，这些优化显著减少了“机器人说的话和页面不搭”“切页后要等一会才开始说”“自己把自己的话识别进去”等典型问题。说明该模块已经不再只是展示型组件，而是开始承担页面说明和轻引导作用。

#### 3.4.2 测试结果综合分析及建议

本模块的当前表现可以总结为：体验框架已建立，语义质量和真机语音链路仍需继续打磨。建议后续重点做：

1. 不同页面进入与点击的文案一致性抽查。
2. 语音播报开关关闭后，是否仍保留文字气泡但不发声。
3. 讯飞不同能力缺失时的错误提示是否明确，例如“未配置”“网络异常”“账号不可用”等。
4. 数字人相关 AIUI、avatar id 和场景参数在新环境迁移时的回归测试。

综合来看，医生问诊功能已经具备较好的展示价值，但依然需要对真机语音链路保持敏感，因为它比普通文本页面更依赖真实设备和网络条件。

#### 3.4.3 测试经验总结

问诊功能的经验在于：对于带语音和数字人的功能，体验问题往往并不来自核心算法，而来自时机控制、播报/录音状态切换和错误提示质量。因此，该模块的功能测试不能只测“能否说话”，必须测“何时说、说什么、说错时是否能快速恢复”。

### 3.5 医检报告分析功能

#### 3.5.1 测试用例与结果分析

医检报告分析功能在 UI 层面承担了从图片输入、OCR 识别、文本整理、可读摘要到下一步建议展示的完整流程。功能测试覆盖以下核心用例：

1. 上传或导入报告图片后是否能生成 OCR 文本。
2. 有结构化结果时，页面是否优先显示 AI 可读摘要。
3. 云端不可用或未登录时，是否仍能显示本地可读兜底。
4. 用户编辑 OCR 文本后重新解析是否生效。
5. 结果页是否明确提示“仅供辅助理解，不替代医生结论”。

根据当前项目修复情况，该模块已解决“AI 整理后的可读报告不显示内容”的问题。现在的逻辑是：

- 云端有可读报告时优先展示云端结果。
- 云端结果缺失时使用本地 `MedicalReportDraftFormatter` 生成兜底文本。
- 即使结构化指标不足，也会基于 OCR 原文生成摘要和下一步建议，而不是直接空白。

这一变化意味着报告页面已经从“依赖云端成功才有内容”提升为“任何情况下都尽量给用户一个最低可用解释”。从测试角度，这是非常重要的稳定性改进。

#### 3.5.2 测试结果综合分析及建议

当前医检报告分析模块总体表现良好，但依然建议重点补测：

1. 图片过暗、倾斜、模糊条件下的 OCR 容错。
2. PDF、多页报告和长表格报告的兼容性。
3. 登录前与登录后报告解释差异是否符合预期。
4. 用户对“建议”和“风险”文字的理解是否足够明确。

从功能测试文档角度，建议把这一模块表述为“具备 OCR、结构化理解和可读报告三层能力”，而不是只说“支持拍照识别”。这样更能体现项目价值。

#### 3.5.3 测试经验总结

该模块最核心的经验是：对于用户最终看到的是“文字解释”的页面，功能测试必须把“空白页风险”作为一级缺陷处理。因为哪怕后台有对象、有字段、有日志，只要用户眼前没有内容，这个功能就等于失败。

### 3.6 干预执行与复盘功能

#### 3.6.1 测试用例与结果分析

干预执行与复盘功能是长庚环区别于单纯监测应用的关键环节。功能测试重点包括：

1. 从今日页、症状自查或报告页是否能进入干预中心。
2. 干预中心是否能展示建议入口和执行入口。
3. 呼吸训练页面是否能进入训练流程。
4. 执行后是否能留下完成状态或体感反馈。
5. 复盘页是否能展示完成率、体感变化和下一步建议。

当前实现已具备：

- 干预中心聚合页。
- 呼吸训练页面。
- 执行复盘页面。
- 从症状自查和旧康复页向干预流程衔接的路径。

尽管这一模块在自动化测试层面的覆盖还不足，但从功能测试角度，它已经形成了可描述、可演示、可继续扩展的闭环。

#### 3.6.2 测试结果综合分析及建议

建议未来重点完善：

1. 干预任务状态流转的自动化验证。
2. 执行记录与趋势统计的一致性验证。
3. 呼吸训练中断、切后台、来电等场景的恢复策略。
4. 症状自查结果与干预推荐之间的映射解释。

该模块最适合在测试文档中强调的不是“有多少种训练动作”，而是“建议可以被执行、执行可以被记录、记录可以回写反馈”，这也是项目闭环价值的直接体现。

#### 3.6.3 测试经验总结

执行模块的经验是：很多项目停在“给建议”，但真正难的是“让用户开始做、做完能记下、下一次建议会更稳”。因此，这一模块的测试在项目评分中非常关键，它代表系统是否真正完成了闭环。

## 4 系统测试

系统测试关注跨模块、跨层次、跨设备和跨网络场景的整体质量。对于长庚环这样的端云协同项目而言，系统测试的作用是把“模块分别看起来没问题”提升为“整个系统协同起来也能稳定工作”。

### 4.1 模型性能测试

#### 4.1.1 测试用例与结果分析

模型性能测试包括端侧模型和云端 AI 整理两类对象。端侧部分主要关注本地异常检测与信号分析是否能在移动设备上快速运行，云端部分主要关注 OCR 整理、问诊生成、讲解文案生成等是否能在合理时间内返回。

测试重点包括：

1. 应用启动后关键页面是否能在用户可接受时间内显示核心内容。
2. 本地信号分析是否会明显阻塞 UI。
3. OCR 后的可读报告是否能在合理时间生成，且失败时能快速走本地兜底。
4. 进入页面的机器人说明是否能先用本地预热文案响应，点击后再由云端生成更细内容。

从当前项目实际优化看，桌面机器人已改成“进入页面走本地预热，点击再走云端 AI”，这正是性能优化的一个系统级例子。它说明本项目没有单纯追求“所有东西都实时走云端”，而是在体验与能力之间做了合理分层。

#### 4.1.2 测试结果综合分析及建议

目前模型性能测试可得出的结论是：架构已经形成“本地即时反馈 + 云端增强”的思路，性能风险大幅降低；但若要形成更正式的性能结论，仍需后续补充量化指标，例如：

- 关键页面首屏可见时间。
- 本地信号分析单次耗时。
- OCR 整理平均耗时。
- 云端问诊或讲解平均响应时间。

当前阶段建议在测试文档中写成“已形成明确性能策略与实际优化结果，后续将补充更细粒度 benchmark 数据”，这样既真实，也不失技术深度。

#### 4.1.3 测试经验总结

性能测试的经验是，不应只追求快，而应追求“对用户体感来说够快”。长庚环当前的做法是把即时反馈放在本地，把更强理解放在云端，这一思路本身就是系统设计与测试协同的结果。

### 4.2 端云协同稳定性测试

#### 4.2.1 测试用例与结果分析

端云协同稳定性是当前系统测试的核心。长庚环既依赖 Android 本地能力，也依赖 `cloud.changgengring.cyou` 下的认证、问诊、报告、同步等接口，因此任何一侧版本或配置不一致都可能造成严重问题。

当前阶段重点验证的端云协同用例包括：

1. Android `BuildConfig.API_BASE_URL` 与生产域名是否一致。
2. 关键认证接口是否存在并返回合理状态码。
3. 忘记密码和重发确认邮件是否可用。
4. 登录页带 recovery 参数时是否能自动跳转重置密码页。
5. 症状自查、问诊、报告分析等功能在登录态与未登录态下的行为是否明确。

从已完成验证看，当前生产域名已统一，关键接口状态如下：

- `login` 路由可访问，并对错误凭据返回明确错误。
- `password-reset` 路由已存在且可返回成功结果。
- `resend-confirmation` 路由已存在且可返回成功结果。
- `/reset-password` 与 `/auth/confirm` 页面均可打开。

这些结果说明，当前端云协同最危险的“客户端调用旧云端、生产接口缺路由、回跳地址失效”等问题已经被解决。

#### 4.2.2 测试结果综合分析及建议

当前端云协同测试最大的改进点在于：域名、部署和 Android 构建配置已经统一，不再存在“本地代码改好了，但手机还打旧服务”的结构性错配。

后续建议重点加强：

1. 接口错误码与客户端提示映射验证。
2. 未登录用户访问需要鉴权功能时的降级路径。
3. 云端部署更新后的回归脚本，例如登录、重置密码、报告分析和讲解接口的 smoke test。

#### 4.2.3 测试经验总结

端云协同最怕的不是功能没写，而是版本不一致。长庚环这轮测试经验表明，统一域名、统一构建配置、统一部署口径，是移动端与云端协同稳定的前提。

### 4.3 BLE 与设备链路稳定性测试

#### 4.3.1 测试用例与结果分析

BLE 稳定性测试重点覆盖：

1. 重复扫描时是否会出现卡死或 UI 提示与状态不一致。
2. 多次连接、断开、重连后是否还能恢复正常。
3. 权限未开启时提示是否明确。
4. 后台服务运行与前台页面状态之间是否一致。

由于本轮更多是工程收口与软件链路验证，BLE 真机长时间稳定性测试仍需结合戒指样机做补充。但就代码结构和页面链路来看，BLE 已经作为真实模块进入当前构建体系，不再是演示型占位。

#### 4.3.2 测试结果综合分析及建议

BLE 系统测试建议在正式验收前至少补充以下内容：

- 30 分钟以上连续连接保持测试。
- 断连后 3 次以内自动或手动恢复验证。
- 数据回调频率与数据库写入频率匹配验证。
- 多设备干扰场景下的搜索列表准确性验证。

#### 4.3.3 测试经验总结

BLE 测试的核心经验是：连接成功只是起点，不是终点。真正要测的是“连接后的持续可用性”和“异常后的可恢复性”。

### 4.4 安全与隐私相关测试

#### 4.4.1 测试用例与结果分析

长庚环虽然是比赛项目，但其处理的内容包括账户资料、健康指标、问诊文本和医检报告，因此安全与隐私不能只做形式化处理。本轮测试关注：

1. 账户未登录时是否要求鉴权。
2. 登录后会话是否能正确保存与清除。
3. 个人中心中的通知、隐私、关于等页面是否真实存在，而非占位按钮。
4. 数据与隐私页面是否具备退出登录、清理本地会话与偏好数据的能力。
5. 医疗类页面是否保留“仅供辅助理解/辅助判断”的安全提示。

根据当前实现，个人中心已补齐真实页面，演示模式已被去除，通知设置与机器人语音播报开关均可持久化，数据与隐私页提供了会话和本地偏好清理能力。这说明系统已经开始具备基础的隐私治理意识，而不是只展示“隐私”字样。

#### 4.4.2 测试结果综合分析及建议

当前系统在安全与隐私方面的优点包括：

- 鉴权接口与会话链路已统一。
- 个人中心中的账号、通知、隐私、关于不再是空壳。
- 用户可以关闭机器人自动语音播报。
- 报告和症状类页面表达总体趋于保守。

后续建议补充：

1. 本地数据库导出、清理与残留验证。
2. 未授权状态下访问敏感页面的行为验证。
3. 日志中是否出现敏感字段的检查。
4. 更多关于医疗免责提示的一致性抽查。

#### 4.4.3 测试经验总结

安全与隐私测试在比赛项目中经常被弱化，但对于长庚环这类健康场景产品，它恰恰是专业度的重要体现。测试中最关键的不是写很多安全术语，而是把账号、权限、日志、数据清理、风险提示这些真实可见的点落下来。

## 5 测试结果总体结论

综合单元测试、功能测试和系统测试，当前长庚环项目可以得出以下阶段性结论。

第一，Android 端已经实现单一入口运行，当前所有主要功能都在同一个软件内可访问。这意味着从产品使用角度看，用户面对的是一个完整的健康管理 App，而不是多个演示入口拼接的原型。

第二，构建与基础自动化门禁已基本建立。`assembleDebug`、`testDebugUnitTest`、`lint` 以及云端 `build`、`lint` 的链路都已打通，这对后续版本迭代和答辩展示具有重要价值。

第三，云端认证链路和域名已完成统一。此前最影响体验的注册、忘记密码、重置回跳等问题已经完成结构性修复，说明项目在从“功能原型”走向“可使用系统”的道路上迈出了关键一步。

第四，症状自查、AI 问诊、医检报告分析、干预执行等核心功能已形成相互衔接的闭环，不再是彼此割裂的页面集合。这一点非常符合长庚环“监测 -> 分析 -> 建议 -> 执行 -> 反馈”的主叙事。

第五，项目仍存在继续完善空间，主要集中在三类：

1. BLE 真机稳定性和长期连接测试仍需更多样机验证。
2. 讯飞与数字人相关能力受配置和账号后台影响较大，需要继续做专项联调。
3. 自动化测试覆盖率还可以继续提升，尤其是干预执行状态流和更多脏输入场景。

总体而言，长庚环当前已经不是“只能展示页面”的项目，而是具备真实运行能力、具备账号体系、具备端云联动、具备问诊与报告解释、具备执行闭环的综合系统。站在测试角度，我们可以将其当前质量等级定义为：

- 基础可运行性：已达到可展示、可安装、可联调水平。
- 核心功能完整性：已达到主流程完整、关键功能齐全水平。
- 工程可维护性：已达到具备构建门禁和部分自动化资产的水平。
- 软硬件协同成熟度：已具备真实接入能力，但真机长期稳定性和部分专项联调仍需继续加强。

## 7 测试经验总结

长庚环项目本轮测试工作的最大经验，并不是“发现了多少 bug”，而是明确了什么样的项目才算真正可用。对于一个集成智能戒指、Android App、云端服务和 AI 助手的综合系统来说，所谓可用，至少意味着以下几点：

1. 用户能进来，不被账号问题挡住。
2. 设备能接上，不是只有空页面。
3. 分析结果能看懂，不是把技术字段甩给用户。
4. 用户知道下一步做什么，不只是被动看报告。
5. 出现错误时，系统能给出明确、保守、可恢复的提示。

本项目在测试中逐步形成了几个非常明确的方法论。

第一，必须把“项目主线”转化成“测试主线”。长庚环的主线是监测、分析、建议、执行、反馈，因此测试就要沿着这五步设计，而不是按页面孤立地测。

第二，必须区分“配置问题”“代码问题”“服务问题”和“体验问题”。例如忘记密码失败可能是路由不存在，也可能是回跳地址错误；讯飞未配置可能是本地配置项未进入 BuildConfig，也可能是账号后台不完整。只有把问题归类清楚，测试文档才有价值。

第三，移动端健康项目必须重视兜底。包括本地 OCR 兜底、本地问诊提示兜底、本地预热文案兜底、离线保守结果兜底。因为这类项目天然存在网络波动、第三方服务波动和真机环境差异，测试中最关键的是验证“失败后还能不能给用户一个可理解结果”。

第四，工程质量本身就是评分项。单元测试、lint、模块归类、统一入口、错误提示、乱码清理、真实页面补齐，这些看似不直接提升演示炫酷度，但它们共同决定了项目在评委和用户眼中是“拼出来的原型”，还是“认真做过工程化收口的系统”。

综上，长庚环当前阶段的测试工作已经证明：项目具备较完整的软硬件协同能力、较清晰的产品闭环和较扎实的工程化基础。后续如能继续补齐真机 BLE 长时稳定性、更多 OCR/问诊样本和更细的自动化覆盖，整体质量将进一步提升，并更适合进入展示、答辩和后续试点阶段。
