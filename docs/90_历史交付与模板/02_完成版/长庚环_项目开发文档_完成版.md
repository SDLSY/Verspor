版本号：V1.0  
项目名称：长庚环  
英文名称：VesperO

## 项目概述

长庚环项目面向居家健康管理、睡眠恢复评估、症状自查、医检报告辅助理解和康复辅助场景，目标不是单纯构建一个“展示监测结果”的健康应用，而是打造一个围绕“监测 -> 分析 -> 建议 -> 执行 -> 反馈”闭环展开的一体化系统。项目由智能戒指、Android 单入口 App、云端 AI 与数据服务共同组成。智能戒指负责连续感知与数据采集，Android 端负责可视化、交互、症状输入、问诊和执行页面承接，云端负责鉴权、同步、深度分析、AI 增强生成和管理接口。三者共同构成完整的端云协同健康辅助系统。

本项目的开发思路并非从单点功能出发，而是从用户在真实生活中的决策困难出发。普通用户在睡眠变差、恢复下降、胸闷头痛、报告看不懂等问题出现时，往往并不缺少数据，而是缺少对数据的解释、对风险的分级以及对下一步行动的指引。因此，长庚环在开发上强调从“监测工具”向“行动辅助系统”转变，把设备、数据、问诊、报告、干预和复盘统一到一个软件产品中，降低用户在多个工具之间切换的成本。

### 项目背景

随着可穿戴设备、互联网医疗平台和大模型技术的持续发展，用户已经能够较为方便地获取心率、睡眠、压力、血氧、活动量、医检报告和在线问诊等信息。但这些能力在现实中通常被分散在不同系统中。可穿戴设备擅长记录指标，却不一定能告诉用户今天应该做什么；互联网问诊平台擅长结构化追问，却没有持续性的生理数据背景；OCR 工具擅长识别报告文字，却不等于真正让用户理解报告，也不一定能够顺利进入后续干预流程。

长庚环项目正是在这样的背景下提出。项目认为，未来的健康产品不能仅靠“更多图表”来体现价值，而需要建立从连续感知到实际行动的完整链路。尤其在居家场景下，用户更需要低门槛、可解释、可执行、保守表达且可持续使用的健康辅助系统。项目因此选择以智能戒指为连续感知入口，以 Android 单入口 App 为交互与执行中心，以云端 AI 为解释与增强引擎，形成适合日常使用的闭环结构。

### 项目定位

长庚环定位为面向居家场景的智能戒指与健康辅助系统，强调“辅助判断”“康复辅助”“健康管理”和“可执行建议”，而非临床诊断工具。产品的主要价值体现在帮助用户更轻松地理解自身状态、降低第一步输入门槛、获得风险分级和下一步建议，并把建议真正转化为可执行的动作与后续反馈。

在技术定位上，项目采用“端侧实时 + 云端增强 + 本地兜底”的混合模式。端侧负责设备连接、基础算法、离线回退和即时交互；云端负责账号体系、数据同步、报告理解、AI 医生推理和文本/语音生成；本地兜底则保障在弱网、第三方接口波动或云端不可用时，仍能维持最低可用体验。

#### 应用场景

长庚环当前面向的核心场景包括：一是睡眠与恢复管理场景，用户通过戒指记录夜间指标，在早晨查看晨报、恢复分和建议；二是症状自查与辅助问诊场景，用户通过 2.5D 身体定位图、症状 chips 和结构化输入完成轻量级自查，再进入 AI 医生问诊；三是医检报告分析场景，用户拍照上传或输入报告内容，系统通过 OCR 和结构化解析生成可读化结果，并给出下一步建议；四是康复与放松场景，用户根据系统给出的干预计划、呼吸训练、旧康复辅助页和执行复盘页面进行实际行动与反馈记录；五是设备连接与同步场景，用户在设备页管理戒指连接、同步状态、工作参数与采集服务。

#### 目标人群

项目的第一阶段目标人群主要包括：长期存在睡眠质量波动或恢复不佳、但不一定有明确临床疾病诊断的青年与职场人群；需要进行轻量化自我管理、放松训练、压力调节或康复辅助的人群；关注体征变化、愿意使用可穿戴设备的数字健康用户；希望通过 AI 辅助问诊和医检报告解读降低理解门槛的普通用户；以及可作为校园、社区、企业福利试点的群体用户。

在更长期的定位中，长庚环也可以服务于高校健康项目、企业健康管理、基层康复随访和社区健康辅助服务等 B2B2C 场景。此类场景强调产品不仅要有单用户体验，也要具备可统一部署、可统一讲解、可统一收集数据、可统一留痕与可运维的能力。

### 项目方案

长庚环的整体方案由四层构成。第一层是硬件感知层，以智能戒指为主要设备，通过 BLE 与 Android 端连接，采集睡眠、PPG、心率、血氧、体温映射和设备工作参数等数据。第二层是 Android 交互层，以 `:app-shell` 作为唯一运行入口，承载首页晨报、医生问诊、趋势页、设备页、我的页、症状自查、医检报告分析、干预中心、呼吸训练、执行复盘等功能。第三层是本地能力层，包含 Room 数据库、BLE 协议管理、本地异常检测、OCR 草案整理、本地文案兜底、语音配置与页面状态管理等。第四层是云端服务层，以 `cloud-next` 为核心，提供用户鉴权、数据同步、AI 路由、问诊处理、报告理解、处方建议、后台管理和模型运行时接口。

项目方案的关键不是把这些能力简单拼接，而是使它们在一个统一入口中以自然流程衔接：晨报可以通向症状自查与干预中心；症状自查可以通向医生问诊和报告分析；报告分析可以继续通向问诊与康复辅助；设备页的数据最终影响趋势与建议；个人中心统一承接账号、通知、隐私与设置。系统因此从“多工具集合”转化为“单入口闭环系统”。

### 项目目标

项目的第一目标是实现一个真正可运行、可演示、可测试的单入口 Android 应用，保证用户能够在同一个 App 中完成健康数据查看、设备连接、症状自查、AI 问诊、医检报告理解和干预执行，而不需要依赖多个独立入口或切换到其他工具。

项目的第二目标是完成智能戒指与 Android 端的有效联动，包括 BLE 连接、设备发现、数据解析、工作参数读取、后台采集与本地数据库入库。没有这部分能力，项目将失去与普通健康 App 的核心差异。

项目的第三目标是建立基于 AI 的解释能力，而不是仅仅展示原始数据或 JSON 结果。无论是医生问诊、OCR 报告还是桌面机器人讲解，输出都应当以用户可理解、保守表达、可执行为原则。

项目的第四目标是保证系统具备工程上的闭环，包括单入口架构、基础模块划分、构建可通过、主要接口可联通、账号体系可用、部署路径明确以及基本的测试链路。

### 项目价值

长庚环的价值首先体现在用户价值层面。对于用户而言，它降低了从“看到问题”到“采取行动”之间的距离。用户无需依赖专业医学表达就可以描述不适，也无需自行理解复杂报告术语，即可获得风险提示和下一步建议。这种价值是当前很多单点工具不具备的。

其次，项目具备明显的系统价值。它不是单纯的可穿戴终端，也不是单纯的问诊 App，而是把感知、分析、解释、执行和反馈组织成同一条链路。这种统一使得数据不再孤立存在，建议也不再停留在页面文字层面。

最后，项目还具备工程与竞赛价值。系统已经完成 Android 单入口收口、账号链路打通、云端统一部署、自定义域名接入、机器人讲解机制、本地兜底策略和多模块工程收敛。这种从产品设想到真实工程的落地能力，直接增强了项目在代码质量、技术路线合理性和可交付性方面的说服力。

## 开发计划

### 最终呈现形式

项目最终呈现为“智能戒指 + Android App + 云端服务后台”的组合产品。用户侧主要使用 Android App 进行健康信息查看、设备连接、症状定位、AI 问诊、报告分析和执行反馈；管理侧通过云端后台查看患者概览、时间线、睡眠数据、问诊摘要、医检结果与干预记录；算法与模型侧通过 `ml/` 和 `contracts/` 提供训练、导出、推理与契约统一支持。对外演示时，产品呈现为一个完整的软件与硬件协同系统，而不是多个孤立模块的拼接。

### 主要功能描述

当前系统主要功能包括以下几类。第一类是首页与晨报功能，负责展示恢复评分、近期趋势、日内建议以及通向症状自查、干预中心和设备页的入口。第二类是设备功能，包括 BLE 扫描、连接、断开、参数读取、数据采集和本地状态提示。第三类是医生问诊功能，包括文本消息、语音输入、问诊引导、数字人活动页与问诊摘要。第四类是放松与症状相关功能，包括 2.5D 症状自查、旧康复中心、呼吸训练、医检报告分析、复盘页和干预执行页。第五类是趋势功能，负责展示睡眠与恢复相关时间序列变化。第六类是个人中心功能，包含登录注册、重发确认邮件、忘记密码、个人资料、通知设置、隐私设置和关于项目页。

除了页面层功能，项目还包含后台支持功能，例如账号体系、云端同步、报告理解、问诊 AI 路由、建议生成、模型运行时接口、内部管理接口和后台患者概览等。这些能力虽然不直接体现在 Android 导航栏中，但对整体系统可运行性至关重要。

### 运行环境

Android 端当前运行入口为 `:app-shell`。项目采用 AGP `8.7.3`、Kotlin `1.9.24`、Java 11，`compileSdk 35`、`targetSdk 35`、`minSdk 24`。应用包名为 `com.example.newstart`，当前版本号为 `1.0`。从设备角度看，项目要求 Android 7.0 及以上系统，支持 BLE、网络、录音、摄像头和通知权限，以满足设备连接、语音、OCR 和云端通信需要。

云端采用 Next.js Route Handlers 与 Supabase 组合实现，承担用户登录注册、确认邮件、密码重置、问诊接口、报告理解接口、同步接口与后台页面。当前项目已经统一到自定义域名部署，便于 Android 端、邮件回跳和后台访问统一。

开发环境上，项目默认在 Windows 或 macOS 环境下通过 Gradle 构建 Android 端，通过 Node.js 环境运行 `cloud-next`，通过 Python 环境运行 `ml/` 中的训练与推理脚本。文档、测试与构建命令均已在仓库根目录 `README.md` 和 `docs/00_入口与维护/DOC_MAINTENANCE.md` 中统一。

### 验收标准

项目开发验收标准包括功能性、稳定性和工程性三个维度。功能性方面，Android 单入口必须能够进入今日、医生、趋势、设备、我的、症状自查、医检报告、呼吸训练、旧康复页和干预执行页；账号系统必须支持注册、登录、重发确认邮件和密码重置；报告分析必须能产出用户可读内容；机器人讲解需要支持本地预热和点击后云端增强。稳定性方面，弱网或云端不可用时系统仍需提供基础兜底体验，BLE 扫描和连接需要有明确状态提示，数据库需要支持核心数据存储和读取。工程性方面，`gradlew.bat :app-shell:assembleDebug`、`:app-shell:testDebugUnitTest` 和 `:app-shell:lint` 必须可运行，云端 `npm run build` 必须通过，核心模块边界和单入口链路必须清晰。

从用户体验层面，验收还要求：症状自查页输入门槛合理，报告理解页不直接暴露结构化原文作为唯一输出，设备页扫描提示和通用加载提示不出现乱码，忘记密码链接应正确落到重置密码页面，邮箱链路应能够完成确认和重置流程。

### 关键问题

项目开发过程中需要持续关注几个关键问题。第一，硬件接入稳定性问题。由于 BLE 链路受设备状态、权限状态和操作系统版本影响较大，必须通过状态机和本地提示避免用户在扫描、连接和同步过程中迷失。第二，AI 能力的可用性问题。项目接入了讯飞语音、OCR、Spark 以及云端 AI 路由，如果过于依赖单一服务，会导致体验不稳定，因此必须始终保留本地兜底和保守输出机制。第三，健康产品的边界问题。系统不能输出确诊式语言，必须通过风险分级与建议分层实现安全表达。第四，账号与邮件链路问题。注册确认、重发邮件和密码重置必须与 Supabase、Resend、自定义域名和 Android 客户端保持一致。第五，工程收敛问题。项目已经完成单入口整合，但仍需维持 feature 模块、core 模块与宿主层之间的清晰边界，避免再次回到混乱状态。

### 进度安排

从开发阶段划分看，项目经历了需求聚焦、单点原型、链路整合、单入口收口和质量补全几个阶段。早期阶段以完成健康监测与设备连接为重点，主要解决智能戒指数据获取和 Android 页面原型问题。中期阶段将医生问诊、报告分析和康复辅助接入系统，开始建立“监测 -> 建议”的基本链路。后期阶段则重点完成 Android 单入口统一、邮箱链路修复、自定义域名统一、桌面机器人重构、症状自查重设计、旧康复页保留和我的页真实功能落地。当前阶段的重点转向代码整理、文档整理、测试闭环和可交付性增强。

### 开发预算

项目开发预算以学生团队现实条件为前提，采用控制成本、优先落地主线功能的原则。预算主要由四部分组成。第一部分是硬件预算，包括智能戒指样机、蓝牙调试设备、安卓测试机和必要的数据线配件。第二部分是云与服务预算，包括域名、云端部署、邮件发送服务、第三方 AI 服务调用和必要的对象存储或数据库配额。第三部分是软件开发预算，包括 Android 与云端开发环境、调试设备、证书与发布相关费用。第四部分是测试与展示预算，包括文档整理、打印、截图、演示环境和路演准备。

从控制策略上看，项目优先把预算投入在“能否形成完整闭环”的关键节点上，例如硬件接入、账号链路、云端部署和核心交互页，而非在非关键视觉细节或过度扩展功能上过早投入。这样的预算策略符合竞赛项目阶段目标，也有利于后续在校园、社区或企业场景中进行低成本试点。

## 可行性分析

### 技术可行性分析

项目的技术可行性首先体现在所使用技术栈的成熟度。Android 端采用 Kotlin、Room、Retrofit、OkHttp、Nordic BLE、MPAndroidChart、Lottie、Shimmer 和 Hilt 等成熟技术；云端采用 Next.js 与 Supabase 构建鉴权和 API 服务；本地与云侧 AI 能力分层明确，既使用讯飞提供的语音、OCR、Spark 等能力，也保留本地解析和本地异常检测能力。上述组件都属于当前移动开发与云端服务中较为成熟的方案，具备稳定文档、丰富示例与良好社区支持。

其次，项目整体架构符合问题本身的技术需求。长庚环需要同时处理设备实时接入、页面即时反馈、AI 解释、账号同步和后台管理，因此纯端侧方案无法满足深度理解与管理需求，纯云端方案又无法满足低时延和离线兜底需求。端云协同架构既符合场景特点，也已在当前工程中得到验证。

再次，项目具备足够的工程基础。Android 单入口 `:app-shell` 已经完成收口，不再依赖旧 `app/` 的 `sourceSets`；`core-ble`、`core-db`、`core-network`、`core-ml`、`core-data` 与 `feature-*` 模块均已建立；云端生产域名与 Android API 基地址已经统一；注册、确认邮件、密码重置、重置密码回跳、机器人文案策略和本地预热机制都完成过实际修复。这表明项目不是停留在概念层，而是具备真实可运行的技术基础。

### 资源可行性分析

从团队资源来看，项目所需的资源结构与大学生团队的现实能力相匹配。智能戒指硬件侧虽然需要一定调试成本，但已经具备基本的数据采集和连接能力；Android 前端、云端 API、模型脚本和文档材料均在同一仓库中统一管理，利于多人协作。项目不依赖自研大模型，而是采用第三方能力 + 本地规则 + 部分端侧算法的组合方式，降低了资源门槛。

在时间资源方面，当前项目已经完成了关键链路打通和单入口整合，这意味着后续主要工作可以聚焦在质量补全、文档完善、测试补强和展示优化，而不再是从零开始开发核心能力。从软件竞赛和课程项目角度看，这种资源分配方式是现实且高效的。

### 市场可行性分析

从市场角度看，长庚环位于数字健康、可穿戴设备、互联网医疗辅助与康复辅助的交叉点。这类方向具有明确需求基础：一方面，可穿戴设备已经培养出用户对连续监测的接受度；另一方面，用户仍然对“报告看不懂”“数据看不懂”“不知道下一步怎么做”存在普遍痛点。因此，能够把监测、解释、问诊和行动建议整合起来的产品，具备较高的实际价值。

在落地路径上，长庚环并不需要直接面对大规模 C 端竞争。相反，它可以先从校园健康、社区康复辅助、企业福利和试点项目切入，以 B2B2C 方式建立信任入口。在这一模式下，产品既可以提供单用户体验，也具备被机构场景采用的可能性。当前项目的系统结构、后台管理能力和多端协同架构，也为这种落地模式提供了基础。

## 需求分析

### 数据需求

#### 静态数据

静态数据主要包括用户的基础信息和系统配置数据。用户基础信息包括账号 ID、邮箱、用户名、年龄、性别、基础睡眠偏好、通知偏好和本地缓存设置。设备基础信息包括设备 ID、设备名称、设备类型、蓝牙地址、连接状态与最后同步时间。系统静态配置数据包括云端基础地址、第三方能力开关、语音播报开关、通知开关、机器人提示策略、本地缓存目录和离线规则参数等。

#### 动态数据

动态数据是长庚环的核心数据部分，包括睡眠记录、PPG 样本、心率与血氧变化、恢复评分、医生问诊消息、医生问诊会话、评估会话与答案、医检报告与指标、干预任务、执行记录、放松训练记录、症状自查结果、周期性趋势结果和账户同步状态等。动态数据既需要在本地用于即时展示和离线使用，也需要在合适时机同步到云端用于统一画像和历史分析。

#### 数据词典

项目在数据语义上强调统一。用户相关对象应能够通过账号唯一标识关联到医生问诊、报告分析、睡眠数据和干预记录。医检指标需要包含项目名称、数值、单位、参考范围和异常状态。问诊消息需要包含消息角色、时间戳、内容、摘要和是否来自 AI。干预任务需要包含协议码、标题、时长、建议级别、执行状态和反馈结果。项目在 `contracts/` 中统一维护跨端契约，并遵循 `{ code, message, data, traceId }` 形式的接口响应结构。

#### 数据采集

数据采集来源主要有三类。第一类是设备侧采集，即通过智能戒指经 BLE 获取的动态体征和设备状态。第二类是用户主动输入，包括症状自查输入、问诊文本、语音、报告拍照上传和个人资料编辑。第三类是系统生成数据，包括 AI 解释结果、风险分级、可读化报告、机器人讲解稿、干预任务、执行评分和趋势汇总结果。不同采集源的数据在本地需要标准化存储，在云端需要按用户上下文进行聚合和增强。

### 功能需求

#### 核心功能模块

- 首页与晨报模块：展示恢复评分、风险提示、近期趋势与关键入口，承接从监测结果进入行动建议的第一步。
- 设备连接模块：完成戒指扫描、连接、断开、工作参数读取、同步状态查看与后台采集控制。
- 医生问诊模块：完成结构化对话、语音输入、问诊摘要和数字人辅助展示。
- 症状自查模块：通过 2.5D 身体定位图、前后侧切换、症状 chips 和风险提示帮助用户低门槛输入。
- 医检报告分析模块：通过 OCR、结构化解析和可读报告整理帮助用户理解医检结果，并引导下一步行动。
- 干预执行模块：承接处方、呼吸训练、康复动作、旧放松页和执行反馈记录。
- 趋势分析模块：展示睡眠、恢复与执行效果的时间变化。
- 个人中心模块：承接登录注册、重发确认邮件、忘记密码、资料编辑、通知设置、隐私设置和关于页。

### 性能需求

#### 时间特性

首页、设备页和我的页等常用页面应在可接受时间内完成首屏加载，且在本地已有数据时能够优先显示缓存结果。BLE 扫描与连接状态变化应及时反馈给用户，不允许长时间无提示。症状自查和报告分析页在本地输入完成后应当尽快给出初步结果，即使云端增强尚未返回。语音和机器人播报应支持本地预热，保证进入页面时不会完全无响应。云端接口在失败时必须提供明确错误提示和本地降级策略，而不是让页面停留在空白或无状态中。

#### 适应性

系统应适应不同 Android 设备分辨率、平板与手机形态、不同蓝牙权限模型、弱网和离线场景。症状自查页需要针对横竖屏提供不同布局方案；我的页与设置页需要在隐藏桌面机器人时仍保持视觉完整；报告分析和问诊页需要在未登录、已登录、云端可用和云端不可用多种状态下给出合理反馈。

### 界面需求

界面层需要满足“可信、克制、可解释”的健康产品风格。首页应以恢复与行动建议为中心，而非单纯图表堆叠；症状自查页需要突出 2.5D 定位图、快捷症状与结果卡；医生页要保证对话清晰、输入方式明确、机器人说明不抢主链路；报告分析页需要让“可读报告”优先于原始 OCR 文本；个人中心应完整呈现账户、通知、隐私与项目说明。整个系统需要保持视觉语言统一，避免功能割裂导致的“像多个 App 拼接”的印象。

### 接口需求

#### 硬件接口

硬件接口主要由智能戒指与 Android 端之间的 BLE 协议组成。系统需要支持设备扫描、连接、断开、参数读取、帧解析、命令发送和采集控制。当前项目中该部分由 `BleManager`、`BleConnectionManager`、`CleveringCommandSender`、`Hi90BCommandBuilder`、`Hi90BFrameParser` 等实现共同支撑。接口需求的重点在于稳定性、可重连性与权限适配。

#### 软件接口

软件接口包括 Android 端与云端之间的 API 通信，以及 Android 内部模块之间的接口调用。云端接口主要覆盖登录、注册、重发确认邮件、忘记密码、同步、睡眠上传、报告理解、问诊处理、干预任务、处方生成、用户资料和机器人讲解稿生成等。Android 内部接口则通过 repository、service、ViewModel 和 Room DAO 完成数据流转。系统需要保证这些接口具备清晰的输入输出结构、错误提示和可追踪性。

### 其他需求

项目还需要满足若干非功能性需求。首先是安全与隐私需求，系统不得将敏感数据直接打印在日志中，报告与问诊结果需要在本地和云端都遵循最小必要原则。其次是可维护性需求，模块边界应清晰，重要能力放入 `core-*` 与 `feature-*` 中，文档与代码事实应保持一致。再次是可演示性需求，系统需要在无复杂部署条件下完成现场展示，包括邮箱链路、机器人播报、设备页、症状自查和报告分析等。最后是可扩展性需求，系统需要支持后续接入更多模型、更多设备或更多机构场景，而不是写死为一次性演示原型。

## 概要设计

### 处理流程

长庚环的核心处理流程可以概括为：设备与用户输入作为上游输入源，本地和云端共同负责解析和组织，系统输出可理解的结果与下一步建议，用户执行干预动作后再把反馈写回系统。具体而言，睡眠与体征数据通过戒指采集进入 Android 本地数据库，再经趋势与恢复算法形成晨报与日内建议；症状自查通过 2.5D 输入与规则引擎生成风险分级，再继续导向医生问诊或康复辅助；报告拍照经 OCR 与结构化解析后形成可读报告，再与问诊和干预模块联动；执行任务完成后，执行记录和体感反馈回流至本地和云端，为后续画像和建议提供依据。

### 总体结构设计

项目总体结构采用“宿主层 + 特性层 + 核心层 + 云端层 + 契约层”的设计。宿主层为 `:app-shell`，负责 Application、MainActivity、Manifest、全局资源与顶层装配。特性层为 `feature-home`、`feature-device`、`feature-doctor`、`feature-relax`、`feature-trend`、`feature-profile`，承接页面、ViewModel、适配器和业务内状态组织。核心层为 `core-common`、`core-model`、`core-data`、`core-ble`、`core-db`、`core-network`、`core-ml`，承接日志、模型、仓库、BLE、数据库、网络、AI 和解析能力。云端层为 `cloud-next`，提供 API、鉴权、AI 路由与后台接口。契约层为 `contracts/`，用于统一 Android 与云端的数据结构与响应封装。

### 功能设计

功能设计围绕用户主链路展开。首页负责汇总当前状态和入口分流；设备页负责接入戒指与采集服务；医生页负责在已有背景下继续追问与解释；症状自查页负责降低第一步输入门槛；报告分析页负责把 OCR 结果转化为用户可理解内容；干预中心负责把建议转为动作；呼吸训练和复盘页负责承接执行与反馈；我的页负责账户、通知、隐私和系统设置。所有页面在导航上统一收口，确保用户只面对一个 App 和一条清晰路径。

### 数据流转设计

数据流转设计分为本地流转与云端流转两部分。本地流转中，BLE 数据、问诊会话、报告原文、执行记录和用户设置进入 Room 或 DataStore，用于离线展示、页面切换和本地兜底。云端流转中，关键数据在用户登录后同步到服务端，用于生成更高质量的问诊摘要、报告理解、处方建议和趋势汇总。为了避免耦合，Android 端通过 `ApiService`、`CloudSessionStore` 与 repository 访问云端，而不直接在页面中拼接网络请求。对于 AI 相关结果，系统尽量保留“原始输入 -> 结构化结果 -> 用户可读内容”的分层方式，以便调试和回退。

### 用户界面设计

用户界面设计遵循“一个底部主导航 + 页面内流程引导 + 机器人适度解释”的原则。底部导航承接今日、医生、趋势、设备、我的五个主入口；二级导航与页面动作承接症状自查、报告分析、干预执行和资料设置等扩展页面。桌面机器人在需要解释的页面显示，在资料、认证和设置类页面隐藏，避免遮挡表单与操作区。机器人文案在页面进入时优先使用本地预热，在点击触发时优先使用云端生成，保证及时性与上下文相关性兼顾。

### 数据结构设计

项目数据结构分为本地实体、跨层模型和接口 DTO 三类。本地实体主要由 Room `Entity` 组成，例如 `sleep_records`、`health_metrics`、`recovery_scores`、`ppg_samples`、`doctor_sessions`、`doctor_messages`、`medical_reports`、`medical_metrics`、`intervention_tasks`、`intervention_executions`、`prescription_bundles`、`prescription_items` 等。跨层模型主要放在 `core-model` 与部分 feature 内状态类中，用于 ViewModel 到 UI 的数据传递。接口 DTO 则通过 `contracts/` 与 `core-network` 中的 `ApiModels` 保持统一。

### 接口设计

#### 外部接口

外部接口主要包括三类：一是 BLE 设备接口；二是云端 HTTP/HTTPS 接口；三是讯飞能力接口。云端当前已经提供 `auth/login`、`auth/register`、`auth/resend-confirmation`、`auth/password-reset`、`sleep/upload`、`sleep/analyze`、`doctor/turn`、`report/understand`、`intervention/daily-prescription`、`user/profile`、`avatar/narration` 等路由。讯飞接口包括 IAT、TTS、OCR、Spark、RTASR、RAASR 和数字人相关配置。所有外部接口都需要通过配置与封装层访问，而不是散落在业务页面中。

#### 内部接口

内部接口主要通过 repository 与 service 抽象暴露。当前 `core-data` 中已经具备 `SleepRepository`、`MedicalReportRepository`、`DoctorConversationRepository`、`DeviceRepository`、`AssessmentRepository`、`InterventionRepository`、`PrescriptionRepository`、`CloudAccountRepository`、`RelaxRepository` 等结构；`core-network` 提供 `ApiClient`、`ApiService`、`CloudSessionStore`、讯飞客户端与配置读取；`core-ml` 提供 `SleepAnomalyDetector`、`MedicalReportParser`、`MedicalReportDraftFormatter`、`HealthAnalyzers` 等本地算法支持。页面层通过 ViewModel 调度这些内部接口，实现职责分离。

### 错误/异常处理设计

#### 错误/异常输出信息

系统对错误提示的要求是“用户可理解、开发可定位”。对用户侧，设备扫描失败应提示权限或蓝牙状态，登录失败应明确区分邮箱未确认、凭据错误和接口缺失，报告分析失败应提示重新拍摄或稍后重试，机器人播报不可用时应保留文字气泡和静态结果。对开发侧，错误返回需要保留 `code`、`message` 和必要的 trace 信息，日志中应记录回退原因而不暴露敏感数据。

#### 错误/异常处理对策

系统采用多级回退策略。BLE 异常时允许断开重连和重新扫描；云端 AI 不可用时使用本地规则、OCR 草案和静态讲解稿；邮件链路异常时通过统一域名、正确回跳页和重发机制修复；配置缺失时通过 BuildConfig 与本地配置检查提示“未配置”，而不是静默失败；症状与报告页在结果缺失时仍应给出最基本的解释和建议，避免出现空白页。整体异常处理思想是保证“最低可用”，同时把错误原因以适合不同对象的粒度展示出来。

### 系统配置策略

系统配置采用本地 Gradle 属性、`local.properties`、云端环境变量与应用内设置结合的方式。Android 侧通过 `DEBUG_API_BASE_URL`、`RELEASE_API_BASE_URL` 和讯飞相关配置控制接口与第三方能力；云端通过 `APP_PUBLIC_BASE_URL`、Supabase、Resend 等环境变量控制鉴权、邮件和回跳域名；应用内通过个人中心的通知设置、机器人语音播报开关等形成用户级配置。系统配置策略要求配置语义清晰，支持区分 debug 与 release，不把机密信息写死在源码中。

### 系统部署方案

Android 端通过 `gradlew.bat :app-shell:assembleDebug` 与 `:app-shell:assembleRelease` 构建 APK，调试包可以通过 `:app-shell:installDebug` 安装到真机。云端通过 Next.js 构建并部署到 Vercel，当前自定义域名统一为 `cloud.changgengring.cyou`。邮件与认证依赖 Supabase Auth 与 Resend SMTP，Cloudflare 负责域名托管与 DNS。部署方案的核心要求是客户端 API 基地址、邮件回跳地址和云端生产域名保持一致，避免出现 Android 调旧域名、邮件跳旧页面的问题。

### 跨端应用架构设计

跨端架构上，长庚环采用“Android 用户端 + Cloud 后台服务 + 契约层 + 模型脚本”的协作方式。Android 用户端负责主要交互和本地能力；云端承担用户态接口与内部管理接口；契约层保证 Android、Cloud 和 ML 在关键字段和逻辑模型上保持一致；模型脚本则负责训练、导出和部分推理验证。跨端架构的重点不是形式上的多目录，而是让数据结构、能力边界和部署链路都能统一解释。当前 `contracts/README.md` 已明确响应封装和 AI 类型定义，这为后续系统演进打下了基础。

### 其他相关技术与方案

除主线技术外，项目还使用了若干辅助技术与方案，例如 Lottie 动画与 Shimmer 提升页面反馈，MPAndroidChart 用于趋势展示，Hilt 用于依赖注入，Room 用于本地数据库，OkHttp + Retrofit 用于网络访问，TFLite 与本地算法模块用于轻量推理和信号处理。整体选型遵循“成熟稳定优先、能解释优先、便于维护优先”的原则。

## 数据库设计

长庚环的数据库设计以 Room 为本地核心存储方案，重点服务于离线展示、历史查询、趋势绘制、执行留痕和云端同步缓冲。数据库主类为 `AppDatabase`，当前已明确包含多类实体。睡眠与体征相关表包括 `sleep_records`、`ppg_samples`、`health_metrics`、`recovery_scores`；设备相关表包括 `devices`；问诊相关表包括 `doctor_sessions`、`doctor_messages`、`doctor_assessments`；评估相关表包括 `assessment_sessions`、`assessment_answers`；医检报告相关表包括 `medical_reports`、`medical_metrics`；干预相关表包括 `prescription_bundles`、`prescription_items`、`intervention_tasks`、`intervention_executions`、`intervention_profile_snapshots`；放松相关表包括 `relax_sessions`。这种设计覆盖了项目当前最重要的主线数据。

数据库设计的原则有三点。第一，围绕主链路建模，而不是为了追求表数量而拆分。第二，支持本地完整闭环，即使云端暂时不可用，系统也能读取历史数据、给出部分建议并保留执行记录。第三，为后续趋势统计与 AI 摘要保留足够的可追踪信息，例如时间戳、会话 ID、任务状态和指标细项。

从关系设计上看，用户与设备、用户与睡眠记录、用户与问诊会话、用户与报告、用户与干预任务之间存在稳定的一对多关系。干预任务与执行记录之间存在明显的主从关系；报告与医检指标之间存在报告头和指标明细关系；医生问诊会话与消息之间存在会话与消息明细关系。数据库因此不仅用于存储，还承担“把碎片事件组织成完整时间线”的职责。

从实现角度看，本地数据库与 repository 层紧密配合。ViewModel 不直接操作 Entity，而是通过 repository 获取领域对象和 UI 友好结果；AI 模块和云端同步模块则基于数据库中的原始和结构化数据进行进一步处理。这种设计增强了可维护性，也便于后续扩展更多分析能力。

## 手机端侧部署设计

### 手机环境需求

手机端侧部署要求 Android 7.0 及以上系统，支持 BLE、网络、相机、录音和通知权限。为了保证智能戒指接入和后台采集能力，建议测试设备支持稳定的蓝牙协议栈，并允许前台服务运行。对于语音与 OCR 功能，设备需具备麦克风和摄像头；对于报告分析和机器人讲解，设备需允许网络访问。若处于 Android 13 及以上系统，还需动态申请通知权限以保证提醒、机器人语音播报和相关服务提示正常工作。

安装部署时，开发版可通过 Gradle 直接安装，正式发布时需使用已配置签名信息的 release 构建。由于项目使用本地配置和第三方能力，部署时还需确认 `local.properties`、云端地址、讯飞配置和账号环境保持一致。若演示设备网络环境不稳定，建议提前准备本地兜底流程并验证主要页面在弱网下是否仍有可读结果。

## 详细设计

### 首页与晨报模块

#### 功能描述

首页与晨报模块负责承接用户最常访问的主入口。它展示恢复评分、睡眠/体征摘要、当日建议与关键导航入口，是从“监测结果”进入“行动建议”的第一步。用户可从首页进入症状自查、旧康复页、干预中心、趋势页和设备页。

#### 性能描述

该模块要求具备较快首屏展示能力。已有本地数据时应优先显示缓存与本地计算结果，再异步刷新云端内容，避免首页因等待网络而失去入口意义。

#### 输入

输入包括本地睡眠记录、恢复评分、近期趋势数据、设备同步状态、云端建议摘要和用户资料。

#### 输出

输出包括恢复评分展示、风险提示、日内行动建议、关键入口卡片以及机器人本地预热说明文案。

#### 程序逻辑

ViewModel 从数据库和仓库获取睡眠与恢复数据，必要时补充云端趋势或摘要结果，再组装为首页 UI 状态。用户点击入口后通过导航进入相应页面。

#### 限制条件

当设备长期未同步或无睡眠数据时，首页只能展示有限摘要，需依赖兜底文案和手动入口。

### 设备接入模块

#### 功能描述

设备模块负责戒指扫描、连接、断开、参数读取和采集服务控制，是整个项目与硬件联动的基础。没有设备模块，项目无法建立连续感知优势。

#### 性能描述

扫描和连接状态必须及时反馈。连接过程不应无提示阻塞，失败后应能继续重试。后台采集需要在前台服务约束下稳定运行。

#### 输入

输入包括蓝牙权限状态、系统蓝牙开关、附近设备广播数据、用户点击行为和设备协议数据帧。

#### 输出

输出包括设备列表、连接状态、错误提示、参数读取结果、采集服务状态和本地入库结果。

#### 程序逻辑

模块通过 `BleManager`、`BleConnectionManager` 和命令构造器完成设备发现与协议交互，解析后的结果进入数据库和 UI 状态层。

#### 限制条件

不同 Android 设备的蓝牙栈差异较大，权限模型和后台限制也会影响体验，因此设备模块必须在兼容性层面预留足够回退。

### 智能问诊与症状自查模块

#### 功能描述

该模块由症状自查页、医生问诊页和数字人活动页组成。症状自查承担轻输入和风险分级，医生页承担结构化追问和结果解释，数字人页承担更强的展示与讲解。

#### 性能描述

首次进入应能快速展示本地预热结果，点击深入交互时再使用云端增强。问诊消息加载与滚动需保持稳定，避免大段文本或音频引发卡顿。

#### 输入

输入包括 2.5D 热点点击、症状 chips、文本输入、语音输入、问诊上下文、用户资料和近期状态数据。

#### 输出

输出包括风险分级、疑似方向、问诊消息列表、问诊摘要、数字人播报与后续建议。

#### 程序逻辑

症状自查先完成结构化输入与本地判断，再把结果传入医生页作为上下文；医生页通过本地与云端结合生成回答；数字人页在需要时调用专门控制器进行展示。

#### 限制条件

该模块不能输出确诊式用语，且在云端不可用时只应给出保守的辅助说明。

### 医检报告分析与干预执行模块

#### 功能描述

该模块负责拍照/OCR、报告文本整理、可读报告生成、异常指标提示、处方建议衔接、呼吸训练、旧康复页入口和执行复盘，是项目把“理解结果”转化为“做点什么”的关键模块。

#### 性能描述

报告页在 OCR 完成后应尽快展示可读内容，即使云端理解未返回也不应空白。呼吸训练和干预执行页要求交互流畅、状态可见、结束可回写。

#### 输入

输入包括图片或文本形式的报告内容、结构化指标、用户症状上下文、近期睡眠/恢复数据和处方任务信息。

#### 输出

输出包括可读报告、重点关注项、下一步建议、干预任务、训练协议、执行状态和复盘结果。

#### 程序逻辑

OCR 与解析结果先在本地整理为中间结构，云端进一步生成用户可读结论；干预中心根据用户状态或报告结果承接下一步动作；执行结束后写回本地数据库并影响后续趋势与建议。

#### 限制条件

报告分析结果必须带有保守表达和免责声明；处方与训练建议应以辅助为目标，不可代替医生诊断。

### 个人中心与系统设置模块

#### 功能描述

个人中心模块负责统一承接账号信息、个人资料、通知设置、数据与隐私、关于项目和系统开关。它也是账号链路和邮件链路的重要入口。

#### 性能描述

登录状态变化后页面应及时刷新；资料保存要有明确回执；设置项切换需要即时生效或明确说明生效范围。

#### 输入

输入包括用户账号状态、资料表单、通知权限状态、开关状态、退出登录操作和本地缓存状态。

#### 输出

输出包括头像和资料展示、登录注册与重发邮件入口、密码重置提示、隐私说明、通知开关状态和版本信息。

#### 程序逻辑

模块根据 `CloudSessionStore` 和云端 `/user/profile` 获取资料，登录注册相关操作通过认证 API 完成，设置项通过本地存储持久化。

#### 限制条件

邮件送达能力依赖外部服务链路，因此界面必须对“待确认”“已发送重置邮件”“接口可用但邮箱未必送达”等情况进行明确提示。
