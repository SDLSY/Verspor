# 移动端 Stitch 全量改版总提示词

更新时间：2026-03-23

## 使用说明
- 这份文档的目标不是解释页面改法，而是提供一条可以直接复制给 Stitch 的超长总提示词。
- 这条提示词面向 Android 移动端整体概念改版，优先服务评审演示，再由后续工程阶段做回收和微调。
- 提示词已经锁定当前项目的核心事实、视觉方向、主页面优先级和非目标，避免 Stitch 只做模板化美化。

## 可直接复制给 Stitch 的总提示词
```text
你现在要为一个真实存在的 Android 原生移动端项目设计一整套世界级水准的移动端界面系统。你的任务不是做网页，不是做后台，也不是做几张概念海报，而是为一套完整的 Android App 做“整体级重设计”。这次设计的第一优先级是评审演示效果、品牌记忆点、未来感和高级完成度，第二优先级才是工程上的保守延续。允许你大胆重组页面结构、信息层级、视觉节奏和关键交互，但不允许你改掉产品的核心能力链路，也不要把它变成一个全新的产品。

项目背景：
这是一个围绕“智能戒指 + 睡眠恢复分析 + AI 问诊 + 医疗报告理解 + 个性化干预 + 闭环反馈”构建的 health-tech 移动端系统。用户佩戴智能戒指采集夜间睡眠与恢复信号，系统基于这些数据形成恢复分、异常判断和趋势理解；用户还可以通过 AI 问诊补充自然语言症状和风险信号，通过医疗报告上传完成 OCR、结构化提取、异常指标识别和风险理解；系统综合这些证据生成个性化建议和干预方案，并在执行后记录反馈、回看效果、形成持续闭环。这个产品不是普通健康记录 App，也不是纯硬件配套页，而是一个具有完整端到端闭环的 premium clinical future 移动端产品。

产品形态与必须保留的基础约束：
1. 这是 Android 原生移动端，不是网页，也不是桌面软件。
2. 整体导航外壳保留底部 5 个一级入口的逻辑心智，不要发明新的一级导航体系。
3. 当前五个一级入口对应的产品角色分别是：
   - Morning Report / Today：首页，负责今日恢复判断和主入口
   - Doctor Chat：医生问诊和建议生成
   - Sleep Trend：趋势和阶段性总结
   - Device：设备连接、同步和可信度状态
   - Profile：身份、设置、云端状态和系统入口
4. 可以重组页面内容、首屏结构、页面节奏和二级流程，但不能破坏核心闭环：
   睡眠恢复首页 -> 问诊/症状补证 -> 报告理解 -> 干预生成/执行 -> 趋势回看
5. App 中存在一个全局 3D avatar / 数字引导体。新设计中请保留它作为品牌识别元素，但它不应该长期抢占主界面的视觉中心或遮挡操作区。它应被重新定义为“辅助讲解、品牌锚点或特定页面的仪式化元素”，而不是每一页都持续悬浮的主角。

当前页面真实问题，你必须主动优化，而不是照抄：
1. 首页目前偏“综合工作台”，信息非常多，睡眠、恢复、AI 判断、建议、个性化状态、干预入口都堆在一起。你要把它重构成更高级的“今日恢复指挥台”。
2. 问诊页目前偏“全功能实验室”，聊天、语音通话、多模态生成、推荐结果、辅助面板混杂在一起。你要把它重构成更清晰的“主问诊区 + 推荐区 + 辅助能力抽屉”。
3. 趋势页目前偏“图表堆叠页”，首屏没有足够强的高层结论。你要把它重构成“先结论、后图表、再深入分析”的节奏。
4. 设备页目前偏功能控制页。你要把它重构成“连接状态与数据可信度页”。
5. 我的页目前偏设置集合页。你要把它重构成“身份与系统控制页”，分区清晰但不过度抢主流程。
6. 症状引导页和报告分析页是这个项目最有差异化的能力，但当前偏长、偏密、偏功能堆叠。你要把它们提升到“临床式工作台”的完成度。
7. 干预中心当前更像入口列表页。你要把它重构成真正的“恢复行动中枢”。

请你基于以下设计哲学工作：
整体气质：
- premium clinical future
- 安静、精密、可信、未来感、克制但强演示感
- 不是传统医疗系统，也不是温柔治愈型 wellness app
- 不是后台 dashboard，也不是普通健康穿戴配套 App

视觉方向：
- 允许电影感首屏、沉浸式 hero、轨道感、脉冲、流线、分层数据流、柔和光晕、结构化玻璃层次、少量精密科幻感
- 要像一款“下一代健康智能终端”，不是一套普通 Material 模板页面
- 版式要有明显主次，不要平均分配每个卡片的重要性
- 允许有更强的视觉锚点、节奏、跨页面统一语言和少量舞台感

色彩方向：
- 瓷白、雾青、深青绿、石墨灰、雾蓝
- 少量铜橙用于关键强调
- 少量警示红只用于高风险和强提醒
- 避免紫色 AI 套板、霓虹蓝紫渐变、廉价玻璃拟态、卡通化饱和色

排版与空间方向：
- 首屏必须更像叙事界面，而不是表格式控制面板
- 大幅强化摘要层，减少一屏并列信息
- 次级内容允许折叠、分段、抽屉化、分页化
- 保留移动端可实现的秩序感，不要做完全不可能落地的超大桌面式布局

交互方向：
- 允许重构关键页面的流程和区块顺序
- 允许把次级能力下沉为抽屉、sheet、折叠区或实验室区
- 不要引入复杂难懂的新一级导航机制
- 不要让功能入口压过主任务

请覆盖以下页面，并明确区分主页面与配套页面，不能所有页面都同模板：

主页面：必须重点设计，并给出明显不同的信息架构
1. Morning Report / Today
2. Doctor Chat
3. Intervention Center
4. Symptom Guide
5. Medical Report Analyze
6. Sleep Trend
7. Device
8. Profile

配套页面：纳入统一系统，但不用与主页面等量展开
1. Breathing Coach
2. Intervention Profile
3. Intervention Session
4. Relax Center legacy
5. Relax Review
6. Medication Analyze
7. Food Analyze
8. Cloud Login / Register
9. Personal Info
10. Notification Settings
11. Privacy Settings
12. About Project

现在请按页面分别理解目标，不要泛泛地做统一卡片语言：

一、Morning Report / Today
它不应该是全量工作台，而应该是“今日恢复指挥台”。
它的首屏只能回答三个问题：
1. 今天状态如何
2. 为什么会得出这个判断
3. 现在最该做什么
它必须把以下内容做成强层级，而不是平铺：
- 恢复状态 / 恢复分
- 睡眠摘要
- AI 风险判断
- 个性化准备度
- 今日建议
- 下一步动作
这个页面要像一天的总控面，而不是信息堆场。

二、Doctor Chat
它不应该像一个堆满功能的实验台，而应该是“主问诊区 + 推荐结果区 + 辅助能力抽屉”。
必须保留：
- 聊天与问诊主流程
- 语音通话能力
- 推荐建议输出
- 历史能力
- 多模态能力
但它们的层级必须重构：
- 主任务始终是问诊和判断
- 语音模式保留，但不能压迫聊天和阅读空间
- 推荐结果必须有独立、清晰、可信的呈现方式
- 多模态生成能力应该下沉为附属实验室抽屉，而不是主流程主角
这页要让人感到“专业问诊工具”，而不是“AI 工具箱”。

三、Intervention Center
这页不该只是入口列表，而应该是“恢复行动中枢”。
需要重新组织这些能力：
- 症状自查
- 放松中心
- 呼吸训练
- 冥想/Zen
- 医疗报告分析
- 用药分析
- 饮食分析
- 执行复盘
要求：
- 给这些入口分优先级和叙事顺序
- 让用户知道现在最适合做哪一步
- 页面整体像行动中枢，而不是目录页

四、Symptom Guide
这页是产品差异化重点。保留“人体定位 + 红旗风险 + 补充信息 + 生成判断”的主线，但整体要做成更高级的“临床式自查工作台”。
必须形成自然漏斗：
1. 快速识别红旗风险
2. 身体定位
3. 录入补充线索
4. 生成结构化判断
5. 给出下一步动作
不要做成长表单，不要让每块都一样重。
人体定位区可以成为视觉主角，但下方判断和建议区必须足够可信和清晰。

五、Medical Report Analyze
这页要从“上传 + OCR + 结果堆叠”重构成“上传、理解、结构化、风险、下一步动作”的清晰链路。
请重点突出：
- 上传与导入动作
- 可读结果
- 结构化指标
- 风险判断
- 下一步建议
请弱化：
- 原始 OCR 文本
- 大段原文编辑
- 技术过程感
OCR 原文和编辑器属于二级层或专家层，不应该占据主任务中心。

六、Sleep Trend
这页要从“长图表页”改成“高层摘要 + 关键走势 + 深入分析”。
首屏必须先给：
- 一段周期性结论
- 风险或状态判断
- 建议或下一步动作
然后再往下展开图表。
图表不应该再简单按列表长堆，而应分层：
- 核心走势
- 关键指标
- 深度分析
- 支撑统计
让用户先懂结论，再看证据。

七、Device
这页要从设备功能页改成“连接状态与数据可信度页”。
重点不是按钮，而是：
- 是否连接
- 数据是否可信
- 最近同步是否成功
- 后台采集状态如何
- 当前最需要做什么动作
扫描、连接、读参、高级工具仍可保留，但要有层级，不要都在一层抢注意力。

八、Profile
这页要从设置集合页改成“身份与系统控制页”。
请清晰区分：
- 身份与账号
- 云端状态
- 通知与隐私
- 项目信息
- 返回主流程的关键入口
这页应安静、克制，不抢主流程，但要有高级完成度。

配套页面的统一原则：
- Breathing Coach、Intervention Profile、Intervention Session、Relax Review 等页要延续主系统语言
- 不必每页都做成大 Hero，但必须有统一的层级和气质
- Cloud Login / Register、Personal Info、Notification Settings、Privacy Settings、About Project 要属于同一套“账户与系统控制”子系统语言
- legacy 功能页可以重做得更现代，但不应偏离主流程系统

请特别注意：
1. 这是一整套 App，不是几个孤立页面。页面之间必须有明显一致的视觉系统，但各自又要有独立的信息架构。
2. 首页、问诊、症状引导、报告分析必须最有辨识度，不能全部沦为“Hero + card list”。
3. 从首页进入干预中心，再到症状引导、报告分析、问诊、执行和趋势，必须形成一条完整且自然的闭环链路。
4. avatar 仍可作为品牌元素出现，但不能再默认每页都以悬浮主角的形式压住内容。
5. 视觉上必须足够惊艳，适合评审演示，但页面仍要看起来像“可以真实落地的 Android 产品”，而不是纯海报。
6. 后台式 dashboard、普通 wellness app、AI 工具市场、杂乱实验室风格都不符合目标。

请输出一整套 Android 移动端概念重设计方案，至少包括：
- 整体视觉系统方向
- 底部 5 Tab 的角色定义
- 主页面与配套页面的层级关系
- 每个主页面的页面结构、首屏重点和信息节奏
- avatar 在新系统中的角色定义
- 主流程如何贯穿多个页面
- 为什么这种设计更适合这个项目

请主动提出一个真正高级、完整、有系统性的移动端设计方案，而不是对现有页面做浅层美化。
```

## 使用后验收要点
- Stitch 输出必须覆盖 5 个一级页面和主要二级能力页。
- 首页、问诊、症状引导、报告分析不能同构。
- 整体必须保持 Android App 心智，而不是网站化稿件。
- 全局 avatar 应作为辅助品牌元素，而不是遮挡主操作区的持续主角。
- 主流程必须能自然串联首页、问诊、报告理解、干预执行和趋势回看。
