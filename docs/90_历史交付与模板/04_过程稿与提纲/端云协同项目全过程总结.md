# 端云协同项目全过程总结

## 一、项目目标
本项目目标是构建可落地的“端云协同”睡眠分析系统：端侧（Android + 指环）负责多维生理数据采集与实时异常信号，云侧负责睡眠分期（WAKE/N1/N2/N3/REM）与异常评分，并输出可解释的恢复建议与趋势结果。

## 二、系统架构与技术路线
- 端侧：Android 应用、蓝牙数据采集、会话与账号管理。
- 云侧：Next.js Route Handlers 作为 API 与 worker 编排层。
- 数据层：Supabase（Postgres/Auth/RLS/Storage）。
- 部署层：Vercel 生产部署，Cloudflare 负责安全与访问防护。
- 模型策略：生产稳态模型 + 研发候选模型并行演进。

## 三、核心工程里程碑
1. **云端基础能力搭建完成**
   - 完成 API 路由、内部 worker、任务调度、模型注册/激活、结果回写。
   - 完成模型回退机制、置信度门控、模型版本管理。
2. **端云数据闭环打通**
   - Android -> Supabase -> 云端推理 -> 报告回写链路贯通。
   - 修复注册登录链路、乱码与配置问题，完成关键可用性改进。
3. **模型训练与评估管线落地**
   - 建立可复现的训练脚本、数据清单、评估与可视化流程。
   - 引入公开数据子集，完成 baseline、半监督、Transformer 多轮实验。
4. **云端增量升级（基于已部署系统）**
   - 在原有云端基础上新增 `ensemble` 推理分支。
   - 新增内部路由并完成 runtime 分发接入，不破坏现有链路。
5. **重新部署并验证**
   - 完成 `lint + build + vercel --prod`。
   - 线上健康检查通过，生产地址已切换到最新版本。

## 四、模型迭代过程与关键结论
### 1) 单模型阶段
- `semi_rf` 在准确率上更稳，适合作为生产默认模型。
- Transformer 在少数类（特别是 N1/REM）有潜力，但存在准确率与宏平均 F1 的权衡。

### 2) 融合阶段（Soft Voting）
- 引入 RF + Transformer 概率级融合后，成功突破单模型“跷跷板”瓶颈。
- 通过多随机种子复验，验证融合方案具备更高上限，但存在一定方差。

### 3) 当前生产策略
- **默认生产模型：`semi_rf`（稳态）**。
- **候选增强模型：`ensemble`（灰度/回归验证后可提升占比）**。
- 该策略兼顾稳定性、可解释性与持续优化空间。
