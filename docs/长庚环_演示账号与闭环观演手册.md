# 长庚环演示账号与闭环观演手册

## 目标与适用环境
本手册对应“单独演示环境”的完整演示方案，目标是用一组可登录的内部演示账号，稳定展示长庚环的 Android 端、云端后台与 AI 闭环，而不是依赖现场临时拼凑数据。

本方案默认包含三部分：
- 云端演示账号与种子数据。
- Android 登录后的自动本地灌库。
- 一份固定的观演顺序与证据说明。

适用范围：
- Android App 演示。
- `cloud-next` 后台管理端演示。
- 比赛答辩、客户路演、内部联调、录像录屏。

不适用范围：
- 真实生产用户环境。
- 未隔离的正式数据库。
- 需要写入真实邮箱、真实密码、真实 token 的仓库文档。

## 账号矩阵
演示环境固定使用 5 个用户场景账号和 1 个后台管理员账号。

| 场景 code | 角色 | 主要用途 | Android 重点页面 | 后台重点页面 |
| --- | --- | --- | --- | --- |
| `demo_baseline_recovery` | 演示用户 | 展示“正常监测 -> 今日恢复分 -> 周/月趋势”主链路 | 今日、趋势、设备 | `dashboard`、`patients` |
| `demo_report_doctor_loop` | 演示用户 | 展示“医检报告 -> 可读报告 -> 医生问诊 -> 干预建议”闭环 | 医检报告、医生、干预中心 | `reports`、患者详情、时间线 |
| `demo_lifestyle_loop` | 演示用户 | 展示“药物/饮食分析 -> 画像 -> 恢复分解释 -> 推荐”闭环 | 药物分析、饮食分析、今日、趋势 | 患者时间线、`recommendations` |
| `demo_live_intervention` | 演示用户 | 展示“戒指连接 -> 实时反馈 -> 呼吸训练 / Zen / 音景 -> 复盘”闭环 | 设备、今日、呼吸训练、Zen、音景、复盘 | 患者干预页、时间线 |
| `demo_high_risk_ops` | 演示用户 | 展示“高风险患者 -> 待处理报告 / 任务 / 失败作业 -> 后台跟踪”闭环 | 今日、医生、趋势 | `dashboard`、`patients`、`reports`、`system/jobs` |
| `demo_admin_console` | 后台管理员 | 只用于后台登录与总览演示 | 不进入 Android 演示 | 后台全部主导航 |

账号邮箱采用统一模板：
- `${场景 code}@${DEMO_ACCOUNT_EMAIL_DOMAIN}`

密码不写入仓库：
- 普通演示用户密码由 `DEMO_ACCOUNT_DEFAULT_PASSWORD` 提供。
- 管理员密码优先读取 `DEMO_ADMIN_DEFAULT_PASSWORD`，未设置时回退到普通演示密码。

后台管理员账号还必须额外加入演示环境的 `ADMIN_EMAIL_ALLOWLIST`。

## 每个账号的场景摘要
### 1. demo_baseline_recovery
入口：登录 Android 后直接进入今日页。  
核心动作：查看今日恢复分、睡眠摘要，再切到趋势页看近 7 天与近 30 天。  
结果：页面中能看到稳定恢复分、连续睡眠记录、周报与月报标题一致。  
后续影响：后台患者概览会同步显示最近睡眠、恢复分、低风险状态。

### 2. demo_report_doctor_loop
入口：登录 Android 后进入医检报告与医生页。  
核心动作：查看一份已解析的异常报告，再生成问诊单并切到干预中心。  
结果：报告异常、问诊摘要、干预建议三段链路都能在同一账号下串起来。  
后续影响：后台 `reports`、患者医疗页和时间线都能看到同一条闭环证据。

### 3. demo_lifestyle_loop
入口：登录 Android 后进入药物分析和饮食分析。  
核心动作：查看最近的药物识别记录、三餐分析记录，再回到今日页看恢复分解释。  
结果：可以明确演示“药物/饮食进入画像并影响恢复分解释”。  
后续影响：后台时间线和推荐轨迹中会有药物、饮食、建议三类证据。

### 4. demo_live_intervention
入口：先进入设备页，再切今日页与干预中心。  
核心动作：现场连接戒指，触发今日生理数据刷新；随后演示呼吸训练、Zen、音景，并查看复盘。  
结果：可以看到“实时数据 -> 执行体验 -> 执行记录 -> 复盘”的现场闭环。  
后续影响：后台患者干预页和时间线会继续显示执行结果，重新 seed 前不会被覆盖。

### 5. demo_high_risk_ops
入口：主要在后台侧演示。  
核心动作：查看高风险患者、待处理报告、失败作业、待执行任务。  
结果：可讲解高风险筛查、任务积压、作业失败与就医优先级。  
后续影响：用于展示后台运营和风控工作台，而不是展示单一用户体验。

## 演示前准备
### 云端
1. 准备独立的演示 Supabase 项目。
2. 配置 `cloud-next` 所需基础环境变量。
3. 额外配置：
   - `DEMO_ACCOUNT_DEFAULT_PASSWORD`
   - 可选 `DEMO_ADMIN_DEFAULT_PASSWORD`
   - 可选 `DEMO_ACCOUNT_EMAIL_DOMAIN`
4. 将管理员邮箱加入 `ADMIN_EMAIL_ALLOWLIST`。
5. 执行 demo seed：
   - `cd D:\newstart\cloud-next`
   - `npm run seed:demo-accounts`

### Android
1. 安装包含 demo bootstrap 接口接线的最新 APK。
2. 保证 App 连接到演示环境 `cloud-next`。
3. 首次切换演示账号时，Android 会自动清空旧演示本地库并导入当前账号对应的本地快照。
4. `demo_live_intervention` 适合现场继续新增记录；其他账号建议只读演示，不做额外写入。

### 后台
1. 使用 `demo_admin_console` 登录后台。
2. 确认 `dashboard`、`patients`、`reports`、`recommendations`、`system` 均可访问。
3. 演示前建议先打开一次需要展示的患者详情页，确保缓存已建立。

## Android 端观演顺序
### 第一段：恢复基线闭环
账号：`demo_baseline_recovery`

顺序：
1. 今日页。
2. 趋势页近 7 天。
3. 趋势页近 30 天。
4. 设备页。

关键看点：
- 今日恢复分不是空白，有稳定的睡眠摘要与生理指标。
- 趋势页近 7 天显示“趋势周报”，近 30 天显示“趋势月报”。
- 周报与月报内容能对应当前时间范围。

后台验证：
- `dashboard` 顶部患者摘要。
- `patients` 中该用户的最新恢复分与低风险状态。

### 第二段：报告 -> 问诊 -> 干预
账号：`demo_report_doctor_loop`

顺序：
1. 医检报告页查看已解析报告。
2. 查看 AI 整理后的可读报告。
3. 进入医生页。
4. 生成问诊单。
5. 切到干预中心看任务建议。

关键看点：
- 异常报告不只是“识别完成”，而是进入问诊。
- 医生页能生成结构化问诊单，而不是停在普通聊天。
- 干预中心能看到与该链路相对应的任务。

后台验证：
- `reports` 列表中有对应异常报告。
- 患者医疗页能看到异常指标。
- 患者时间线能看到报告、问诊、任务三类事件。

### 第三段：药物 / 饮食 -> 画像 / 恢复分
账号：`demo_lifestyle_loop`

顺序：
1. 药物分析页。
2. 饮食分析页。
3. 今日页恢复分解释。
4. 趋势页。

关键看点：
- 已有药物识别记录和三餐分析记录。
- 今日页解释中能讲清楚为什么恢复分受到生活方式影响。
- 趋势页可作为“不是孤立单次记录，而是持续影响”的补充证据。

后台验证：
- 患者时间线出现药物识别与饮食分析两类事件。
- `recommendations` 中能看到近期建议轨迹。

### 第四段：现场干预闭环
账号：`demo_live_intervention`

顺序：
1. 设备页连接戒指。
2. 切回今日页，确认生理数据刷新。
3. 进入呼吸训练。
4. 进入 Zen 交互。
5. 进入音景干预。
6. 进入复盘页。

关键看点：
- 今日页的生理数据能跟随设备连接而更新。
- 呼吸训练具备实时视觉反馈和触觉节拍。
- Zen 页可交互，音景页可演示白噪音/环境音。
- 执行后复盘页能看到本次会话结果。

后台验证：
- 患者干预页有任务与执行记录。
- 患者时间线新增执行事件。

### 第五段：高风险运营闭环
账号：`demo_high_risk_ops`

Android 端只做辅助：
1. 今日页查看低恢复分与高风险提示。
2. 医生页查看高风险问诊摘要。

真正主讲在后台：
- `dashboard`
- `patients`
- `reports`
- `system/jobs`

关键看点：
- 低恢复分、高风险报告、失败作业、待处理任务能串起来讲。
- 强调“先就医评估，再用保守干预作辅助”的边界。

## 后台观演顺序
### dashboard
看点：
- 患者总览不为空。
- 能看到高风险患者数、待执行任务、失败作业等摘要。

### patients
看点：
- 每个 demo 账号对应不同风险层级与不同链路重点。
- 患者详情能展开看概览、睡眠、干预、医疗、时间线。

### reports
看点：
- 至少能看到异常报告及其风险等级。
- 报告与患者链路可相互跳转。

### recommendations
看点：
- 可以看到 demo 场景下生成过的建议轨迹。
- 用于解释“为什么给出这些建议”。

### system/jobs
看点：
- `demo_high_risk_ops` 中至少有失败作业和排队作业。
- 可用于讲“模型运行、作业队列、运营排查”。

## 每个闭环的关键证据点
### 恢复基线闭环
- App 今日页有恢复分。
- 趋势页周报/月报有连续数据。
- 后台患者概览能看到最近恢复分和风险。

### 报告问诊闭环
- App 报告页存在已解析报告。
- App 医生页能生成问诊单。
- 后台时间线能看到报告、问诊、干预任务。

### 生活方式闭环
- App 药物和饮食页存在结构化记录。
- App 今日解释能将生活方式与恢复分关联。
- 后台时间线能看到药物分析、饮食分析、推荐轨迹。

### 现场干预闭环
- 设备连接后今日页生理数据更新。
- 呼吸训练 / Zen / 音景能实际完成。
- 复盘页与后台时间线都能看到执行回写。

### 高风险运营闭环
- 患者恢复分低且风险高。
- 后台报告页能看到高风险报告。
- `system/jobs` 能看到失败与排队作业。

## 如何 reset 种子数据
1. 确认 `cloud-next` 指向演示环境。
2. 确认以下环境变量存在：
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `SUPABASE_SERVICE_ROLE_KEY`
   - `DEMO_ACCOUNT_DEFAULT_PASSWORD`
3. 在 `D:\newstart\cloud-next` 执行：
   - `npm run seed:demo-accounts`
4. 重新登录需要演示的账号。
5. Android 若已导入旧版本演示数据，重新登录后会按 `demoSeedVersion` 自动重灌本地库。

## 常见异常与排查
### 1. Android 登录成功，但页面还是空的
排查：
- 确认该账号 metadata 中 `demoRole=demo_user`。
- 确认 `/api/demo/bootstrap` 能返回数据。
- 确认当前 APK 包含自动 demo bootstrap 逻辑。

### 2. 切换账号后仍然显示上一个账号的数据
排查：
- 确认登录确实切换了账号。
- 确认 `demoSeedVersion` 有变化或账号 id 不同。
- 必要时退出登录后重新登录。

### 3. 后台管理员无法登录
排查：
- 确认 `demo_admin_console` 已创建。
- 确认该邮箱已加入 `ADMIN_EMAIL_ALLOWLIST`。
- 确认登录的是演示环境而不是生产环境。

### 4. system/jobs 没有内容
排查：
- 重新执行 seed。
- 确认 `demo_high_risk_ops` 的 `inference_jobs` 已成功插入。

### 5. recommendations 页面为空
排查：
- 确认 `recommendation_traces` 已插入。
- 确认后台时间范围筛选没有把 demo 数据过滤掉。

## 安全规则
- 不在仓库中写入真实邮箱、真实密码、真实 token。
- 文档中只写账号模板、场景 code、环境变量名。
- demo 环境必须与正式环境隔离。
- 演示账号只用于内部演示，不进入真实用户体系。
- 需要共享账号时，优先共享 seed 脚本与环境变量配置方式，不共享明文密码。
