## 2. 项目基本信息

### 2.1 项目名称

- 中文名称：长庚环
- 英文名称 / 历史代号：VesperO
- 团队名称：夜巡者

### 2.2 项目形态

长庚环是一个以智能戒指为感知入口、以 Android 应用为交互中心、以云端 AI 服务为增强能力的端云协同健康管理项目。它面向的场景并不只是单一的“睡眠监测”，而是围绕“睡眠、恢复、压力、症状自查、问诊辅助、医检报告理解、日内干预执行”构建一条闭环。

项目当前的系统主线可以概括为：

`感知 -> 分析 -> 建议 -> 执行 -> 反馈`

如果用仓库中更稳定的产品语言表述，则是：

`统一画像 -> 处方 -> 执行 -> 反馈`

这意味着项目的核心目标不是做更多静态图表，也不是只把指标展示出来，而是基于设备采集、主观输入、问诊结果和报告内容形成一个可持续更新的个人健康画像，再围绕这个画像给出可执行的干预与建议，并通过后续执行效果反向修正推荐逻辑。

### 2.3 当前工程边界

当前仓库不是单纯的 Android App 仓库，而是一个多端协同的工程仓库，主要包含以下几类内容：

- Android 运行入口与业务代码
- 云端 API 与 AI 路由编排
- 共享契约定义
- 模型训练、导出与推理脚本
- 项目文档、阶段性总结和答辩材料

这意味着任何试图描述该项目的 AI 都不应将其简化为“一个 App”，而应将其理解为：

1. 有设备端输入。
2. 有移动端承载。
3. 有云端 AI 和数据存储。
4. 有结构化契约。
5. 有模型与算法侧支撑。

## 3. 项目定位与问题背景

### 3.1 项目要解决的核心问题

从项目当前代码与文档事实出发，长庚环主要聚焦以下问题：

#### 3.1.1 单纯监测不能直接转化为行动

很多健康类项目可以采集大量睡眠、心率、体温、恢复、压力相关数据，但这些数据往往只停留在“展示”层面。用户可能可以看到分数、趋势、异常提示，却无法直接获得一个足够具体、足够可执行、足够贴近当前状态的行动建议。

长庚环试图解决的问题之一，就是把监测结果与“下一步怎么做”之间建立更短的路径。

#### 3.1.2 睡眠、恢复、症状、问诊、医检之间长期割裂

很多系统是按能力孤岛构建的：

- 一个模块做设备监测
- 一个模块做问答咨询
- 一个模块做报告 OCR
- 一个模块做放松训练

但真实用户面对的是一个连续的健康问题，而不是独立模块。长庚环当前代码结构虽然仍在持续整理，但产品主线已经明确试图把这些能力放在同一个软件和同一条业务链上。

#### 3.1.3 用户需要“辅助判断 + 执行动作”，而不是单纯文本结论

项目当前已经存在：

- 症状自查与辅助诊断向导
- AI 医生问诊页
- 医检报告分析页
- 干预中心、执行页与复盘页
- 呼吸训练与康复辅助页

这说明项目定位已经从“静态健康看板”向“可交互的健康辅助系统”演进。

### 3.2 当前产品定位

如果要为后续 AI 提供一个稳定的定位描述，可以使用以下表述：

> 长庚环是一个以智能戒指连续感知为基础、以 Android 单入口应用为用户交互中心、以云端 AI 和本地算法为分析增强的睡眠恢复与健康干预系统。系统将设备数据、症状自查、AI 问诊、医检报告理解和干预执行整合在一起，形成“画像—建议—执行—反馈”的闭环。

这个描述比“智能睡眠戒指 App”更准确，因为它明确包含了：

- 戒指端感知
- Android 单入口
- 本地算法
- 云端 AI
- 症状与问诊
- 干预执行闭环

### 3.3 当前不应被夸大的内容

为了防止后续 AI 生成夸张或失真的材料，以下内容必须明确：

1. 项目不是一个经过医疗器械注册认证的临床诊断系统。
2. 系统中的“症状自查”“辅助诊断”“可读报告”均应理解为辅助理解与辅助判断，不应描述为确诊。
3. 当前项目虽然有端云一体能力，但多模块架构仍在持续收敛中，不能写成“所有模块都已经彻底解耦并完全完成”。
4. Android 端已经统一为单入口运行，但代码组织层面仍有进一步优化空间。
5. 云端使用了多个 AI 能力入口，但不同能力的重要性和成熟度不同，不能将所有 AI 路由一概表述为“都已正式稳定上线”。

## 4. 系统组成

### 4.1 Android 端

Android 端当前唯一运行入口是 `:app-shell`。这意味着从软件交付和 APK 构建的角度看，当前项目已经是一套统一的 Android 应用，而不再是历史上的多入口混合状态。

Android 端承担的职责主要包括：

- 承接用户可见的所有页面与交互
- 设备连接与扫描
- 本地数据缓存
- 本地数据库持久化
- 本地异常检测与部分 AI 兜底能力
- 访问云端 API
- 承接桌面虚拟机器人、TTS 播报、语音相关交互

Android 端的主要导航结构包括：

- 今日
- 医生
- 趋势
- 设备
- 我的

在这些一级导航之外，还包括多个二级功能页，例如：

- 症状自查
- 干预中心
- 旧放松中心
- 呼吸训练
- 医检报告分析
- 评估页
- 干预会话页
- 个人信息
- 通知设置
- 数据与隐私
- 关于项目
- 云端账号登录/注册/忘记密码

### 4.2 云端

云端主线位于 `cloud-next/`，技术栈为 Next.js Route Handlers + Supabase。

云端的职责包括：

- 提供公共 API
- 提供登录、注册、密码重置等认证接口
- 提供 AI 路由和统一编排入口
- 处理睡眠分析、问诊、医检理解、语音、图像和视频类任务
- 保存结构化结果和业务数据
- 提供与 Android 对应的数据同步与用户资料接口

当前公开或产品相关的云端路由族包括：

- 认证类：`/api/auth/*`
- 睡眠类：`/api/sleep/*`
- 恢复类：`/api/recovery/*`
- 报告类：`/api/report/*`
- 干预类：`/api/intervention/*`
- 建议类：`/api/advice/generate`
- 医生类：`/api/doctor/*`
- AI 多模态类：`/api/ai/speech/*`、`/api/ai/image/*`、`/api/ai/video/*`
- 用户类：`/api/user/profile`
- 同步类：`/api/sync`

### 4.3 契约层

契约层位于 `contracts/`，承担 Android、Cloud 与 ML 之间共享数据结构的角色。

当前共享的内容重点包括：

- JSON Schema
- TypeScript 类型
- Kotlin DTO
- AI provider / capability / logical model id 等统一枚举与约束
- 统一响应包络约定 `{ code, message, data, traceId }`

契约层的重要性在于，它不是文档，而是工程中的“跨端共识”。如果后续 AI 需要生成更严谨的技术文档，应该把 `contracts/` 理解为跨栈数据结构的权威来源之一。

### 4.4 ML / 算法脚本层

模型相关脚本与训练导出推理流程位于 `ml/`。这一层不直接面向用户，但承担算法验证、训练、导出与推理脚本的角色。

它与 Android/Cloud 的关系是：

- Android 端使用部分端侧模型和算法逻辑
- 云端承担更重的生成与编排
- `ml/` 用于支撑模型训练、导出和验证流程

因此在系统边界上，`ml/` 更像研发资产层，而不是直接运行时入口。

## 5. 当前产品能力总览

### 5.1 今日页

今日页是主入口之一，承担“当日状态概览”和“快速进入主流程”的角色。按照当前项目结构，今日页重点承接：

- 恢复分与当日状态摘要
- 风险提示
- 关键入口引导
- 干预中心或相关链路入口

它不是一个纯展示页，而更像系统的“日常使用首页”。

### 5.2 医生页

医生页是 AI 问诊和医生交互主线的核心页。当前医生页包含：

- 文本消息对话
- 问诊状态组织
- 本地与云端结合的医生推理链路
- 语音问诊交互
- 数字人/桌面机器人联动

当前产品中，医生页不是单纯聊天框，而是一个围绕“主诉、持续时间、伴随症状、追问、结构化结果”组织的辅助问诊页面。

### 5.3 趋势页

趋势页承担的是历史趋势展示与周期总结能力。它更关注“变化”，而不仅是“当前一刻”。项目中的趋势页一般用于：

- 睡眠数据趋势
- 恢复趋势
- 周期报告
- 按阶段观察改善或波动

### 5.4 设备页

设备页用于承接智能戒指相关连接与状态管理，当前重点包括：

- 扫描设备
- 连接与断开
- 设备列表
- 数据采集入口
- 部分参数或同步状态查看

从项目整体定位看，设备页是感知入口，不是最终价值交付页面。真正的系统价值仍在后续分析、建议和干预。

### 5.5 我的页

“我的”页承担账户、资料与设置管理功能。当前已具备真实页面而不是占位项，主要包括：

- 云端账号登录/注册/忘记密码
- 个人信息
- 通知设置
- 数据与隐私
- 关于项目

这表明项目已不只是技术演示页面，而具备一定完整产品化账户链路。

### 5.6 症状自查与辅助诊断向导

这是近阶段重点改造的能力之一。当前项目中原“3D 干预中心”被扩展为更偏互联网医疗风格的症状定位与辅助判断页面，具有以下特征：

- 支持前后视角的人体图切换
- 支持症状圈选与快捷症状选择
- 强调“定位 -> 补充症状 -> 辅助判断”
- 结果以风险分级、疑似方向、建议下一步呈现
- 与 AI 医生、医检报告和干预建议形成联动

这一能力使项目从“恢复干预工具”向“症状自查与康复辅助系统”更进一步。

### 5.7 旧放松中心 / 康复辅助场景

旧的放松中心仍然保留，并没有因为症状自查页上线而消失。它当前更适合被理解为：

- 康复辅助场景
- 分区动作展示
- 放松训练入口
- 与干预执行主线的历史能力兼容层

### 5.8 医检报告分析

医检报告分析页当前承担的功能包括：

- 报告图片输入
- OCR 识别
- 结构化指标解析
- 风险等级与异常项提取
- 可读报告生成
- 云端 AI 增强理解
- 本地兜底文本生成

这一模块的重要性在于它将“影像或纸质报告 -> 用户可理解信息”的链路纳入系统。

### 5.9 干预中心、执行页与复盘页

这些页面共同构成从建议到执行到反馈的闭环：

- 干预中心：决定当前建议与处方
- 干预执行页：承接某个干预任务的具体执行
- 复盘页：记录执行效果、体感与结果变化

这是项目区别于单纯问答或纯监测产品的关键部分。

### 5.10 呼吸训练与辅助训练页

当前项目中还包含呼吸训练等辅助功能，用于在较低风险或需要放松/调节的场景下提供可立即执行的动作。

## 6. 当前技术架构总览

### 6.1 Android 当前已统一为单入口

当前 Android 端已经完成“单一入口收敛”：

- `:app-shell` 是唯一 Android 运行入口
- 旧 `app/` 已作为历史归档保留
- 当前 APK 不再通过 `sourceSets` 编译 legacy `app/`

这意味着对于后续 AI 来说，可以稳定地把 Android 端表述为“一个软件、一个入口、一个主工程”，而不必再描述历史上的双入口混合状态。

### 6.2 模块化正在持续收敛

虽然 Android 端已经统一为单入口，但从代码组织看，多模块架构仍在持续收敛中。当前大致状态如下：

- `app-shell`：宿主层与当前运行时代码承载层
- `core-*`：基础能力模块
- `feature-*`：特性模块

已经明确存在的模块包括：

- `:core-common`
- `:core-model`
- `:core-data`
- `:core-ble`
- `:core-network`
- `:core-db`
- `:core-ml`
- `:feature-home`
- `:feature-device`
- `:feature-doctor`
- `:feature-relax`
- `:feature-trend`
- `:feature-profile`

后续 AI 在生成技术文档时应这样描述：

> 当前项目已完成 Android 单入口统一，基础模块和特性模块边界已经建立，部分核心能力已迁入 core/feature 模块，整体架构处于“可运行、功能完整、持续收敛”的阶段。

### 6.3 端云协同是当前真实主线

本项目不适合被描述为纯本地离线 App，也不适合被描述为纯云端 SaaS。它的真实形态是端云混合：

- 设备端负责感知
- Android 端负责交互、本地存储、本地分析和云端接入
- 云端负责统一 API、认证、数据保存与较重的 AI 任务
- 契约层负责跨端结构统一
- 模型脚本层负责训练、导出与验证

## 7. 当前用户主链路

### 7.1 设备驱动主链路

一个典型的用户主链路可能是：

1. 用户佩戴戒指。
2. App 在设备页扫描并连接设备。
3. 设备数据被采集并保存到本地。
4. 睡眠、恢复、压力等分析结果在今日页或趋势页展示。
5. 系统基于这些结果给出建议或推荐进入干预中心。
6. 用户执行干预任务。
7. 执行结果和后续恢复表现继续反哺系统。

### 7.2 症状自查主链路

另一条重要链路是：

1. 用户从相关入口进入症状自查页面。
2. 用户选择前后身体视图、圈选部位或点击症状。
3. 用户补充严重度、持续时间、伴随症状等信息。
4. 系统给出风险分级和辅助判断结果。
5. 用户可以继续进入 AI 医生页、上传报告、或者进入康复/放松建议。

### 7.3 医检报告主链路

1. 用户导入或拍摄报告。
2. 系统进行 OCR 与结构化解析。
3. 云端或本地生成可读报告。
4. 用户查看重点异常、指标解释和建议下一步。
5. 结果可继续与医生页、趋势页、干预页形成联动。

### 7.4 医生问诊主链路

1. 用户进入医生页。
2. 输入主诉、症状、历史情况或语音描述。
3. 系统进行结构化追问和问诊组织。
4. 输出阶段性判断与建议。
5. 用户可继续回到其他业务链路执行。

3. 云端已具备认证接口、用户资料接口、睡眠与干预相关接口、AI 多模态相关接口。
4. 项目已存在结构化契约层 `contracts/`。
5. 项目已存在端侧算法、本地 OCR、本地兜底与云侧 AI 增强的混合策略。

以下内容可以写，但要注意措辞：

## 9. 可直接供其他 AI 使用的项目摘要

以下内容可以作为后续 AI 写作时的精简输入：

### 9.1 一句话版本

长庚环是一个围绕智能戒指、Android 单入口应用和云端 AI 构建的端云协同健康辅助系统，面向睡眠恢复、症状自查、AI 问诊、医检报告理解和干预执行闭环。

### 9.2 三句话版本

长庚环以智能戒指作为连续感知入口，以 Android 应用作为统一交互中心，以云端 API 和 AI 编排作为能力增强层。系统将设备数据、症状输入、AI 问诊结果、医检报告解析和干预执行记录整合为统一画像，再围绕该画像生成建议、处方与执行动作。当前项目已经形成单一 Android 入口、云端统一接口和跨端契约层，但代码架构仍在持续收敛与优化。

### 9.3 扩展摘要版本

长庚环项目由夜巡者团队开发，英文或历史代号为 VesperO。项目并非单一的睡眠监测工具，而是一个围绕“感知—分析—建议—执行—反馈”构建的智能健康辅助系统。它以智能戒指采集睡眠、恢复、压力等数据，以 Android 单入口 App 承担用户交互、设备连接、本地分析、问诊、报告理解和干预执行，以 Next.js + Supabase 构建云端 API 与数据底座，并通过契约层统一 Android、云端和模型脚本间的数据结构。当前系统已整合今日页、医生页、趋势页、设备页、我的页，以及症状自查、旧放松中心、医检报告分析、呼吸训练、干预中心、执行页和复盘页等功能模块。其核心特点在于，不再将监测、问诊、报告和干预割裂处理，而是将其组织为一条可验证的健康辅助闭环。

## 10. 后续 AI 使用本文档时的约束

后续任何 AI 如果基于本文档生成计划书、研究报告、开发文档或答辩材料，应遵循以下约束：

1. 以本文档中的已实现事实为准，不要凭空补充未证实模块。
2. 不要将辅助判断表述为确诊。
3. 不要将“架构正在收敛”写成“所有模块都已完全重构”。
