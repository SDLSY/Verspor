# 长庚环 AI 填充用：关键数据流与时序链路

## 1. 文档定位

本文档的目标不是重复模块说明，而是把当前项目里最关键的“数据怎么流、页面怎么触发、云端在哪一环参与”写清楚，让后续 AI 在写技术材料、理解系统方案、推导模块关系时不至于只停留在目录结构层面。

本文档重点覆盖以下几条主链：

- 戒指采集 -> BLE -> 本地数据库 -> 今日页 / 趋势页
- 症状自查 -> 风险分层 -> AI 医生 / 干预入口
- 医检报告 OCR -> 结构化指标 -> 可读报告 -> 画像 / 建议
- 登录 / 注册 / 邮箱确认 / 忘记密码
- 桌面机器人页面进入播报与点击重播
- 干预任务 -> 执行 -> 效果回写 -> 趋势 / 推荐

每条链路都按统一结构描述：

- 触发源
- 中间层
- 数据落点
- 页面表现
- 云端参与点
- 当前边界

## 2. 戒指采集 -> BLE -> 本地数据库 -> 今日页 / 趋势页

### 2.1 触发源

用户在设备页完成戒指连接，或系统在前台采集服务中持续接收戒指数据。触发源不是云端，而是 Android 端的 BLE 连接状态和本地采集生命周期。

### 2.2 中间层

当前链路可概括为：

1. 戒指设备通过 BLE 向 Android 发送原始传感器数据。
2. Android 侧 BLE 管理层负责连接、发现服务、读写特征值与命令发送。
3. `DataCollectionService` 作为前台服务接收和组织采集流。
4. `SleepRepository` 等仓库把数据写入本地数据库。
5. 本地分析和聚合结果再供今日页、趋势页和必要的云端同步使用。

### 2.3 数据落点

第一落点是 Android 本地数据库，而不是云端。  
也就是说，长庚环当前的设备数据主链是“先本地持久化，再按业务需要同步云端”，而不是“所有数据原样直通云端后再回显”。

### 2.4 页面表现

- 今日页读取恢复分、风险提示、建议入口等聚合结果。
- 趋势页读取历史曲线、阶段统计和趋势变化。
- 设备页则负责连接状态、电量、同步、扫描、调试等设备层 UI。

### 2.5 云端参与点

云端主要参与：

- 睡眠上传与分析
- 恢复趋势聚合
- 数据同步

对应接口组主要是：

- `/api/sleep/*`
- `/api/recovery/trend`
- `/api/sync`
- `/api/data/upload`

### 2.6 当前边界

- 不要把戒指原始采集写成“完全云端处理”，当前本地数据库和本地分析链路是明确存在的。
- 不要把 BLE 层写成可绕开 Android 的独立系统，当前它严格依附于 Android 端。
- 不要把设备页的扫描、连接、工作参数提示等 UI 直接等同于云端接口行为。

## 3. 症状自查 -> 风险分层 -> AI 医生 / 干预入口

### 3.1 触发源

用户从“症状自查”页面进入，通过部位点击、症状 chips、强度和持续时间等轻量交互组织当前主诉信息。

### 3.2 中间层

当前链路大致为：

1. `SymptomGuideFragment` 和 `SymptomGuideViewModel` 收集部位、症状、强度、持续时间等状态。
2. 本地视图模型先生成风险分层与结果页结构。
3. 用户可继续进入医生页，或进入干预 / 康复辅助链路。
4. 若需要更深度的追问或生成解释，会进一步调用云端医生增强接口。

### 3.3 数据落点

这条链路的第一落点主要是本地 UI 状态和本地业务对象，不是立即写库。  
当用户继续到医生问诊、干预记录或评估摘要时，才进一步进入云端或持久化链路。

### 3.4 页面表现

- 症状自查页先输出风险等级、疑似方向、建议下一步。
- 结果页可以继续跳转到 AI 医生页。
- 结果页也可以进入康复辅助或干预执行页。

### 3.5 云端参与点

云端主要参与两类增强：

- `doctor/turn`：对当前主诉和历史上下文做下一轮追问或解释。
- 相关评估 / 建议接口：把自查结果进一步沉淀到更长期的建议链路中。

### 3.6 当前边界

- 不要把症状自查写成确诊系统；当前更准确的表述是症状定位、风险分层和辅助判断向导。
- 不要把自查页写成强依赖长对话聊天；当前交互更偏点击、圈选和结构化补充。
- 不要把“症状自查结果”写成一定已经入库；很多状态是在后续动作中才进一步持久化。

## 4. 医检报告 OCR -> 结构化指标 -> 可读报告 -> 画像 / 建议

### 4.1 触发源

用户在医检报告分析页上传图片或文本，或将 OCR 识别结果导入后进入分析流程。

### 4.2 中间层

当前链路可分为：

1. OCR 文本或报告原文进入 Android 端。
2. 本地解析层先尝试提取结构化指标、生成草案与可读摘要兜底。
3. 若云端可用并已登录，可进一步调用 `report/understand` 做更强解释。
4. 结构化指标可通过 `metrics/upsert` 等接口写入云端。
5. 结果页回到用户侧，以“可读报告、异常项、建议下一步”的形式展示。

### 4.3 数据落点

数据落点分两层：

- Android 本地：OCR 原文、草案、页面状态、必要的本地解析结果。
- 云端：结构化指标、最新报告摘要、阶段汇总结果。

### 4.4 页面表现

用户最终看到的不是原始 JSON，而是：

- 可读摘要
- 异常项列表
- 指标解释
- 建议下一步
- 必要时保留原始 OCR 文本作为校对区

### 4.5 云端参与点

云端参与点主要包括：

- `/api/report/understand`
- `/api/report/metrics/upsert`
- `/api/report/latest`
- `/api/report/period-summary`

### 4.6 当前边界

- 不要把当前链路写成“OCR 后立即生成医学诊断结论”，系统更准确的定位是报告结构化理解与用户可读化。
- 不要写成“没有云端就没有结果”，当前系统已经做了本地兜底。
- 不要把“可读报告”理解成完全脱离原始 OCR 的真相层；它仍然依赖 OCR 与结构化抽取。

## 5. 登录 / 注册 / 邮箱确认 / 忘记密码时序

### 5.1 触发源

用户在“我的”页或认证页点击登录、注册、重发确认邮件或忘记密码。

### 5.2 中间层

当前时序如下：

1. Android 认证页收集邮箱和密码。
2. Android 通过 `ApiService` 调用 `auth` 路由。
3. 云端转而调用 Supabase Auth。
4. 若注册需要邮箱确认，云端返回 `PENDING_CONFIRMATION` 状态。
5. 若忘记密码，云端请求 Supabase 发送重置邮件，并把回跳地址指向 `/reset-password`。
6. 若用户点击邮件确认链接，会先进入 `/auth/confirm`；点击重置链接则进入 `/reset-password`。
7. 用户完成确认或修改密码后，再回到 Android 进行登录。

### 5.3 数据落点

- 会话令牌与基本账号资料落在 Android 本地会话存储中。
- 用户主账号状态由 Supabase 维护。
- 用户资料由云端用户资料接口提供。

### 5.4 页面表现

- 登录页支持登录、注册、忘记密码。
- 待确认邮箱会显示明确状态，而不是简单报错。
- 忘记密码邮件会引导用户到 Web 重置页，而不是在 Android 内直接改密码。

### 5.5 云端参与点

- `/api/auth/register`
- `/api/auth/login`
- `/api/auth/resend-confirmation`
- `/api/auth/password-reset`
- `/auth/confirm`
- `/reset-password`

### 5.6 当前边界

- 不要把忘记密码写成 Android 内部纯本地流程；当前是 Android 发起、云端转发、邮件回跳到 Web 页的混合方案。
- 不要忽略邮箱确认状态；这是当前认证链路的关键分支。
- 不要把 Web 页误写成单独的前台业务站点，它们主要是认证回跳页。

## 6. 桌面机器人页面进入播报与点击重播时序

### 6.1 触发源

触发源分为两类：

- 进入页面：如今日页、医生页、症状自查页、报告分析页等。
- 用户点击桌面机器人：要求补充说明、重播或生成更贴近当前页面状态的话术。

### 6.2 中间层

当前策略是两段式：

1. 页面进入时，优先使用本地预热文案与本地缓存语音，减少等待。
2. 点击机器人时，再根据当前页面和状态调用云端 `avatar/narration` 生成动态文本。
3. 若云端不可用，则回退到本地 Spark 或本地规则兜底。
4. 最终由 Android 端完成文字展示与语音播报。

### 6.3 数据落点

- 本地：页面语义、预热文案、缓存音频、当前播报状态。
- 云端：只参与生成页面说明文本，不承担长期状态存储主责。

### 6.4 页面表现

- 页面进入时，用户先听到短而稳定的本地导览文案。
- 点击机器人时，文案更贴合当前页面与状态。
- 通知设置中已经支持“机器人语音播报”开关。

### 6.5 云端参与点

- `/api/avatar/narration`
- 语音链路还可能经由 `ai/speech/*` 做合成或其他增强

### 6.6 当前边界

- 不要把桌面机器人写成开放式聊天助手；它更偏页面导航机器人。
- 不要把本地预热和云端生成混成一个不可区分的链路；当前两者的分工很清楚。
- 不要把所有页面都写成同一套固定播报，当前已经按页面语义区分。

## 7. 干预任务 -> 执行 -> 效果回写 -> 趋势 / 推荐

### 7.1 触发源

用户从今日页、症状自查结果页、报告分析页或康复辅助页进入干预执行流程。

### 7.2 中间层

典型链路如下：

1. 系统根据当前状态组织建议和候选动作。
2. 用户进入干预任务或执行页面。
3. 执行过程中记录完成率、体感和阶段结果。
4. 执行后将任务、执行记录和效果摘要写回本地与云端。
5. 趋势页、推荐链路和复盘链路进一步消费这些结果。

### 7.3 数据落点

- 本地：执行中的 UI 状态与即时反馈。
- 云端：任务记录、执行记录、效果趋势、评估摘要。

### 7.4 页面表现

- 干预页展示动作、节律、提示与完成情况。
- 复盘页展示执行后的效果变化。
- 趋势页可进一步呈现长期改善或波动趋势。

### 7.5 云端参与点

- `/api/intervention/task/upsert`
- `/api/intervention/execution/upsert`
- `/api/intervention/effect-trend`
- `/api/assessment/baseline-summary/upsert`
- 与推荐建议相关的接口组

### 7.6 当前边界

- 不要把执行效果写成完全自动评估；当前仍有用户体感与半结构化反馈参与。
- 不要把干预、趋势、推荐当成完全割裂的功能块；它们在业务上构成闭环。
- 不要把“执行一次动作”直接写成“已经完成长期疗效评估”。

## 8. 当前边界与不要误写的点

1. 当前系统是端云协同，不是“所有链路都从云端开始、在云端结束”。
2. Android 本地数据库、本地分析、本地兜底逻辑在多条主链中都是真实存在的。
3. 症状自查、报告理解、医生问诊、干预执行不是相互孤立的模块，而是通过数据摘要和页面跳转串成闭环。
4. 认证链路混合了 Android 发起、云端封装、Supabase 执行和 Web 回跳页，不能写成单层流程。
5. 桌面机器人不是单纯的 UI 装饰，它已经接入本地预热与云端页面叙事生成。

## 9. 适合写进正式材料的统一标准表述

长庚环的关键业务链路采用典型的端云协同模式：底层设备采集、即时状态管理和部分分析逻辑优先在 Android 端完成，结构化摘要、趋势聚合、报告理解、问诊增强和账号体系由 `cloud-next` 统一承接。项目通过症状自查、医检报告、AI 医生、干预执行与趋势复盘等链路，把“感知—理解—建议—执行—反馈”组织成连续闭环，而不是若干孤立页面的堆叠。
