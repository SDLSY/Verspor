# 长庚环 AI 填充用：标准投喂提示词模板

## 1. 文档目的

本文档提供一套可以直接复制给其他 AI 的提示词模板，用于让对方在读取长庚环项目资料时，尽可能按正确顺序理解项目、避免误写、减少脑补。

模板分为三种：

- 短版：适合上下文紧张时使用
- 标准版：适合大多数写作任务
- 定向版：适合要求 AI 只写某一类文档或章节

## 2. 使用前说明

在投喂其他 AI 时，默认优先附带以下材料：

## 3. 短版提示词模板

适合：只想让另一个 AI 先快速理解项目。

```text
你将根据我提供的长庚环项目资料理解整个项目，并在后续写作中严格以资料为准，不要自由脑补。请优先把项目理解为“面向智能戒指场景的端云协同健康辅助系统”，而不是单一睡眠监测 App，也不是正式医学诊断平台。

请特别注意：
1. 当前 Android 已完成单入口整合，唯一运行宿主是 app-shell。
2. cloud-next 是当前云端服务层，负责认证、报告理解、问诊增强、干预回写等接口。
3. 症状自查、AI 医生、医检报告分析、干预执行、趋势反馈都属于当前系统能力。
4. 推荐模型、数字人、UI 收敛等内容必须区分“已落地”和“研究/持续优化中”。
5. 不要把辅助判断写成正式医学确诊，不要把研究文档写成已全面上线。

阅读完资料后，请先输出：
1. 项目一句话概述
2. 系统三层结构概述
3. 你认为当前最重要的 5 个事实边界
```

## 4. 标准版提示词模板

适合：让另一个 AI 参与写计划书、开发文档、技术研究报告、测试文档等。

```text
你现在要基于我提供的“长庚环”项目资料进行写作和总结。请把自己当作一个严格依赖事实源的技术写作助手，而不是自由发挥的宣传文案模型。

项目信息：
- 项目名称：长庚环
- 英文名：VesperO
- 团队名称：夜巡者

请按以下原则工作：
1. 以我提供的文档为第一事实源，不要凭经验补写未出现的系统能力。
2. 优先使用“辅助判断、问诊增强、报告理解、康复辅助、持续收敛中”等稳妥表达。
3. 不要把系统写成正式医学诊断平台，不要把研究中的推荐模型写成已全面上线。
4. 不要把 legacy app 写成当前 Android 运行入口；当前唯一运行入口是 app-shell。
5. 不要混淆 Vercel、Cloudflare、Supabase、Resend 的角色。
6. 如果文档之间出现差异，以基线事实文档和工程事实文档为准。

你的输出风格要求：
- 用语稳定、专业、克制
- 先写事实，再写归纳
- 避免营销化夸张表达
- 避免未证实的量化结论

在开始正式写作前，请先完成以下步骤：
1. 用 5-8 句话总结项目整体定位
2. 列出系统的主要模块与职责
3. 列出你认为后续写作时最容易误写的 5 个点
4. 明确哪些能力属于“已落地”，哪些只能写成“持续优化或研究中”

之后，再根据我的具体任务继续写作。
```

## 5. 定向任务版模板

### 5.1 用于写开发文档

```text
请基于我提供的长庚环项目资料，帮助我编写项目开发文档。写作时请重点围绕：
1. 系统总体架构
2. Android 单入口与模块边界
3. 云端 API 分组与关键链路
4. 设备采集、本地数据库、云端增强的协同机制
5. 功能模块与数据流

注意：
- 不要把所有模块写成已完全迁移完成
- 不要伪造数据库字段和接口字段
- 不要把辅助能力写成正式诊断功能
- 默认使用“长庚环 / VesperO / 夜巡者”这组名称
```

### 5.2 用于写技术研究报告

```text
请基于我提供的长庚环项目资料，帮助我编写技术研究报告。请重点关注：
1. 项目为何采用端云协同路线
2. 智能戒指数据采集、本地兜底与云端增强的协作方式
3. 症状自查、AI 医生、报告理解和干预反馈如何形成闭环
4. 推荐模型、数字人和多模态能力当前的实际状态

注意：
- 不要把研究规划写成既成事实
- 不要把项目写成纯算法论文型系统
- 更强调系统设计、技术路线和综合工程能力
```

### 5.3 用于写测试文档

```text
请基于我提供的长庚环项目资料，帮助我编写项目测试文档。请重点覆盖：
1. Android 单入口运行验证
2. 登录、注册、邮箱确认、忘记密码链路
3. 戒指 BLE 采集与本地落库
4. 症状自查、报告分析、AI 医生、干预执行等主流程
5. 云端接口、数据回写与趋势反馈

注意：
- 测试目标是主链路闭环，而不是只罗列页面名称
- 不要凭空编造未在资料中出现的测试结果或量化指标
```

## 6. 给 AI 的禁止脑补说明

如果要单独附一段“禁止脑补”说明，可以直接复制以下内容：

```text
请不要自行补写以下内容，除非资料中已经明确出现：
1. 未实现的接口字段、数据库字段或部署拓扑
2. 未证实的准确率、召回率、用户量、性能指标
3. 医学确诊、自动治疗、正式诊断报告等结论
4. “所有模块已完全重构完成”之类的过度成熟表述
5. 把研究规划文档写成当前已全面上线能力
```

## 7. 建议输出风格说明

如果你想进一步约束另一个 AI 的风格，可附以下要求：

```text
输出要求：
- 优先写事实，不先写评价
- 语言专业、稳妥、克制
- 面向正式文档，不要用口语化网络表达
- 若某项能力仍在持续优化，请直接写“持续收敛中”或“持续增强中”
- 若存在边界，请明确指出，不要为了完整性强行补齐
```

## 8. 一句话使用建议

如果后续你只想快速把项目交给另一个 AI，默认做法就是：把资料包里的入口索引、基线事实、写作补充一起给它，再使用本文档中的“标准版提示词模板”作为开场提示。
