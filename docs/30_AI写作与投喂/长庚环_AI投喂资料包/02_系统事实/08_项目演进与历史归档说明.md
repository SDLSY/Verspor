# 项目演进与历史归档说明

- 适用任务：历史到现状的映射、架构演进说明、旧代码引用边界、源码对齐写作。
- 推荐阅读优先级：高。
- 是否允许对外直接引用：可以转写后引用。
- 最后核对日期：2026-03-15。

## 结论先行

当前仓库不是“两套彼此独立的新旧产品并存”，而是同一条业务主线从旧单体 `app/` 演进为 `app-shell + core-* + feature-*` 模块化结构后的状态。

## 旧单体 `app/` 的定位

- 是历史归档。
- 仍保留大量老代码，可做演进参考。
- 不再是当前 Android 构建运行的主入口。
- 某些旧实现与新模块中的代码存在一一映射关系。

## 演进方式的本质

这次演进更接近：

- 先按业务拆模块。
- 再保留大部分包名与实现。
- 逐步替换或增强关键能力。

而不是：

- 一次性全部推倒重写。
- 一次性把所有历史包名清理干净。

## 典型映射关系

### 壳层与应用入口

- 旧单体中的 `MainActivity`、`SleepHealthApp`、部分服务与全局 UI，已迁入 `app-shell`。

### 数据与仓储

- 旧单体中的 `database`、`repository`、`service` 等能力，已拆入 `core-db`、`core-data`、`core-network`、`core-ml`。

### 页面功能

- 旧 `ui.home`、`ui.device`、`ui.doctor`、`ui.relax`、`ui.intervention`、`ui.trend`、`ui.profile` 对应迁入各自 `feature-*` 模块。

### 共享 UI

- 图表、控件、导航、主题、页面资源等抽到 `core-common`。

## 新旧并存的真实现象

### 现象 1：包名仍然老

很多现行页面仍然长这样：

- `com.example.newstart.ui.home.*`
- `com.example.newstart.ui.relax.*`

这不是旧单体仍在跑，而是拆模块后保留了包名。

### 现象 2：有些“新名字的 Fragment”只是包装壳

个别 `feature.xxx.*Fragment` 形式的页面更像模块化后的包装入口或占位壳，不一定是当前主导航实际接入的页面。

### 现象 3：历史资源仍然保留

- 某些旧资源文件仍在仓库里。
- 某些旧音频仍可能作为历史资产保留。
- 但运行时优先链路可能已经绕开它们。

## 哪些历史代码还能参考

- 业务演进脉络。
- 旧页面到新模块的功能映射。
- 某些本地算法或 demo 设计思路。

## 哪些历史代码不应用来描述当前事实

- `app/` 是否是当前入口。
- 旧 README 中对模块职责的说法。
- 旧供应商链路是否仍然是当前主链。
- 旧 UI 截图或旧资源是否仍代表当前界面。

## 当前项目演进的几个标志性变化

### 1. 构建入口切换

- 从历史单体入口转为 `:app-shell`。

### 2. 云端能力增强

- `cloud-next` 由单纯接口层演进为“API + 后台工作台 + 模型/作业/审计管理”的系统。

### 3. AI 能力从单点变为矩阵

- 报告理解、医生问诊、桌面机器人、图片分析、TTS、推荐解释等形成多链路能力矩阵。

### 4. 干预能力扩展

- 从传统页面型放松能力扩展到呼吸反馈、音景、Zen 轻交互、药物/饮食分析、画像与恢复分联动。

## 写作时如何处理“历史”

推荐写法：

- “项目由旧单体逐步演进为模块化 Android 架构。”
- “仓库保留历史实现用于演进参考，但当前运行时已切换到 `app-shell` 与模块化主线。”

不推荐写法：

- “项目已经把旧单体彻底删除。”
- “当前 Android 仍由 `app/` 直接运行。”
- “所有旧资源和旧逻辑都已经完全清空。”
