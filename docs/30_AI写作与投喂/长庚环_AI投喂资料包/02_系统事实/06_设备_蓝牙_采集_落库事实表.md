# 设备_蓝牙_采集_落库事实表

- 适用任务：硬件接入说明、BLE 链路说明、数据采集说明、端侧感知能力介绍。
- 推荐阅读优先级：高。
- 是否允许对外直接引用：可以转写后引用。
- 最后核对日期：2026-03-15。

## 一句话结论

项目里的戒指不是“挂名设备”，而是真正通过 BLE 接入 Android，并把实时指标落到本地数据库，进一步参与今日页、干预反馈、趋势和恢复分。

## 设备接入总表

| 层级 | 主要位置 | 作用 |
| --- | --- | --- |
| 设备 UI | `feature-device` | 扫描、连接、展示状态、触发采集 |
| BLE 协议 | `core-ble` | GATT、命令、帧解析、mock/demo |
| 全局壳层 | `app-shell` | 保持应用级生命周期与页面间共享 |
| 数据落库 | `core-db` | `devices`、`health_metrics`、`ppg_samples` 等 |
| 数据消费 | `feature-home`、`feature-relax`、`feature-trend` | 今日页、干预反馈、趋势聚合 |

## 设备链路的真实路径

### 扫描与连接

- 用户在设备页发起扫描。
- 设备页 ViewModel 调用 BLE 层进行连接。
- 当前连接状态不应随着离开设备页就直接销毁。

### 数据接收

- 戒指通过 BLE 持续发送原始或半结构化数据。
- `core-ble` 负责帧解析。
- 应用提取健康指标与 PPG 数据。

### 数据落库

主要表包括：

- `devices`
- `health_metrics`
- `ppg_samples`
- `sleep_records`
- `recovery_scores`

### 数据消费

- 今日页读取最新指标。
- 呼吸训练和音景会话读取最新反馈。
- 趋势页读取周期汇总。
- 恢复分读取最近 24 小时或周期内指标。

## 典型数据项

### 健康指标

- 心率
- 血氧
- 体温
- HRV
- 运动/动作信号

### 波形类数据

- PPG 样本

### 设备状态

- 连接状态
- 最后连接时间
- 设备标识

## 这些数据在页面里分别怎么用

### 今日页

- 用来显示最近生理指标。
- 参与恢复分和建议解释。

### 呼吸训练

- 生成实时放松反馈信号。
- 驱动视觉化节律和音景微调。

### 音景干预

- 根据当前状态微调多轨音量权重。

### 趋势页

- 构成周报/月报的生理维度摘要。

## 需要特别强调的事实

- 当前数据链路是“连接 -> 采集 -> 落库 -> 页面消费”，不是只在设备页临时显示一串数字。
- 生理数据已经影响到干预反馈和恢复分，不只影响“设备状态卡”。
- 数据断流时页面会降级，但不会因此否认整个设备链存在。

## 对外写作可用表达

- “戒指通过 BLE 持续采集关键生理指标，Android 端本地落库后进入恢复评估与干预反馈。”
- “硬件感知不是独立配件展示，而是整个分析和干预闭环的输入层。”

## 不推荐写法

- “设备只是做展示。”
- “连接后数据只在设备页可见。”
- “项目没有真实硬件数据链路。”
